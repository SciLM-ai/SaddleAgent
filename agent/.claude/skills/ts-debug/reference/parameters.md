## `LREAL` must match across any two energies you compare ‚Äî `PREC` silently sets its default, so a `PREC` change shifts the projector and offsets *every* energy equally
Real- vs reciprocal-space projection (`LREAL=.TRUE.`/`Auto` vs `.FALSE.`) uses *different* projector functions, so two otherwise-identical runs differing only in `LREAL` give total energies offset by a **systematic, uniform** amount ‚Äî the *same* ~0.1‚Äì1 eV on every image/geometry, not a geometry-dependent scatter (that uniformity is the fingerprint, distinguishing it from a real intermediate or a per-image bug). Silent trap: **`PREC` carries `LREAL`'s default**, so switching `PREC` between an NEB and a single-point (or endpoint vs band) changes the projector *even when neither INCAR names `LREAL`* (the exact `PREC`‚Üí`LREAL` default mapping is version-dependent ‚Äî don't trust it to match).
- **Tell.** A single-point recompute of an NEB image at the *same geometry* gives a constant offset across all images (uniform ‚áí projector, not geometry); `grep -i 'lreal\|prec' INCAR` differs ‚Äî or is *absent* in one run, leaving the `PREC`-driven default; the OUTCAR echo shows the resolved `LREAL`.
- **Rule.** Pin `LREAL` *and* `PREC` explicitly and identically in every INCAR whose energies you compare (NEB images, endpoint references, external single-points); never rely on `PREC`-driven defaults. A *uniform* offset between two same-geometry calcs is an `LREAL`/projector mismatch, not physics ‚Äî distinct from the drift that *spikes* images (neb-diagnostics.md inputs-inconsistent). (Expert-stated 2013, VASP 4.6/VTST 2.03 & 5.3.3/VTST 3.0; projector behavior long-standing, not version-specific.)

## `LDIPOL`/`IDIPOL` dipole correction in NEB ‚Äî computed *per image*, it can shift image energies inconsistently and blow up the band; disable for band relaxation, restore only once it's stable
The slab dipole correction (`LDIPOL=.TRUE.` + `IDIPOL=1/2/3`) is evaluated **independently for each image**, and the dipole moment differs image-to-image along the path, so each image's energy is shifted by a *different* amount ‚Äî when it misbehaves the inter-image energies turn non-comparable, **jump by hundreds‚Äìthousands of eV**, and the band never converges (forces won't drop systematically even at a loose `EDIFFG`). It is a prime suspect for an NEB whose per-image energies are wildly non-monotonic.
- **Tell.** `grep -i dipol INCAR` shows `LDIPOL`/`IDIPOL` active; per-image OSZICAR/OUTCAR energies differ by huge, erratic amounts across SCF cycles / between adjacent images (‚âÝ the *uniform* `LREAL` offset above; ‚âÝ the endpoints‚Üîband inconsistency in neb-diagnostics.md).
- **Fix.** Turn it **off** for the whole band (`LDIPOL=.FALSE.`, drop `IDIPOL`) and set `ISYM=0` (ISYM entry); run `IBRION=3 POTIM=0.01` ~50 steps, confirm forces drop systematically, then scale up (`POTIM‚Üí0.1` / a faster `IOPT`). Bring the correction back only once the band is stable ‚Äî well-pinned (below) ‚Äî for a final reconverge / single-point if the field is physically needed, validating the inter-image energies are sane.
- **Not a ban on dipole corrections.** They stay valid on slabs (e.g. the single-face cross-check, neb-diagnostics.md symmetric-slab entry) ‚Äî pin `IDIPOL` to the surface normal (and `DIPOL` to the cell/charge center) so the correction is well-defined; this entry is the *convergence-destabilizing* failure mode, not a prohibition. (Expert-stated 2013; usage not version-specific.)

## Ultrasoft pseudopotentials (USPP) + VTST/high precision ‚Üí `RSPHER: NONLR_S%LMMAX is wrong` crash ‚Äî use PAW
A VTST optimizer (`IOPT>0`, e.g. LBFGS) aborting at ionic step 1 with `internal ERROR: RSPHER: NONLR_S%LMMAX is wrong` is a **pseudopotential-family fault, not an optimizer bug**: ultrasoft pseudos (`US <elem>` in POTCAR TITEL) size the partial-wave l-channel dimension `LMMAX` differently from PAW, and with `PREC=High` (or `ENCUT` well above the pseudo's default) the projector dimension comes out wrong. USPP is legacy anyway ‚Äî PAW is the standard for VTST/NEB.
- **Tell.** `grep -i 'rspher\|lmmax' OUTCAR`/stdout; `grep TITEL POTCAR` shows `US ` (ultrasoft), not `PAW_PBE`/`PAW_LDA`; `grep -iE 'prec|encut' INCAR` shows `High` / a cutoff far above the POTCAR `ENMAX`.
- **Fix.** Switch to PAW (`PAW_PBE`), `PREC=Normal`, `ENCUT` near the PAW-recommended value (~300 eV for light alkali/alkaline-earth). Don't pair USPP with high precision or VTST optimizers.
- **Rule.** VTST optimizers want PAW; ultrasoft pseudos with `PREC=High` throw the fatal `RSPHER`/`LMMAX` error ‚Äî check the POTCAR family before blaming the optimizer. (The same USPP+precision setup also drives the artificial endpoint IS‚ÜîFS energy gap in neb-diagnostics.md.) (Expert-stated 2017, VASP 5.4.4; USPP-legacy/PAW-standard not version-specific.)

## `EDIFF` for the saddle *search* (NEB & dimer) ‚Äî `1e-5` suffices; tighten only for the *final* reconverge/frequency, or when the mode estimate *degrades*
The dimer/Lanczos rotation reads the lowest-curvature mode from the *finite-difference force* between its images ‚Äî more noise-sensitive than a relaxation *or NEB*, **but the curvature tolerates moderate noise far better than feared.** So `EDIFF‚âà1e-5` is the right *default* for both curvature optimizers (`IOPT=1`/`IOPT=2`); a reflexive `1e-8` only burns cluster time, and the VASP **default `1e-4` is too loose**. Same economy on the **NEB band**: run the whole hunt at `~1e-5`, reserving `1e-7`/`1e-8` for a *final* reconverge or frequency once the geometry is in hand. And **tightening `EDIFF` does *not* speed ionic/geometric convergence** ‚Äî that is the force-criterion `EDIFFG`'s job (its entry below); the electronic SCF and the ionic-force loop are two independent hierarchies, so don't conflate them (a recurring confusion ‚Äî "will a tighter `EDIFF` help my relaxation/charged system converge?": no, set `EDIFFG`).
- **Escalate mid-search only on the noise *tell*.** Go to `1e-7`/`1e-8` *when* rotation curvature **flips negative‚Üípositive**, *or* the **torque stays high/erratic** (DIMCAR `Torque`) while force falls and curvature stays negative ‚Äî *or* a second-order **`IOPT=1`/`2` band whose forces oscillate/grow** (incl. **CG bracketing-failure**) ‚Äî worst in the **final approach to a minimum**, where loose `EDIFF` leaves electronic force-noise within 1‚Äì2 orders of `EDIFFG`, ‚âà the target gradient, so the LBFGS/CG finite-difference Hessian/harmonic model breaks: keep `EDIFF` *well below the `EDIFFG` force floor* (`‚â§1e-7`), fallback lower `INVCURV`, last resort first-order `IOPT=3/7` (IOPT entry). Either *looks* converging but never settles: the estimate is *degrading*, and `1e-6`/`1e-7` can read "converged" yet leave torque chaotic, so `1e-8`. Distinct from per-step rotation merely *not converging* (by-design far from the saddle ‚Äî dimer-diagnostics.md); often co-occurs with a start not near a saddle.
- **Rule.** Electronic `EDIFF` (‚âÝ force-criterion `EDIFFG` ‚Äî its entry): search band/dimer at `~1e-5`; tighten to `1e-7`/`1e-8` only for the *final* reconverge/frequency, or mid-search when curvature flips / torque stays high while force creeps (or a 2nd-order `IOPT=1`/`2` band's forces oscillate/grow) ‚Äî never to chase ionic convergence. (Expert-stated 2015‚Äì2022, VASP 5.4.4; not version-specific.)

## OSZICAR ionic `d E` ‚âà total energy (huge, not a small delta) ‚Äî cosmetic format-string bug in old VASP, not a broken SCF
On the ionic-step line (`N F= ‚Ä¶ E0= ‚Ä¶ d E =‚Ä¶`), an old-build format overflow prints the **total energy** in the `d E` field instead of the step-to-step delta (e.g. `d E =-.590E+02`, same magnitude as `F=`/`E0=`) while the SCF converges normally ‚Äî display-only, it changes neither energy, forces, nor the optimizer.
- **Tell.** `d E` magnitude ‚âà `F=`/`E0=` (not a small incremental change) yet the `DAV`/`RMM` SCF lines descend and the `rms`/`rms(c)` columns shrink each cycle.
- **Fix.** Ignore the `d E` field ‚Äî judge electronic convergence from `rms`/`rms(c)`, ionic from OUTCAR `reached required accuracy`/`FORCES: max atom` ‚Äî and upgrade VASP to clear the formatting error. (Affected ~VASP 4.x / early 5.x; cosmetic only, fixed upstream ‚Äî OLD/2007.)

## `SIGMA`/smearing doesn't corrupt NEB/dimer forces ‚Äî VTST just optimizes the OUTCAR `TOTAL-FORCE` ionic forces
VTST drives the band/dimer with the **standard VASP forces** printed in the OUTCAR `TOTAL-FORCE` block (SKILL force-criterion row) ‚Äî there is no separate "NEB force." The recurring conceptual worry ‚Äî *does NEB differentiate the electronic free energy `F = E ‚àí TS_elec` or the energy, and does finite-`T` smearing spoil the gradient?* ‚Äî is moot for the MEP: at normal smearing (`SIGMA`‚âà0.05‚Äì0.2, MP/Gaussian) the entropy term and its effect on the force are negligible, so those printed forces are the right quantity to seat the saddle. Don't hunt a "free-energy force" switch or distrust converged forces over this. The OUTCAR prints both `free energy TOTEN` (its gradient *is* the forces) and `energy without entropy`/`E0` (the œÉ‚Üí0 extrapolation ‚Äî forces are **not** its derivative); the two split only at finite œÉ. So if `SIGMA` is genuinely large, or energy conservation along the MEP matters, erase the split at the source: set `ISMEAR=-5` (tetrahedron) or shrink œÉ‚Üí0, then read the barrier from `E0`. (Expert-stated 2009; smearing/force behavior stable across VASP versions.)

## Normal-mode frequencies / Hessian ‚Äî VASP-native finite-difference `IBRION=5/6` (5 no-sym, 6 symmetry-reduced) *or* VTST dynamical matrix `ICHAIN=1`, on a converged geometry; DFPT `7/8` blocked under DFT+D
Harmonic normal-mode frequencies at **any** fixed geometry ‚Äî reactant, product, *or* saddle (ZPE, TS check, hTST prefactor ‚Äî workflows.md) ‚Äî come from a finite-difference Hessian on an **already-converged structure**: `IBRION=5`/`6`, **never a relaxation `IBRION=1/2/3`**. The Hessian sets its *own* step count ‚Äî **‚âà`NFREE`√ó3N + 1 force evaluations** (N = free atoms; `NFREE=1` one-sided = 3N+1, default `NFREE=2` central = 6N+1) ‚Äî so **`NSW` does not cap it**: a `NSW=1` job that keeps running is working as designed, not a bug. Both modes **respect selective-dynamics `F` flags ‚Äî only free (`T`) DOF are displaced, frozen atoms aren't perturbed** ‚Äî so the constrained POSCAR from the NEB/dimer run is the correct freq input reused unchanged, *and* freezing spectator/substrate atoms is the **dominant cost lever** (step count is linear in N); dropping to `NFREE=1` (one-sided, less accurate) roughly halves it. The perturbation-theory alternative `IBRION=7/8` (DFPT/linear-response) has **no dispersion term**, so it refuses any DFT+D correction (`IVDW`/`LVDW`, any DFT-D2/D3): the job aborts `perturbation theory cannot be used for DFT+D calculations`. The restriction is **unique to DFPT** ‚Äî every *force-based* method (relaxation, **NEB/CI-NEB**, dimer) takes DFT-D transparently because the correction enters through the forces/energy, so just set `IVDW` as in any single-point; no special workaround.
- **Keep the relaxation's KPOINTS ‚Äî `IBRION=5`'s "Œì point" is *vibrational*, not electronic.** The manual's Œì-point language for `IBRION=5/6` means **zone-center phonons only** (no phonon wavelength longer than the supercell ‚áí no inter-cell vibrational coupling), *not* that electronic k-sampling must be Œì. Dropping to a single Œì k-point ‚Äî or hunting a workaround ‚Äî believing a k-mesh is forbidden misreads it: reuse the **same electronic k-mesh the geometry was relaxed with** (a full mesh is valid and often necessary for accurate forces).
- **Fix.** On a DFT+D system take the finite-difference modes (`IBRION=5/6`) ‚Äî no such restriction. They differentiate *numerical* forces, so compensate: tighten `EDIFF`‚â§`1e-7` and use a *small* displacement (`POTIM`‚âà0.01‚Äì0.015 √Ö) to keep the second derivatives out of the noise; partial Hessian (`dymseldisp.pl`) for large cells.
- **VTST dynamical matrix (`ICHAIN=1`) ‚Äî CHAIN-driver alternative to `IBRION=5/6`, with one `EDIFFG` trap.** VTST builds the Hessian via `ICHAIN=1` driven by `IBRION=3 POTIM=0` (the dimer-style CHAIN driver ‚Äî *not* a relaxation despite the `IBRION=3`), displacing the components listed in **`DISPLACECAR`** (one `dx dy dz` row per atom) over `NSW`‚âà3N+1 steps. **Trap ‚Äî a too-loose `EDIFFG` aborts the displacement loop early:** *any* `EDIFFG` the near-converged geometry already meets ‚Äî a relaxation-grade negative (‚àí0.01‚Ä¶‚àí0.1) *or* a positive/default *energy* criterion ‚Äî is satisfied within the first few displacements, so VASP prints `reached required accuracy - stopping structural energy minimisation` and quits after a **handful** of steps (often just one, returning the TS with **zero** displacements) ‚Äî `freq.mat` then carries far fewer rows than `3N+1` (`6N+1` central-difference). **Fix: `EDIFFG=¬±1e-10` (or `0`)**, effectively unsatisfiable either sign, so every `NSW` displacement step runs (and confirm a `DISPLACECAR` is present). **`EDIFF` is the wrong knob** ‚Äî tightening *electronic* `EDIFF` (keep ~`1e-8`) does nothing for the premature exit; it's `EDIFFG` (the *G* = force/energy criterion) that guards the loop, a recurring confusion. Don't instead delete `IBRION=3` to "stop confusing VASP" ‚Äî that drops back to an un-driven `NaN rms(c)` hang; keep the `IBRION=3 POTIM=0` driver and fix `EDIFFG`.
- **Tell.** Wanting frequencies but INCAR carries a relaxation `IBRION` (1/2/3) **with no `ICHAIN=1`** ‚áí wrong mode (`grep -i 'ibrion\|ichain' INCAR`; VTST's dynamical matrix legitimately uses `IBRION=3 POTIM=0`). An `ICHAIN=1` freq exiting after far fewer force calls than `3N+1` (`freq.mat` short, `reached required accuracy` after a handful of steps) ‚áí `EDIFFG` already met ‚Äî set `¬±1e-10`/`0`, **not** a tighter `EDIFF` (`grep -i 'ediffg\|ediff\|displacecar' INCAR`). A freq job running far more ionic steps than `NSW` is `IBRION=5/6` working as designed ‚Äî read its step count from `NFREE` √ó free-`T` POSCAR rows, not `NSW` (`grep -i 'ibrion\|nfree\|nsw' INCAR`). `grep -i 'ivdw\|lvdw\|vdw\|dft-d' INCAR` shows a dispersion correction alongside `IBRION=7`/`8` (then the OUTCAR abort above).
- **Rule.** Frequencies = a finite-difference Hessian (`IBRION=5/6`, *not* a relaxation `IBRION`) on a pre-converged geometry, honoring selective-dynamics flags (reuse the NEB/dimer POSCAR as-is) ‚Äî *or* VTST's dynamical matrix (`ICHAIN=1`, `IBRION=3 POTIM=0`+`DISPLACECAR`), which needs `EDIFFG=¬±1e-10`/`0` (a too-loose `EDIFFG` ‚Äî *not* `EDIFF` ‚Äî aborts the displacement loop after a few steps). Step count ‚âà`NFREE`√ó3N+1 ‚Äî set by the algorithm, *not* `NSW`; shrink it by freezing spectators (reduce N) or `NFREE=1`. On DFT+D, DFPT (`7/8`) is the *sole* method that aborts, so finite-difference is mandatory (offset its noise with tighter `EDIFF` + small `POTIM`) ‚Äî every force-based method (NEB/CI-NEB, dimer, relaxation) takes DFT-D fine, just set `IVDW` like any single-point. (`IBRION=5/6` freq + selective-dynamics expert-stated 2019; `NSW`-ignored / `NFREE`√ó3N+1 step count 2017; DFT+D-excludes-DFPT VASP 5.4.1 / PBE+D3, force-methods-take-DFT-D expert-stated 2016 ‚Äî both long-standing, not version-specific; `ICHAIN=1` `EDIFFG=¬±1e-10`/`0` trap expert-stated 2014‚Äì2015, VASP 5.3.3 / VTST 3.1, not version-specific; Œì-point-is-vibrational-not-electronic MID/2011, VASP 4.6 / VTST 2.03d, not version-specific.)

## Implicit solvation (`LSOL`/VASPsol) in NEB ‚Äî relax *with* it, don't bolt it on as single-points
`LSOL=.TRUE.` (+ `EB_K` for the solvent) changes the electrostatic environment, so the nuclei relax to *different* positions with it on. Adding it only as a post-hoc single-point (`IBRION=-1`, `NSW=1`) on vacuum-optimized images leaves every geometry unrelaxed in the solvent field, so the solvation energies aren't comparable along the path ‚Äî the tell is the **transition state landing *below* the initial state** (or any interior image below an endpoint), a physically impossible ordering. (A flavor of the "inputs inconsistent endpoints‚Üîband" failure in neb-diagnostics.md: geometry and energy computed under different settings.)
- **Fix.** Relax initial *and* final states with solvation on, then re-run the *whole* NEB with solvation active throughout ‚Äî never mix vacuum geometries with solvation single-points for energy comparison. Solvation is rarely the expensive part of the run, so "single-points are cheaper" buys only meaningless relative energies.
- **Charged cells (`NELECT` ‚âÝ neutral).** VASP neutralizes a charged supercell with a *uniform background charge* that interacts with the real charge over the whole cell volume, distorting the PES of charged adsorbates. Implicit solvation **screens the charge locally** and suppresses that background-charge artifact ‚Äî so for a charged-adsorbate NEB, pair `NELECT` with `LSOL` rather than treating solvation as optional. With strong solvent‚Äìsolute coupling, add an explicit first solvation shell + implicit continuum (the solvation energy then converges with ~one shell). (Distinct from the stale-`CHG*`/`WAV*` BRMIX trap on a `NELECT` change ‚Äî see the NELECT entry below.)
- **Tell.** `grep -i 'lsol\|eb_k' INCAR` (solvation on) with `grep -iE 'ibrion|nsw' INCAR` showing `IBRION=-1`/`NSW=1` = a static single-point ‚Äî confirm it sits on a *solvation-relaxed* geometry, not a vacuum one. Non-default `NELECT` flags a charged cell that needs the local screening.
- **Rule.** Implicit solvation belongs in the geometry relaxation *and* the band, not a single-point afterthought on vacuum structures (post-hoc single-points give unphysical relative energies, TS below IS); for charged systems it is effectively mandatory ‚Äî it screens charges locally and kills the uniform-background-charge artifact. (Expert-stated 2022; VASPsol `LSOL`/`EB_K`.)

## Selective-dynamics fixed atoms (`F F F`) in NEB ‚Äî inert passengers, identical in every image
Fixing atoms `F F F` works in NEB, but those atoms are *inert*: **no spring force** (the spring acts on image-to-image displacement, zero for an atom fixed at the same coordinate everywhere) and their **potential force is zeroed** by VASP ‚Äî passengers, not participants. Only mobile `T T T` atoms trace the MEP. A band with *all* atoms fixed is meaningless; the lone exception is a solid-state cell optimization (cell relaxes, atoms fixed in relative coordinates).
- **Setup ‚Äî the freeze must originate at the endpoints, and the `T/F` *mask* (which atoms are `F F F` vs `T T T`) must be identical across all images *and* both endpoints, alongside the fixed coords.** Keep the cell fixed (`ISIF=2`). A *mask* mismatch ‚Äî an atom free in some images/endpoints but frozen in others ‚Äî lets those images relax DOF the rest can't, so the band runs smooth for ~100 steps then diverges chaotically, **immune to optimizer/timestep restarts**. A *coord* mismatch is the same trap: IS/FS built/relaxed *separately* carry frozen rows differing sub-√Ö, which `nebmake.pl` smears across the band as artificial force along supposedly fixed coordinates ‚Äî masking or shifting the saddle. `nebmake.pl` copies endpoint constraints into the images, but **hand-editing an image POSCAR (e.g. to fix a short bond) can silently reset its selective-dynamics flags** ‚Äî re-diff after any manual edit. Copy frozen coords+flags from *one* reference into both endpoints, re-relax both, and rebuild; freezing only interior images (endpoints free) is the same inconsistency. **Never recycle intermediate POSCARs from a *prior* run:** regenerate every interior image fresh after any endpoint re-relax (reused intermediates carry stale coords/flags ‚Äî the same optimizer-immune chaos).
- **Match the physics ‚Äî a bulk hop leaves the host free, a surface slab freezes its base.** NEB relaxes *every* atom by default; for a *bulk* hop the lattice relaxation around the migrating species is part of the MEP and barrier (non-diffusing atoms moving is correct behavior), so freezing nearly the whole host to leave one ion is a *bad approximation*, not a safe speed-up ‚Äî fix the *cell* (`ISIF=2`), leave host atoms **free**, freeze only genuine spectators far from the reaction coordinate. A **surface slab is the opposite case**: freeze the **bottom substrate layers** (`F F F`) to model the semi-infinite bulk underneath ‚Äî and that freeze doubles as optimizer stabilization (next bullet).
- **Freeze to kill the low / rigid-body modes a gradient optimizer can't damp.** A *fully unfrozen* surface slab (free whole-slab translation/rotation), a freely-translating species (weakly-bound adsorbate, mobile interstitial ion, isolated molecule), *or any fully periodic cell with no anchored atom* carries near-zero-curvature translational modes the saddle optimizer can't damp ‚Äî two tells: forces *oscillate*/*spike* instead of systematically dropping (selective-dynamics all `T T T`), *or the whole band rigid-translates by a lattice vector so inter-image spacing collapses and every image's final `E0` collapses to one value (all OSZICARs identical, barrier erased, saddle unresolved)*. **LBFGS (`IOPT=1`) is most sensitive** ‚Äî a fully-unfrozen slab is a common root of its `>1‚Üí>10` eV/√Ö force-blowup (IOPT entry). Cure by freezing **one** atom to the *same* fractional coordinate in every image ‚Äî best one at an **identical position in both endpoints**, so it removes only the rigid-body translation, never an MEP DOF: a slab's **bottom layers** (also the physical model above), or one representative atom of a floating species (`F F F`) ‚Äî the floating-species/periodic-cell pin is purely *numerical* (apply it in **every** band image ‚Äî an endpoint-only freeze leaves the interior images free to drift ‚Äî endpoints themselves free), the slab freeze is physical (consistent across all images *and* endpoints, per the setup bullet). If LBFGS still blows up, lower `INVCURV` or drop to QuickMin (`IOPT=3`).
- **Judging endpoint convergence ‚Äî only free atoms count toward `EDIFFG`.** VASP applies the force criterion to the *unconstrained* DOF only, so an endpoint (or any image) is converged once every `T T T` atom is below `|EDIFFG|` ‚Äî the fixed atoms need not, and generally will not, meet it. The OUTCAR TOTAL-FORCE block still *prints* the full physical force on a fixed atom (often well above `|EDIFFG|`); that printed value is excluded from the optimizer and the "reached required accuracy" test, so ignore it. "Not all atoms are below EDIFFG, but the free ones are" describes a *converged* relaxation, not an incomplete one ‚Äî don't keep relaxing or loosen the criterion chasing the frozen rows.
- **Tell.** `diff` the selective-dynamics block across all image+endpoint POSCARs ‚Äî both the `T/F` flags *and* the fixed-atom Direct-coordinate rows must be bit-for-bit identical. A `CONTCAR`/`OUTCAR` sitting in an *interior* image dir is the smoking gun that images were carried over from a prior run, not regenerated by `nebmake.pl`.
- **Version caveat ‚Äî `F` atoms drifting in an NEB can be a *since-fixed* VTST bug, not your setup; check the build before debugging coords/masks.** Two distinct ones, both resolved upstream: **(1)** VTST L-BFGS (`IOPT=1`) **before VTST 3.1** (2014) intermittently moved fixed atoms with *uniform, unphysical* motion (not the force-driven creep of a real bug) on **any** cell ‚Äî the L-BFGS rework that added autoscaling fixed it. **(2)** `chain.F` **before rev 169** (fixed 2016-05-23) zeroed the constrained force in the **Cartesian** frame though selective-dynamics `T/F` flags are defined in **direct (fractional)** coords: an **orthogonal** cell is safe (frames coincide), but a **non-orthogonal lattice or a partial-axis (single-component `T`/`F`) freeze** cancels the wrong components and `F` atoms silently drift (any optimizer). Tell: uniform/unphysical drift under `IOPT=1` ‚áí (1); drift only on a non-cubic cell / partial-axis freeze ‚áí (2). Fix: **upgrade first** (VTST ‚â• 3.1 for (1), rev ‚â•169 for (2)) before touching the setup; to patch old `chain.F`, rotate the force into direct coords, apply the `Free` mask, rotate back ‚Äî as VASP's own `SET_SELECTED_FORCES_ZERO` (`dyna.F`) does. Neither bug bites current builds. (VTST L-BFGS < 3.1 fixed 2014; chain.F < rev 169 fixed 2016-05-23.)
- **Rule.** `F F F` atoms in an NEB are zero-spring, zero-potential-force passengers: introduce any freeze at the *endpoint relaxations* (not merely in the images), put the fixed atoms at the same fractional coordinate *and the same `T/F` mask* in every image and endpoint, and keep the cell fixed (`ISIF=2`); only `T T T` atoms participate ‚Äî and judge endpoint/image convergence on the free atoms alone, since fixed atoms print a force but never count toward `EDIFFG`. The default all-atoms-free band is the correct MEP. (Expert-stated 2017‚Äì2023; usage not version-specific ‚Äî but see the two since-fixed freeze caveats above: VTST L-BFGS < 3.1, chain.F < rev 169.)

## `ISIF` for NEB ‚Äî fix the cell (`ISIF=2`); genuinely different endpoint cells ‚áí variable-cell GSSNEB (`LNEBCELL=.TRUE.`), not plain NEB
Run the whole band ‚Äî every image *and* both endpoint relaxations ‚Äî at **constant volume and shape** (`ISIF=2`), never relaxing the cell (`ISIF=3`). Two reasons: (1) at fixed volume the basis-set (Pulay) and k-sampling errors are near-identical across images so they **cancel** in the relative energies ‚Äî that cancellation *is* the band's accuracy, and a per-image cell change breaks it; (2) a relaxing cell conflates the migration barrier with a coupled *elastic* distortion and shifts the inter-image reference frame ‚Äî for a bulk hop the activation energy of interest is the atomic rearrangement, not lattice strain. Cell DOF belong in a *separate* static relax. The exception is a reaction that genuinely changes the cell (volume, lattice parameters, *or* angles ‚Äî endpoints in distinctly different bulk phases/symmetry): **plain NEB is *fixed-cell* and cannot interpolate between two different cells**, so a TS search between structures with **different unit cells** needs the **generalized solid-state NEB (GSSNEB)** ‚Äî turn it on with **`LNEBCELL=.TRUE.`** (in vtstcode ‚Äî but SS-NEB *also* needs a **`main.F` source patch**, not just the bundled chain code: stock `main.F` calls `CHAIN_FORCE` *without* the stress-tensor argument, so a `CHAIN_FORCE` arg-count/type **compile** error ‚Äî Intel `#6784` "actual arguments cannot be greater than dummy arguments" / `#6633`/`#6634` at `CALL CHAIN_FORCE(‚Ä¶TSIF,LATT_CUR%A,‚Ä¶)` ‚Äî means `main.F` and the VTST chain source are out of sync ‚áí apply the ssneb `main.F` change from the vtsttools downloads page, matched to your VASP major version, *not* a code bug to report ‚Äî SKILL version reflex), which makes the cell itself a band degree of freedom that varies along the path (paired with `ISIF=3`). The **solid-state dimer** (cell-variable min-mode search) takes the *same* switch: `LNEBCELL=.TRUE.` is what enables *either* method's cell DOF ‚Äî without it the cell never enters the *reaction coordinate* (a VTST optimizer leaves it frozen; a *built-in* VASP optimizer + `ISIF=3` relaxes each image's cell *independently*, off the band ‚Äî physically wrong for a solid-state transition) so you are running an ordinary fixed-cell dimer/NEB no matter what else is set.
- **GSSNEB/`LNEBCELL=.TRUE.` gotchas.** (a) **The optimizer is a free choice ‚Äî `LNEBCELL=.TRUE.` does *not* lock SSNEB/SS-dimer to `IOPT=3`**, but only `IOPT=3` (Quick-Min) / `IOPT=7` (FIRE) have a variable-cell routine; `IOPT=1/2` (LBFGS/CG) **abort immediately** (`Must set IOPT = 3 or 7 when using LNEBCELL=.TRUE.`). **`IOPT` is a VTST-optimizer-only tag** ‚Äî valid solely when a VTST optimizer is active (`ICHAIN=3` dimer or `LNEBCELL=.TRUE.`). The *legacy native-VASP cell-variable NEB* some older papers reproduce instead lets **`IBRION=3`** drive the optimizer (`POTIM‚âà0.05`, `ICHAIN=0`, `LNEBCELL=.FALSE.`, **no `IOPT`**, `IMAGES=N`) ‚Äî there **delete `IOPT` entirely**, since `IOPT` set alongside the native `IBRION=3` optimizer conflicts and silently breaks the run. For genuinely different endpoint cells prefer VTST GSSNEB (`LNEBCELL=.TRUE.`); the native drag is a legacy reproduction path only (native-drag/`IOPT`-conflict MID/2012, not version-specific). (b) **Restart-to-compare only when the cell shape evolves** ‚Äî *not* a per-iteration ritual: with a static cell there's nothing to re-reference. `ENCUT` is fixed while each image's cell changes, so the basis is effectively *incomplete*/varies along the path (variable-cell Pulay) ‚Äî a drifted *or cross-image-inconsistent* cutoff can **reverse the trend / invert the barrier**: an interior image coming out *more stable than its endpoint*, or the profile *flipping* when `ENCUT` is changed. After a cell change **or any mid-run `ENCUT` change**, **restart from CONTCAR and repeat until ssNEB completes one *full* iteration** ‚Äî only then is the cutoff *uniform across all images* and the image energies comparable (a mid-run bump leaves already-stepped images on the *old* cutoff). **Cheap test for a suspect low-energy intermediate** (an interior image below an endpoint): before trusting it, run a single-point (`NSW=1`) on that image ‚Äî a significant energy shift means it was a stale-cell artifact, not a real well. If the profile still shifts between *consistent* restarts the default is genuinely unconverged ‚Äî **raise `ENCUT` (the 1.3√ó "accurate" target) and converge the band w.r.t. it like any DFT parameter**. (c) match k-*density* (same spacing) across images, not merely the same KPOINTS grid ‚Äî a fixed grid means a *changing* density when the cell varies. (d) **A high-symmetry phase-transition cell false-converges ‚Äî tighten `EDIFFG` far below the NEB norm (‚âà`-1e-9`).** Symmetry cancels many force components, so the residual gradient reads deceptively small and a normal `-0.02`‚Ä¶`-0.05` `EDIFFG` declares the band converged before it has truly relaxed; use a very tight criterion (~`-1e-9`) for high-symmetry SSNEB. And leave **`FMAGFLAG`** at its default (off) ‚Äî the reference GSSNEB deck explicitly disables it (it switches the force-convergence criterion to global total RMS ‚Äî `EDIFFG` entry), never as SSNEB boilerplate. (e) **No applied pressure, and keep images few.** `PSTRESS>0` was **not validated for `LNEBCELL` SSNEB** in early VTST ‚Äî stress coupling to the elastic band destabilizes forces, so image energies/forces stay erratic and never settle; run the band at **zero pressure** (drop `PSTRESS`) to isolate the cause, or use the pressure-aware **TSASE** SSNEB (verify your VTST version before assuming a current build still has the gap). And SSNEB wants **few images** (‚âà4‚Äì6 for a single barrier) ‚Äî the cell DOF compound the band's dimensionality, so a 16-image SSNEB amplifies band noise and crawls; **split a multi-minimum path into independent bands at the minima**, never densify (neb-diagnostics multi-step entry).
- **SSNEB barrier *wildly* different from the fixed-cell (`ISIF=2`) result ‚Äî endpoints not re-relaxed at `ISIF=3`, and/or the wrong read script.** Two coupled traps on switching a band to SSNEB (`ISIF=3` images): **(1)** the `00`/`N+1` endpoints left relaxed at `ISIF=2` while the images run `ISIF=3` ‚Äî post-processing finds no valid `ISIF=3` reference energy for the terminal images, so the barrier comes out far off the fixed-cell value *and migrates* along the path (`neb.dat` endpoint rows off the interior trend). Re-relax **both endpoints at the same `ISIF` as the images** ‚Äî the bidirectional form of the fix-the-cell rule: all five inputs share *one* `ISIF` (fixed-cell = 2 everywhere, SSNEB = 3 everywhere), never mix. **(2)** read the barrier from **`nebefs.pl`/`nebefs.dat`** (made by `nebresults.pl`), not `nebef.pl`/`neb.dat` ‚Äî the standard script may print solid-state barriers as **eV/atom**, mis-scaling the value. Sanity-check against volume: **near-constant cell volume across images ‚áí the SSNEB barrier should stay close to the `ISIF=2` one**; a big swing at ~constant volume points to the mismatch or a misread script, not physics. (Output format expert-uncertain, vtstscripts ~2016 ‚Äî verify with current `nebresults.pl`.)
- **Tell.** `grep ISIF */OUTCAR` shows `ISIF=3` in any image/endpoint with no physical cell-change reason ‚Äî *and* no `LNEBCELL`; conversely, a user wanting a TS between different unit cells on plain NEB ‚Äî *or a solid-state dimer whose cell won't move* ‚Äî is missing `LNEBCELL=.TRUE.` (`grep -i lnebcell INCAR`). **Whenever the cell *is* allowed to relax on the band (legacy native `ISIF=3`, *or* GSSNEB), audit cell *continuity across images* before trusting the barrier: `grep 'volume of cell' */OUTCAR` (and diff CONTCAR lattice vectors) image-by-image ‚Äî volume/shape must vary *smoothly* with image index; an abrupt jump means the cell DOF entered the reaction coordinate and the barrier is meaningless (discard it; drop to `ISIF=2`, or for a genuine cell change use GSSNEB above, not native per-image `ISIF=3`).** `nebmake.pl`'s **`WARNING: BASIS VECTORS`/`LATTICE CONSTANTS ARE NOT THE SAME` (the `I HOPE YOU KNOW WHAT YOU ARE DOING` line) is informational, not an error** ‚Äî endpoints of different cell shape/volume; it forks: *negligible* drift ‚áí a fixed-cell band is fine (the cell merely drags linearly IS‚ÜíFS); *intended* (phase transition / real cell change) ‚áí set `LNEBCELL=.TRUE.` (cell vectors join the reaction coordinate); *accidental* (endpoints each relaxed `ISIF‚â•3`, settled differently) ‚áí the barrier-less endpoint-cell mismatch ‚Äî share one fixed cell (neb-diagnostics.md).
- **Rule.** Fix the cell (`ISIF=2`) for every NEB image and endpoint; relaxing it conflates the migration barrier with an elastic distortion *and* breaks basis-set/k-point error cancellation. Use cell freedom only when the reaction itself changes the cell ‚Äî and for genuinely different endpoint cells run variable-cell **GSSNEB** (`LNEBCELL=.TRUE.`, vtstcode; `ISIF=3`), not fixed-cell NEB which can't interpolate between them. (Fixed-cell rule expert-stated 2020‚Äì2022; GSSNEB/`LNEBCELL` expert-stated 2014‚Äì2016, solid-state *dimer* same switch 2015, vtstcode; high-symmetry tight-`EDIFFG` (~`-1e-9`) + `FMAGFLAG`-off-by-default from the reference GSSNEB deck 2014; ssneb `main.F` source-patch / `CHAIN_FORCE` arg-count compile error expert-stated 2012, VASP 5.x-era ‚Äî patch signature may drift in later VASP/VTST, grep `CHAIN_FORCE` by name to match; otherwise not version-specific.)

## `SPRING` constant ‚Äî canonical ‚àí5 eV/√Ö¬≤ for solid-state NEB; spacing-only (spring *restores* it, so manual image edits are safe); ‚âà‚àí0.5 is a copied-from-molecular red flag, and lowering it never cures a convergence stall
`SPRING` keeps NEB images evenly distributed along the band ‚Äî it sets *image spacing only* (its **sign is irrelevant** ‚Äî only |SPRING| enters), not reaction chemistry **and not convergence quality** (acts on image-to-image displacement, ~no force on the saddle image). Canonical for solid-state diffusion NEB is **‚àí5 eV/√Ö¬≤**, across all systems and bond types ‚Äî leave it there. The old **variable-spring NEB** ‚Äî which crowded images near the saddle by *varying* the spring through a verbose block (`SPRING2`/`ISPRING`/`SPOWER`/`EFIRST`/`ELAST`) ‚Äî was *removed* once **CI-NEB superseded it**: a single `SPRING=-5` replaces that *entire* block, never set those tags separately (in pre-~2007 builds the negative sign *also* selected the improved-tangent INEB, now the default ‚Äî which is why today only |SPRING| matters). For sharper saddle resolution add `LCLIMB=.TRUE.` (entry below), never hunt a variable spring (present in pre-~2007 vtstscripts; the CI-supersedes-VNEB redirect holds for all modern builds). Three misreads: too *weak* (`SPRING=-0.5`, 10√ó under, a copied molecular/gas-phase deck) lets images bunch toward the endpoints and resolves the saddle poorly; *raised* to "dissociate" a strongly-bound molecule (e.g. O2) on the logic that stiffer bonds need stiffer springs ‚Äî the spring distributes images, it doesn't model the bond; or, on a **convergence stall** (forces oscillating/diverging), nudging |SPRING| *down* (‚àí5‚Üí‚àí4/‚àí3) to "help it converge" ‚Äî it can't, so leave ‚àí5 and chase the stall in `IOPT`/optimizer step (`TIMESTEP`/`INVCURV`), `EDIFFG`/`EDIFF`, or image spacing/interpolation (their entries).
- **Manually editing image geometries is safe ‚Äî the spring *restores* equal spacing.** Hand-shifting an atom in an interior image (e.g. to open an unphysically short bond `nebmake.pl` left, ~0.3 √Ö) unequalizes inter-image distances, but the spring pulls them back to even spacing during optimization (either side of the climbing image for CI-NEB) ‚Äî no fix needed at `-5`. Lowering `SPRING` toward `-0.5` only makes that redistribution *more gradual*; **not** license to run a *production* band there (bunched / under-resolved saddle). Re-diff selective-dynamics flags after any manual edit ‚Äî they can silently reset (selective-dynamics entry).
- **Tell.** `grep SPRING INCAR` shows ‚âà‚àí0.5 (too weak), a value raised above ‚àí5 chasing a strong bond, or one lowered toward ‚àí4/‚àí3 chasing convergence; spacing in `nebef.dat`/`neb.dat` clusters at the endpoints.
- **Rule.** Default `SPRING=-5` eV/√Ö¬≤ for solid-state NEB; it tunes image *spacing*, not chemistry and not convergence ‚Äî neither bunched images, a strong adsorbate bond, nor a stalled band justifies deviating, and it *self-corrects* unequal spacing from manual image edits. (Expert-stated 2021‚Äì2023; manual-edit spacing-restoration MID/2011, convergence-stall misread OLD/2010, sign-irrelevant + variable-spring-removed‚ÜíCI-NEB OLD/2007 ‚Äî consistent; not version-specific.)

## Non-fatal `INTERNAL ERROR` inside the electronic SCF ‚Äî check recovery before changing tags
An `INTERNAL ERROR` printed *inside the electronic minimization* (mid-SCF, not at launch) is frequently **non-fatal**: VASP reports it and then continues, and the ionic run still converges. So the first move is *not* a tag change ‚Äî confirm whether the SCF recovered and ionic steps kept advancing.
- **Tell.** The error sits within an electronic-step block in OUTCAR/stdout and steps keep coming after it: `grep -n 'INTERNAL ERROR' OUTCAR`, then check `F= `/`reached required accuracy` appear past that line. Distinct from the *fatal* `us.F` internal error / segfault at **launch** (a diverging, never-relaxed endpoint ‚Äî see neb-diagnostics.md), which never recovers.
- **Fix.** Only if it keeps biting (no convergence): switch the electronic minimizer ‚Äî change `IALGO`/`ALGO` (e.g. RMM-DIIS ‚Üî blocked-Davidson) ‚Äî and/or soften charge mixing (`AMIX`, `BMIX`, plus `AMIX_MAG`/`BMIX_MAG` when spin-polarized). It's an electronic-convergence instability, system/algorithm-dependent.
- **Rule.** A mid-SCF `INTERNAL ERROR` the run survives is usually electronic-minimizer noise, not a structural/VTST bug ‚Äî verify the calculation continued and converged first; only if it persists, change `IALGO` or soften mixing.

## `LCLIMB` is for the saddle, not the SCF ‚Äî set it from the outset; the DAV "not hermitian" warning is a diagonalizer issue, independent of it
`LCLIMB=.TRUE.` is the VTST **code default** *and* the right setting for *any* NEB/SS-NEB whose goal is a saddle: it inverts the highest image's band-parallel force so that image climbs to the TS (climbing-image entry, neb-diagnostics.md). **CI-NEB *is* this tag and nothing more** ‚Äî not a separate code, build, or version, and needing no recompilation, so a user asking "which version is CINEB / how to set it" just adds `LCLIMB=.TRUE.` (if plain NEB runs in your VTST binary, so does CI-NEB). **Set it from the start** ‚Äî CI *aids* convergence with no downside, so the belief it destabilizes a not-yet-converged band (and so must be applied only to a nearly-converged path) is the misconception behind needless two-step plain-NEB‚ÜíCI workflows; a plain-NEB warm-up is *optional economy, never a prerequisite* (for a path *very* far from the MEP, decompose the mechanism ‚Äî `LDNEB` entry below ‚Äî not a warm-up pass). The **mirror** worry ‚Äî that switching `LCLIMB` on *after* a converged plain NEB will **instant-exit** because forces already met `EDIFFG` ‚Äî is equally wrong: inverting the highest image's band-parallel force lifts that image's net force back above the threshold, so the band simply reconverges onto the saddle, no other tag touched. **Don't lower `EDIFFG` to force it to keep running** ‚Äî tighten it only for saddle *accuracy* (`EDIFFG` entry below), never to defeat a non-existent instant-exit. It governs *geometry*, nothing electronic ‚Äî so `LCLIMB`/`LNEBCELL` toggles that seem to summon or banish the `WARNING: Sub-Space-Matrix is not hermitian in DAV` are **coincidence, not causation** (blocked-Davidson noise, orthogonal to the flag; Fix/Rule below).
- **Absent/low plain-NEB barrier ‚áí try the climbing image before concluding "no barrier".** A low or narrow saddle can fall *between* images, so a *plain* NEB (no CI) reads barrier-less or implausibly low (flat/monotone near an endpoint) though a TS exists. Turn on `LCLIMB=.TRUE.` ‚Äî the highest image is driven *onto* the saddle, independent of image count ‚Äî before declaring there's none; if still ambiguous, hand off to a dimer (`neb2dim.pl` ‚Üí `ICHAIN=2`, workflows.md), which converges to the saddle independent of band resolution. (First rule out a symmetry-pinned false endpoint ‚Äî ISYM entry above ‚Äî which also reads barrier-less.) **Conversely, a *confirmed* barrierless/downhill path ‚áí turn CI *off*:** because `LCLIMB` is default-on (above), the highest image keeps hunting a saddle that doesn't exist ‚Äî so once you've *verified* no `neb.dat` local max (don't assume), set `LCLIMB=.FALSE.` to run plain NEB, or skip the band entirely (NEB is the wrong tool for a barrierless reaction ‚Äî prefer a relaxed/constrained scan, neb-diagnostics.md).
- **Fix (warning).** A diagonalizer instability ‚Äî fix as the mid-SCF `INTERNAL ERROR` entry above (`ALGO=Fast`/`IALGO`, soften mixing). Keep `LCLIMB=.TRUE.` regardless; one the SCF recovers from is benign.
- **Tell.** `grep -i 'hermitian\|DAV' OUTCAR` (warning during SCF) vs `grep -i 'lclimb\|lnebcell' INCAR` (saddle/SS-NEB intent) ‚Äî unrelated facts. To confirm CI actually *ran*, read the `LCLIMB` echo in the OUTCAR **header**, *not* the word `climb` in the body: the runtime climbing-image message is version-dependent (`node is climbing` ‚Üí `NEB: image is at an extrema` ~VTST 2.03d/2009; newer builds may print an explicit climbing line), so a `grep climb OUTCAR` that finds nothing is **no** proof CI is off.
- **Rule.** `LCLIMB=.TRUE.` is the VTST default for saddle searches, purely geometric, left on from the outset (CI aids convergence; no plain-NEB warm-up needed or beneficial) ‚Äî but for a *confirmed* barrierless/downhill path set `LCLIMB=.FALSE.` explicitly, else the default-on climbing image chases a non-existent saddle. The DAV not-hermitian warning is an electronic diagonalizer issue fixed by `ALGO=Fast`/`IALGO`, never by toggling climbing-image flags. (Expert-stated 2021; CI-from-scratch corroborated 2014; CI-NEB-is-just-`LCLIMB`-not-a-version 2016; barrierless-disable MID/2011; not version-specific.)

## `ALGO` for force-based NEB / dimer / relaxation ‚Äî `Fast` or `Normal`, never `ALGO=All`; for a force-projection NEB band / dimer never `Very_Fast`/`IALGO=48`
`ALGO=All` (all-bands-simultaneous update) is for hard *electronic-structure* problems (hybrids, awkward metals), not the routine SCF inside a force-driven NEB/relaxation ‚Äî there it only inflates per-step cost without better forces. For a first-pass NEB use `ALGO=Fast`, or omit `ALGO` for the `Normal` blocked-Davidson default, with the other cheap-first settings (PREC=Normal, ENCUT near the POTCAR min, sparse k-mesh ‚Äî workflows convergence-economy). Tighten precision only after the geometry converges ‚Äî and `PREC=High` rarely helps the *barrier* itself (a *difference* whose basis/FFT-grid errors cancel across images, like the `ISIF=2` cancellation), so keep `PREC=Normal` and spend any tightening on `ENCUT`/k-density, not `PREC`. The *low* end bites **force-projection** methods: `ALGO=Very_Fast`/`IALGO=48` (RMM-DIIS) is noisier than `Normal`/`IALGO=38` ‚Äî in a **dimer/min-mode** search it corrupts the finite-difference torque/curvature, and on an **NEB band** its SCF instability (worst for **d-/f-metals**, spin-polarized/DFT+U) scatters per-image *electronic* energies so the profile reads **non-symmetric / physically implausible even after `ISYM=0` + a loose `EDIFFG`** (and at coarse k / low `ENCUT`) while the *endpoints* are fine ‚Äî masquerading as an NEB geometry fault. Switch to `IALGO=38`; it's electronic noise, not the band/dimer setup. Fine only for static single-points; see dimer-diagnostics.md. **Parallelization is orthogonal to the saddle search:** `NPAR`/`NCORE`/`KPAR`/`NSIM` only partition the *same* SCF across ranks ‚Äî the VTST dimer/NEB ionic driver is independent of them, so they change *speed, never the result*; set them freely on large cells ‚Äî but **tune them to the *per-image* core count** (each NEB image runs on only total/`IMAGES` cores), so a single-image job's parallel deck doesn't auto-transfer: with few cores/image, `NPAR=1` is the safe default (on VASP 5.x+ use `NCORE`, which superseded `NPAR`), scaling up only if benchmarking shows a gain. Disabling them "to be safe" for a TS run is a misconception, and unlike the `Very_Fast`/`LREAL=Auto` force-noise above they carry *no* accuracy cost (2009 forum; architectural, not version-specific; per-image-tuning OLD/2007).
- **Tell.** `grep -i algo INCAR` shows `ALGO=All` on a force-based NEB/relax deck (usually with PREC=Accurate + ENCUT‚â´400 ‚Äî the over-precise first-pass smell), or `ALGO=Very_Fast`/`IALGO=48` on a dimer; for the NEB case a **non-symmetric/scattered profile surviving `ISYM=0` + loose `EDIFFG`** on a d-/f-metal ‚Äî confirm with `grep -i ialgo OUTCAR` (48) and erratic per-ionic-step SCF iteration counts / `BRMIX`/`EDDDAV` warnings.
- **Rule.** Force-based NEB/relaxation wants `ALGO=Fast` or the `Normal` default; reserve `ALGO=All` for electronic-structure optimization, and for any *force-projection* method ‚Äî dimer/min-mode **and** the NEB band ‚Äî use `Normal`/`IALGO=38`, never the noisier `Very_Fast`/`IALGO=48`: a non-symmetric NEB profile that survives `ISYM=0` + loose convergence is usually RMM-DIIS SCF noise (try `IALGO=38`), not the geometry. (Expert-stated VASP 5.4.1 2022; min-mode caution confirmed 2019; NEB-profile-noise corroborated 2014 / VASP 5.3.3 / VTST 3.0b; not version-specific.)

## NEB barrier from unconverged SCF ‚Äî every image hits `NELM`, so the optimizer trusts wrong forces
NEB forces are only as good as the SCF that produced them. If the electronic loop fails to reach `EDIFF` within `NELM` steps on an ionic step, VASP prints `aborting loop EDIFF was not reached (unconverged)` and hands the ionic optimizer a force from a *non-self-consistent* density ‚Äî yet the band can still show low force norms and read "converged," so a wrong barrier passes silently. Most common on **DFT+U, spin-orbit, or transition-metal** systems run with `ALGO=Fast`/`VF` and no damping.
- **Tell.** `grep 'aborting loop' */OUTCAR` across *every* image dir: lines ending `EDIFF is reached` = good; `EDIFF was not reached (unconverged)` (i.e. `NELM` hit that ionic step) = the SCF never finished. Do this *before* trusting any barrier ‚Äî a force norm at `EDIFFG` does not mean the density was self-consistent.
- **Fix.** Tighten electronic convergence, cheapest first: set `LMAXMIX=4` (d-electrons) / `6` (f) so DFT+U charge-density mixing is correct; reduce `SIGMA`; soften mixing (`AMIX`/`BMIX`, plus `AMIX_MAG`/`BMIX_MAG` when spin-polarized); escalate `ALGO` `Fast‚ÜíNormal‚ÜíAll`; raise `NELM` only if the SCF is genuinely still *descending*, not if it plateaus. Confirm `EDIFF‚â§1e-6` is actually *reached* each ionic step.
- **Reconcile with the ALGO entry above.** Routine force-based NEB wants `ALGO=Fast`/`Normal`, not `All` ‚Äî but a DFT+U/SOC/TM system whose SCF won't reach `EDIFF` *is* the hard-electronic-structure case that entry reserves `ALGO=All` for; escalate only once the `aborting loop` grep proves non-convergence, never as a default.
- **Rule.** For DFT+U (and SOC/TM) NEB, grep every image OUTCAR for `aborting loop ‚Ä¶ unconverged` before trusting barrier heights ‚Äî *and before blaming a slow / persistently-high-force band on the ionic optimizer or settings*: unconverged SCF yields wrong forces the ionic optimizer reports as converged, or that never let it settle. Suspect the electronics (LMAXMIX, SIGMA, mixing, ALGO, EDIFF reached) before touching ionic/optimizer tags. (Expert-stated VASP 5.4.1 / VTST 3.1, 2021; slow-DFT+U-NEB ‚áí electronics-first 2017; SCF-quality-governs-forces not version-specific.)

## `ISYM=0` for the whole NEB *and dimer* ‚Äî symmetry pins endpoints/path into a symmetric subspace AND injects per-image ghost forces; drop `ADDGRID`
Run *every* band image ‚Äî and any **dimer/min-mode** search ‚Äî with `ISYM=0` (`-1` also kills charge-density symmetrization); relax both endpoints with it too, unless verified stable without it (reason 1). Two independent reasons:
1. **Endpoint pinning.** Default `ISYM=2` constrains relaxation to the detected space group, so a symmetry-*breaking* minimum (vacancy, interstitial, adsorbate, surface) freezes at high-symmetry sites and settles *above* its real floor ‚Äî an endpoint higher than its adjacent interior image (penultimate below FS / early below IS ‚Äî MEP-topology, neb-diagnostics.md), or ‚Äî when *both* endpoints are pinned ‚Äî a wholly **barrier-less / monotone-descent** band (no saddle, tangent forces one sign) *even with the band INCAR correctly `ISYM=0`*: the trap is setting `ISYM=0` on the images but **not** the endpoint relaxations, so grep the **endpoint-relax** INCAR, not the band's. Relaxing endpoints *with* symmetry is allowed for efficiency **only if you then verify stability** ‚Äî a near-zero-force `ISYM=0` `NSW=1` step on the relaxed geometry (Rule); non-trivial forces ‚áí symmetry pinned it above its floor, re-relax `ISYM=0` ‚Äî seeding from the lower-energy adjacent image's CONTCAR (`01`/`N-1`) ‚Äî before building the band. And a symmetry-relaxed endpoint coming out **lower** than its `ISYM=0` value is *not* grounds to run the band with `ISYM=2`: `ISYM=2` vs `0` are symmetry-enforced vs symmetry-broken wavefunctions on **different energy surfaces**, so their energies aren't comparable and a barrier mixing them is meaningless ‚Äî keep `ISYM=0` for the endpoints *and* every image.
2. **The MEP breaks symmetry ‚Äî band never exempt.** Even between *symmetric* endpoints the minimum-energy path almost always leaves their symmetry subspace, so a non-zero `ISYM` pins the optimizer *inside* it: the band "respects" the symmetry, never reaches the true asymmetric MEP, and converges poorly. **Worse, VASP runs symmetry analysis *independently per image*** (an interior image generically lacks the endpoints' space group), injecting **spurious "ghost" forces and corrupted image energies** ‚Äî and **zeroing the force outright on any image that lands *on* a high-symmetry point**, so the CI can never climb ‚Üí a band that *reads converged but sits on a false/suspect saddle* (you only escape this when *no* image lies on a symmetry point ‚Äî why `ISYM=2` sometimes seems fine). Tells: image energies **anomalously low / deeply negative** (far below the endpoints or that geometry's static-SCF energy), *or* an **asymmetric profile between symmetry-related endpoints / a spuriously inflated barrier** on a physically symmetric path, with a re-relaxed single image disagreeing with the band profile. (Same corruption hits a dimer's rotated images.)
Also drop **`ADDGRID`** on a saddle-search deck ‚Äî its extra-fine force grid adds SCF cost with no NEB benefit (convergence-economy, workflows.md).
- **Tell.** No explicit `ISYM` (or `ISYM=1/2`), or `ADDGRID=.TRUE.`, in any NEB/dimer INCAR (`grep -E 'ISYM|ADDGRID' */OUTCAR`); a *dimer/min-mode* OUTCAR printing `All symmetrisations will be switched off` / `NOSYMM: ‚Ä¶point group C_1` = symmetry was still **on** and VASP self-dropped it once the displacement broke the point group ‚Äî don't lean on that auto-catch (near a symmetric geometry it can instead *re-impose* symmetry and corrupt the finite-difference curvature): set `ISYM=0` explicitly; an endpoint energy suspiciously high vs its neighbor, **interior-image energies absurdly low / deeply negative, or an asymmetric / over-high barrier on a symmetry-related-endpoint path** ‚Äî and note an identical-but-nonzero `ISYM` in *every* dir won't surface in an INCAR `diff` (distinct from the inconsistent-inputs trap, neb-diagnostics.md).
- **Rule.** Band (and dimer) always `ISYM=0`; endpoints with `ISYM=0`, or symmetry-relaxed then verified by a near-zero-force `ISYM=0` single-point (`NSW=1`); no `ADDGRID`. Pick one `ISYM` for endpoints *and* every image and **never compare energies across `ISYM` settings** (symmetry-enforced vs broken = different surfaces); per-image symmetry injects ghost forces and unphysical energies ‚Äî none of it buys barrier accuracy. (Expert-stated 2017 & VASP 5.4.4, corroborated 2011‚Äì2014 / VASP 5.3 / VTST 3.0; long-standing, not version-specific.)

## Invisible INCAR faults ‚Äî a tag the user swears is set misbehaves: misspelling (silent ‚Üí wrong default), a malformed value (explicit parse error), *or* a per-species list with too few values (silent abort after the first SCF)
Reasons VASP/VTST mis-handles a tag the user can plainly see in INCAR:
- **Misspelling ‚Üí silently ignored ‚Üí wrong default.** VASP discards any *unrecognized* tag **without warning** and uses the hardcoded default ‚Äî *any* tag, nothing complains. A one-char typo (`EDDIFG`/`EDIFG`/`EDDIFFG`) leaves EDIFFG at its *positive* default, so a VTST optimizer aborts `Must set EDIFFG < 0 when using IOPT > 0` while the user stares at a negative value. The wrong name needn't be the user's typo ‚Äî **stale documentation** is a source: when a webpage and the *bundled example INCAR* disagree on a tag's spelling, the example is authoritative (the Lanczos step tolerance is `Sltol`, not the old-doc `STOL`; the dimer finite-difference separation is `DdR` (DdR entry below), not the old-doc `DNdR`; both silently dropped). **Tell:** `grep -i 'edif' INCAR` finds the near-miss; the OUTCAR INCAR-echo lists only *recognized* tags, so the real tag **absent** from the echo = never read (also confirms which spelling the build accepts). *OLD build-side exception (mirror case):* the tag echoed **present but as `0.0E+00`** despite a nonzero INCAR value isn't a line fault but a since-fixed VTST source bug ‚Äî a local `ediffg` in `chain.F` shadowed VASP's `dyn%ediffg` and zeroed it (affected vtstcode < 2.02a, fixed 2007-05) ‚áí upgrade, don't touch the line. **Fix:** correct the spelling; confirm it appears in the echo.
- **Malformed line/value on a *correctly-spelled* tag ‚Üí *explicit* `Error reading item 'IOPT' (any tag) from file INCAR`, with the `VTST: version` banner *present* (binary linked ‚Äî *not* missing-vtstcode; the SKILL.md version reflex splits the same error on that banner).** Two flavors: (a) **trailing tabs/whitespace** after the value ‚Äî invisible in an editor, they break VTST's line parser; `cat -A INCAR | grep IOPT` exposes them as `^I` ‚áí strip them. (b) **a float given to an integer tag** (`IOPT`/`IBRION`/`IMAGES`/`ISYM` = `0.2`) ‚Äî the Fortran integer read chokes on the decimal ‚áí use a bare integer (`IOPT = 0/1/2/3`). The name and value *look* right, so don't blame the build.
- **Per-species list with too few values ‚Üí abort *after the first SCF* (masquerades as a premature stop).** Tags taking **one value per species** ‚Äî `VDW_C6`/`VDW_R0`/`VDW_CN_R0` (DFT-D), `MAGMOM`, `LDAUU`/`LDAUJ`/`LDAUL`, `ROPT` ‚Äî must carry **exactly as many entries as there are species** (POSCAR element-symbol line / POTCAR `TITEL` blocks). Too few and VASP finishes one ionic step, then aborts at the end of that SCF ‚Äî so an **NEB/relax that terminates cleanly after a single ionic step despite `NSW‚â´1`, with no `NELM`-exceeded error and the band machinery (LCLIMB/IMAGES) correct**, is often this, not an NEB failure. **Tell:** count entries on each list vs the species count (`grep -c TITEL POTCAR`); `grep -i 'vdw parameters\|ERROR' OUTCAR` near the top for the parameter echo / parse-failure line. **Fix:** add the missing value(s) so every per-species list matches the species count.
- **Rule.** A tag the user swears is set but misbehaves ‚áí suspect the *line/value*, not the build: grep spelling near-misses (silent ‚Üí wrong-default abort), `cat -A` for trailing tabs, integer tags carrying no decimal (explicit `Error reading item`, binary fine), **and every per-species list (`VDW_C6`/`R0`, `MAGMOM`, `LDAU*`) holding exactly one value per species** (a short list aborts after the first SCF and looks like an NEB stopping early); confirm against the OUTCAR echo. (Misspelling VASP 6.4.2 / VTST 4.1; trailing-tab 2017 / VTST 3.1; float-for-integer RECENT 2023 / VTST 3.2; per-species-count 2016 / VASP 5.x-era; `Sltol`/`STOL` webpage typo fixed 2014, current docs show `Sltol`; `DdR`/`DNdR` website typo circa 2011, since corrected ‚Äî all line-parsing, long-standing not version-specific.)

## `EDIFFG` for NEB ‚Äî must be *negative* (force criterion); positive = energy criterion (band runs indefinitely); `-0.05` canonical, `-0.5` 10√ó too loose; loose EDIFFG also leaves the CI climbing image short of the saddle (CI ‚âà plain NEB)
The **sign** of `EDIFFG` selects the criterion: **negative = force-based** (what NEB needs); **positive ‚Äî or the default ‚Äî = an *energy*-difference criterion** that stops on ŒîE and can leave large residual forces ‚Äî and is *non-rigorous*: ŒîE is optimizer-history-dependent (two runs to the same energy threshold can land at very different geometric convergence), whereas the projected band force rigorously ‚Üí0 at the true MEP, so force is the only well-defined NEB/TS criterion. On a *band* the energy test must be met by **every image simultaneously**, so a positive `EDIFFG` makes the NEB **never terminate** ‚Äî one image's `dE` plateaus near the threshold while others lag, so the collective test never passes; forces, checked per-image, are the right measure. A positive `EDIFFG` on an *endpoint* relaxation is especially insidious: the endpoint reads "converged" on energy yet carries residual gradients, so the interpolation is ill-posed and every image is corrupted from step 1 (and a VTST `IOPT>0` band aborts `Must set EDIFFG < 0`). Among negative values the canonical criterion is `-0.01`‚Ä¶`-0.05` eV/√Ö ‚Äî the routine range for *any* ionic relaxation (plain endpoint, band, *or dimer/saddle search*), with `~-0.001` near the numerical floor and **`‚â≤-0.001` reserved for frequency/Hessian input**; an **ultra-tight `EDIFFG` (e.g. `-0.0001`) on a dimer is the mirror trap to `-0.5`** ‚Äî the saddle search then grinds *hundreds* of steps with force still well above it and the user wrongly reads it as broken (loosen to `-0.01`‚Ä¶`-0.05`, raise `NSW`‚Üí500‚Äì1000 for a complex PES, never tighten further); `-0.5` is 10√ó too loose ‚Äî "converges" fast at ~0.5 eV/√Ö but the MEP/barrier is untrustworthy, so a band **stuck above** `EDIFFG` is never a cue to *loosen* it: first confirm the force is actually *descending* (TIMESTEP entry's stalled-vs-crawling tell) and drive it with the conservative Quick-Min (`IBRION=3 POTIM=0 IOPT=3`, which can't overshoot) ‚Äî only a force staying flat under Quick-Min means the *surface* is genuinely flat/noisy, not the criterion. **CI-NEB needs it tighter still:** the climbing image reaches the saddle *only as its force‚Üí0*, so a loose criterion (even the ~`-0.01` search default) leaves the highest image *below* the true TS ‚Äî the band then reads like plain NEB (CI ‚âà non-CI) or the peak sits off-saddle. Confirm `LCLIMB` is active (OUTCAR `NEB: LCLIMB`; else it's the VTST-not-linked / inactive-LCLIMB case ‚Äî neb-diagnostics.md), then tighten toward `-0.002` to settle it *at* the saddle. **Exception ‚Äî if only the *barrier height* matters (not the full MEP geometry), only the highest/climbing image need meet the force criterion; interior images may stay looser.** Relax both endpoints to the *same* negative criterion.
- **`FMAGFLAG` picks the force norm `EDIFFG` gates.** Default (off): the per-atom **max RMS** force (OUTCAR `FORCES: max atom, RMS` ‚Äî the value `nebef.pl` reads). `FMAGFLAG=.TRUE.`: the **global total RMS** force over all atoms (OUTCAR `FORCE total and by dimension`) ‚Äî the whole-image norm, not the single most-stressed atom. Choose by which you want gating convergence; leave **off** for routine NEB and the reference GSSNEB deck. (Expert-stated 2011; not version-specific.)
- **Tell.** `grep EDIFFG INCAR` for a *positive* value **or no `EDIFFG` line at all** (both = the energy-criterion default ‚Äî the band then just runs out its `NSW` cap unconverged, barrier unverified) or a loose negative one (`-0.1`‚Ä¶`-0.5`); a band that **never stops** despite one image looking energy-converged, one "converged" with per-image force norms still large, or a **CI-NEB whose peak matches the plain-NEB estimate / sits off-saddle** with LCLIMB active, is the giveaway.
- **Rule.** NEB `EDIFFG` must be **negative** (positive = energy criterion ‚Üí residual forces, band never terminates, corrupted endpoints ‚Äî above). Use `-0.05` or tighter, matched on the endpoints; `-0.5` is 10√ó too loose, and the **mirror** ‚Äî an ultra-tight `‚â≤-0.001` (e.g. `-0.0001`) on a *dimer* ‚Äî just grinds hundreds of steps reading as broken (use `-0.01`‚Ä¶`-0.05`, raise `NSW`; reserve `‚â≤-0.001` for frequency). A band stuck above the criterion is never a reason to *loosen* it (force-trend + Quick-Min first, above). **CI-NEB wants tighter (toward `-0.002`)** ‚Äî the climbing image seats the saddle only as force‚Üí0 (loose ‚áí CI ‚âà plain NEB / peak off-saddle); but if only the barrier height is wanted, gate just that image. (Expert-stated 2010‚Äì2021; not version-specific.)

## `NELECT` change ‚Üí BRMIX from stale `CHG*`/`WAV*` (charged-defect NEB)
Changing the electron count (`NELECT`, e.g. neutral ‚Üí charged defect) while leftover `CHGCAR`/`WAVECAR` from the previous (neutral) run sit in the run dirs makes VASP read the *old* count's density. It no longer matches NELECT, so SCF throws BRMIX "very serious problems" at the first few NELM steps ‚Äî old vs new charge density differ by exactly the electron-count change.
- **Fix.** Purge `CHG*` and `WAV*` from *every* image (and endpoint) directory before rerunning with the new NELECT; also set `ISTART=0 ICHARG=2` to build the density from scratch. Deleting the files is the reliable cure ‚Äî stale density can still bite regardless of ISTART/ICHARG.
- **Tell.** `grep BRMIX OUTCAR` firing at low NELM steps; check each image dir for `CHGCAR`/`WAVECAR` presence.
- **Rule.** Any NELECT change for a charged-defect NEB requires deleting stale `CHG*`/`WAV*` in *all* image dirs first. (Observed VASP 6.4.2 / vtstcode-205.)

## `TIMESTEP` (quickmin/FIRE) ‚Äî step-size tuning: too large oscillates, too small crawls, high-force start overshoots
`TIMESTEP` is the integration step for the velocity-based VTST optimizers ‚Äî quickmin (`IOPT=3`) and FIRE (`IOPT=7`); default `0.1`. **`POTIM` is the analogous step-size tag for VASP's *built-in* optimizers (`IBRION`; `IOPT` absent/0); the two are not interchangeable ‚Äî which one is live is set *solely* by whether `IOPT‚â•1`, and current builds silently ignore the inactive tag.** `IBRION=3`+`POTIM=0` thus optimizes *only* through the VTST `IOPT` interface ‚Äî on **stock/un-patched** VASP (no `IOPT`) that combo takes **zero steps**, the run finishing with no ionic motion (the VTST-not-linked tell ‚Äî SKILL version reflex), so stock `IBRION=3` needs `POTIM>0` (IOPT-tiers native-`IBRION` analog). So `POTIM` must stay `0` under any `IOPT` optimizer ‚Äî and set it *explicitly*: an absent or misspelled (`OPTIM`) `POTIM` takes VASP's *nonzero* default, so `grep -i potim INCAR` must show a real `POTIM=0`, not a missing/typo'd line. A nonzero `POTIM` under `IOPT>0` **double-stepped in older VTST** ‚Äî VASP's built-in mover added a step on top of the VTST step, so **NEB forces spike and explode within 2‚Äì3 ionic steps** (worst near a steep barrier); newer builds ignore it instead, but set `POTIM=0` regardless. It mis-tunes in *several* ways, and the step is the first thing to check before declaring an NEB non-convergent:
- **Too large** (‚â•0.1) once the band carries a stiff high-frequency mode ‚Äî a light-atom bond (CO‚ÇÇ/CO/H‚ÇÇO/NO, C‚ÄìH/O‚ÄìH/N‚ÄìH stretches), *the stiff curvature near the saddle itself*, **or a *raised* `ENCUT`** (counter-intuitive ‚Äî a deck converging at *low* cutoff oscillates at *high*, the higher cutoff stiffening the PES; seen in SSNEB): the optimizer over-steps it, so the band **oscillates near the energy maximum without settling** / converges slowly / stalls, *or a subset of images diverges (forces stay high), or an image's max force spikes ‚Äî from small/benign up to tens of eV/√Ö ‚Äî at a **regular ionic-step cadence** then drops back: the periodicity is quickmin's velocity-accumulate‚Üíovershoot‚Üívelocity-zero cycle (the reset on each overshoot gives a recurring transient force rise) ‚Äî benign, it doesn't block convergence but grows more pronounced at tighter thresholds.* **First confirm the electronic SCF reaches `EDIFF` each ionic step** (a high cutoff can itself destabilize the SCF ‚Üí noisy gradients ‚Äî unconverged-SCF entry); *then* reduce the step 3‚Äì5√ó (`0.05` or lower; ~`0.02` for a light/gas-phase adsorbate; ~`0.01` if it oscillates near the TS or at high `ENCUT` ‚Äî accept slow convergence) to limit velocity buildup, keeping `SPRING=-5` and `IOPT` unchanged ‚Äî the step is the fix, **not `MAXMOVE`** (it caps step *length*, not velocity buildup, so it's the wrong handle for the velocity cycle) and not the spring or optimizer (for LBFGS `IOPT=1` the analogous lever is a smaller `INVCURV` ‚Äî IOPT entry).
- **Too large for a high-force *start*** ‚Äî when the *initial* band carries large forces (‚â´ a few eV/√Ö, e.g. 10‚Äì30 eV/√Ö at iterations 1‚Äì5, from a rough linear path), the default `0.1` overshoots every early step and the band never settles. ‚Üí *Temporarily* drop `TIMESTEP‚âà0.01` for the first ~20 iterations until **all** image forces fall below ~1 eV/√Ö, then restore `0.1` and **continue**. Stabilising the first steps outranks image count or optimizer choice. (Interim cure; cf. the `MAXMOVE` cap, lifted once forces drop; durable fix = IDPP band ‚Äî workflows.)
- **Too small** (‚âà0.03), *or simply needing more steps on final approach*: the band descends *glacially*, max force plateauing just above threshold (~0.1 eV/√Ö) ‚Äî *looks* stalled but still slowly descending; **a barrier energy still dropping step-to-step is the decisive "not stalled" tell even when the force has flatlined near `|EDIFFG|`**. ‚Üí raise `TIMESTEP` to ~0.1 *if* the step is the cause, and/or just **continue** the *same* trajectory ‚Äî not a restart, an IOPT switch, *or a tighter `EDIFFG`* (a 2nd-order jump mid-stall doesn't fix it; IOPT-tiers entry).
- **Tell.** `grep 'OPT: QM, TIMESTEP' OUTCAR` shows the step used (absent/commented ‚áí 0.1 default). To split "stalled" from "crawling," compare max force over the last several ionic steps (`grep 'F= ' OSZICAR`) ‚Äî still trending down ‚áí converging, just slowly.
- **Rule.** Before blaming NEB non-convergence on the optimizer, check `TIMESTEP`: ~0.03 is over-conservative (merely slow), ~0.1 the typical start and orders of magnitude faster on the same trajectory. Tune the step and *continue* ‚Äî don't restart or switch `IOPT`.

## `MAXMOVE` ‚Äî cap the per-step displacement: crawl a strained NEB band, and balance dimer translation against `DRotMax` rotation
`MAXMOVE` bounds the largest Cartesian step any VTST `IOPT` optimizer takes per ionic step ‚Äî a trust-radius *ceiling* (normal 0.1‚Äì0.5 √Ö), not a convergence dial. A strained linear-`nebmake.pl` band (an unphysically short pair, ~1.2 √Ö, below the covalent-radius sum ‚Äî linear-overlap entry, neb-diagnostics.md) gives huge first-step forces, so a default step *overshoots* and the run diverges/crashes on iteration 1.
- **Tell.** First-iteration max force ‚â´ later steps; an image POSCAR with a too-short pair; step-1 divergence with all else correct.
- **Dimer corollary ‚Äî balance `MAXMOVE` (translation) against `DRotMax` (rotation).** In a dimer `MAXMOVE` caps how far the *center* translates per step; size it against `DRotMax` so each translation turns the dimer only a *few degrees*. A `MAXMOVE` too large relative to `DRotMax` lets the center outrun the rotation, so the orientation stops tracking the lowest mode between steps and the **curvature estimate is corrupted** ‚Äî the complement of the *over*-rotation case (many sub-steps per translation ‚áí *lower* `DRotMax`, dimer-diagnostics.md).
- **Rule.** NEB: leave `MAXMOVE` at default and raise `TIMESTEP` for speed (TIMESTEP entry); tighten to ~0.05‚Äì0.1 √Ö *only* to crawl a strained linear band until forces relax (interim cure, interchangeable with dropping `TIMESTEP`); durable fix is IDPP (`nebmake.py --idpp`). Dimer: keep translation and rotation in step ‚Äî a few degrees per translation ‚Äî so curvature stays well-resolved. (Expert-stated 2011‚Äì2023; not version-specific.)

## `DdR` ‚Äî dimer separation is the *half*-step: images at POSCAR ¬± DdR¬∑MODECAR, so full image‚Äìimage distance is 2¬∑DdR
The two dimer images straddle the center as **IMAGE1 = POSCAR + DdR¬∑MODECAR, IMAGE2 = POSCAR ‚àí DdR¬∑MODECAR** with **MODECAR a *unit* vector** ‚Äî so `DdR` is each image's displacement *from the midpoint* and the full image-to-image distance is **2¬∑DdR**. Common misreads: `¬± 0.5¬∑DdR` (halves the intended separation) or a non-normalized MODECAR (breaks the distance meaning). `DdR` is the curvature finite-difference step, **not** a rotation knob (dimer-diagnostics.md torque entry).
- **Value & system size.** Canonical **0.001‚Äì0.01 √Ö**. A **local** process keeps a small DdR regardless of cell size; only when **most atoms move** does a larger DdR help keep per-atom displacements comparable ‚Äî a bigger system per se does *not* need a bigger DdR (per-atom displacement, not atom count, is what matters).
- **Small DdR ‚áí tighter `EDIFF`.** Shrinking DdR shrinks the force *difference* the curvature reads, so SCF noise can swamp it: to push to a very small per-atom displacement (~0.001 √Ö) set `EDIFF‚â§1e-8` (the noise-tell escalation, EDIFF entry above).
- **Rule.** DdR = per-image displacement from the dimer center (full separation 2¬∑DdR); build images POSCAR ¬± DdR¬∑MODECAR with a *unit* MODECAR. Default 0.001‚Äì0.01 √Ö, enlarge only when most atoms move, and pair a very small DdR with `EDIFF‚â§1e-8`. (Expert-stated 2011; definition not version-specific.)

## `IOPT` optimizer tiers + efficiency ranking ‚Äî Quick-Min establishes; global-LBFGS/FIRE finish, and are *far* faster
Optimizer choice governs convergence **efficiency/stability, never MEP accuracy** ‚Äî every well-converged band yields the *same* path, so `IOPT=1/2` aren't "more accurate" than `3/7`, only faster near the minimum. It keys off the **force landscape** (rough vs near-minimum) and system size ‚Äî **not the electronic-structure level**: `IOPT`/`IBRION`/`POTIM` act on *ionic forces* however those forces are computed, so a band's optimizer carries over unchanged from GGA to a hybrid/HF (B3LYP) calc ‚Äî the ionic optimizer is method-independent (the *electronic* `ALGO` *does* change for hybrids ‚Äî its entry). The VTST optimizers split into tiers: velocity-based first-order `IOPT=3` (Quick-Min) / `IOPT=7` (FIRE) are robust *far* from the minimum (early/rough band) but slow once forces are small; quasi-Newton second-order `IOPT=1` (LBFGS) / `IOPT=2` (CG) exploit curvature to sharply cut final-approach iterations. **Efficiency when stable ranks global-LBFGS (`IOPT=1 LGLOBAL=.TRUE.`) > FIRE (`IOPT=7`) > Quick-Min (`IOPT=3`)**: `LGLOBAL=.TRUE.` treats the *whole band* as one system so LBFGS exploits *inter-image* curvature ‚Äî the most efficient finisher (benchmark JCP 128 134106); FIRE needs no curvature info, a strong curvature-free alternative. **Forum myth:** the most-recommended Quick-Min is actually the *least* efficient ‚Äî don't linger on it; switch to FIRE/global-LBFGS once the band looks physical.
- **Use.** Drive a fresh/rough band with `IOPT=3` until the path is physically reasonable (single barrier, no image cross-overs) **and forces have fallen to ~0.05 eV/√Ö**; then switch to `IOPT=1 LGLOBAL=.TRUE.` (global-LBFGS) or `IOPT=7` (FIRE) to finish ‚Äî **the second-order optimizer is the lever for the slow final approach, not a re-tuned first-order step: changing `POTIM`/`TIMESTEP` mid-run on a velocity-based optimizer only perturbs accumulated velocity and transiently raises forces (the barrier shouldn't change, only the convergence rate).** Keep `IBRION=3 POTIM=0` under any IOPT optimizer. **Dimer corollary:** when a *dimer* oscillates/diverges on a rugged/anisotropic saddle and convergence stability outranks speed, drop to `IOPT=3` first ‚Äî Quick-Min's velocity-projection takes no uphill step (the most conservative VTST optimizer, rarely diverges), trading more steps for robustness vs LBFGS/CG overshoot.
- **Premature LBFGS *diverges or stalls* ‚Äî Quick-Min-precondition either way.** LBFGS builds curvature from force *history*, so a badly-conditioned start fails two ways. *Diverges:* `IOPT=3 ‚Üí 1` while forces are large (‚â´ a few eV/√Ö, far from a basin) makes them *grow* (>1 ‚Üí >10 eV/√Ö) and distorts geometries ‚Äî small `INVCURV` can't rescue it; *many soft DOF* (big adsorbate on a flexible surface, free molecule) is badly-conditioned the same way, *spiking* 1‚Äì2 orders step-to-step even from ~0.2 eV/√Ö, *worse as you tighten*. *Stalls (milder):* a band that converged easily to a *loose* `EDIFFG` (‚âà‚àí1) barely progresses toward a *tight* one (‚àí0.02) on `IOPT=1` alone ‚Äî the loose-path history seeds too poor a Hessian, and `INVCURV` tweaks don't help. Same cure: stay on/return to **`IOPT=3` (`TIMESTEP`‚âà0.01 if forces start high) until forces fall well below ~1 eV/√Ö (~0.3 for the divergent case), then hand to `IOPT=1`** with a small `INVCURV` (below); tighten `EDIFF`‚â§1e-7 first (2nd-order amplifies SCF noise). Proof it's the optimizer: **the same band on `IOPT=3` converges cleanly.** *Third trigger ‚Äî too coarse a band:* `IOPT=1`/`2` demand a *smoother* band than Quick-Min, so a coarse band QM converged can diverge on LBFGS/CG ‚Äî **add 1‚Äì2 images** (more beyond the stable minimum don't help) before blaming the start (OLD/2006, empirical-potential; qualitative ‚Äî exact delta system-dependent). (Distinct from *ridge-trapped* below ‚Äî forces already *low*.)
- **Not a stall remedy.** Switching tiers only speeds a *healthy* trajectory; it does NOT cure a timestep- or walltime-induced *crawl* stall ‚Äî fix `TIMESTEP` / extend walltime and *continue the same trajectory* (TIMESTEP entry; wall-time entry in neb-diagnostics.md). Likewise, when *several* IOPT values (0/2/3/7) all fail, the optimizer is rarely the culprit ‚Äî suspect upstream structure: recycled intermediates (selective-dynamics entry above), too-loose `EDIFFG`, or unrelaxed endpoints (neb-diagnostics.md). Robust default for a fresh CI-NEB: `IBRION=3 POTIM=0 IOPT=3 TIMESTEP=0.1`; fix structure/criterion before changing it. **Exception:** a *ridge-trapped* band (low forces, stuck-high barrier) isn't a crawl stall (INVCURV below).
- **`INVCURV` ‚Äî initial inverse curvature for LBFGS (`IOPT=1`).** Default `0.01` (value version-dependent ‚Äî older VTST seeded ~`0.1`; confirm in the bundled INCAR docs). The first LBFGS step is steepest-descent `dR = F¬∑INVCURV`, so INVCURV *scales that first step* = 1/(initial-curvature guess, eV/√Ö¬≤): match it to system stiffness ‚Äî *smaller* for stiff/high-curvature surfaces (‚âà1/max-curvature), *larger* (toward `0.1`) if first steps are too timid and convergence is glacial. An `IOPT=1` run on the bare default (no explicit `INVCURV`) can over-step a rough band ‚Äî forces spike (>1, even >10 eV/√Ö), no trend; **lower `INVCURV` below `0.01` first** (‚âà`0.001` for the LBFGS *finishing* stage on a Quick-Min-preconditioned band ‚Äî a fast, stable approach to a tight `EDIFFG`), dropping to `IOPT=3` only if that won't settle it (the badly-conditioned far-from-basin/soft-DOF case above). For the opposite ‚Äî a band *trapped on a ridge* far from the saddle ‚Äî `IOPT=1 INVCURV‚âà0.005` drives it off fastest of all optimizers (judge by *barrier descent*, not force; transient spikes to tens of eV/√Ö expected ‚Äî neb-diagnostics.md ridge-trapped).
- **`LLINEOPT`/`LGLOBAL` ‚Äî JCP-paper names ‚Üî INCAR tags; the "a NEB has no Hessian" non-issue.** The optimizer names in JCP 128 134106 map onto VTST tags: *global-LBFGS* = `IOPT=1 LGLOBAL=.TRUE.` is the inverse-**Hessian** form ("GL-BFGS(hess)"); set `LLINEOPT=.TRUE.` to switch it to the **line-search** BFGS variant. Don't be put off that a NEB *band* has no single formal Hessian ‚Äî the inverse Hessian LBFGS accumulates from force history is an *algorithmic convenience*, not a formal object, and still converges the band correctly; the approximation is safe to use. (Expert-stated 2011; tag map & argument not version-specific.)
- **`IBRION=2` (VASP CG) is forbidden on *any* force-projection method ‚Äî the NEB *band* AND a dimer/Lanczos.** Its energy-based line search assumes F=‚àí‚àáE, but a *projected* force (NEB or dimer min-mode) is *non-conservative*, so CG missteps ‚Äî **terminating early** (erratic force, no steady descent) *or*, on the band, **failing to terminate**: CG lets one image stop independently, breaking NEB's *collective* stopping criterion, so the run stalls / burns all `NSW` while some images read "converged". Safe instead **on the band**: any VTST `IOPT`, or `IBRION=1`/`IBRION=3` ‚Äî all force-driven. A common (old-forum) ladder wrongly slots `IBRION=2 POTIM=0.1` as a "faster, still safe" *middle* tier (moderate forces, between `IBRION=3` for high forces and `IBRION=1` near convergence) ‚Äî it fails for the same non-conservative-force reason; use `IBRION=1` for that rung (`1` vs `2` both use curvature, so picking `1` is energy-consistency, not cost). **A dimer/Lanczos is stricter:** its CHAIN driver *requires* `IOPT>0`, rejecting `IBRION=1` too ‚Äî aborts `CHAIN: Must set IOPT>0 to use the Dimer method` (a code restriction, not physics; `IOPT=1` LBFGS ‚âà `IBRION=1` for one image). Drive it with `IBRION=3 POTIM=0` + any `IOPT>0`. Fine for plain *endpoint* relaxation. **Scope:** a VTST `IOPT‚â•1` optimizer is itself a valid drop-in for VASP's built-in optimizer on *any* run ‚Äî ordinary ionic relaxation as much as an NEB band ‚Äî so dimer/Lanczos are the *only* methods that **mandate** it; everywhere else it's an optional (often faster) alternative (architectural, holds VASP 5/6). Gotcha: a stray `IBRION=2` *earlier* in INCAR silently overrides a later `IBRION=3` ‚Äî `grep -n IBRION INCAR`, delete the duplicate. (Expert-stated as early as OLD/2006 (the IOPT optimizer split was a CVS-development feature then, the then-promised force-only CG long since superseded by the now-standard `IOPT` LBFGS/FIRE/QM optimizers above; pre-VASP-5), reported VASP 5.2 2012, confirmed 2018 & 2020; non-conservative-force argument not version-specific.)
- **VASP-native `IBRION` analog (`IOPT=0` ‚Äî an NEB on VASP's own optimizer, or a plain endpoint relax; `POTIM>0`).** Same robust‚Üísecond-order tiering *off* the VTST path: the second-order `IBRION=1` (RMM-DIIS) started with **high** forces *diverges* ‚Äî CONTCAR images scatter though EDIFF is met (quasi-Newton assumes a near-harmonic PES, unstable far from a minimum). So a **chaotic/unphysical band from a rough linear interpolation** (esp. near point defects/vacancies or large displacements ‚Äî neb-diagnostics.md) is **tamed first with damped-MD `IBRION=3`** ‚Äî the live step is `POTIM`: for the *band* `POTIM‚âà0.1` (`0.01‚Äì0.05` if forces start high), **not** the MD/relax-scale `0.5‚Äì1`, which overshoots (forces *rise* not fall early, energies blow up tens of eV); hold it small until forces fall to ~1 eV/√Ö *and geometries look physical*; only then a fast finisher ‚Äî native `IBRION=1` (‚â≤~0.2 eV/√Ö), or enable a VTST `IOPT` optimizer (`POTIM=0`). (damped-MD taming expert-stated 2008‚Äì2022; not version-specific ‚Äî the LBFGS/CG/FIRE `IOPT` finishers it precedes were new circa 2008, now standard.)
- **Rule.** Quick-Min establishes (can't overshoot); global-LBFGS (`IOPT=1 LGLOBAL=.TRUE.`)/FIRE finish *far* faster ‚Äî switch once forces are small, never *early* (LBFGS diverges off a high-force/soft band, or stalls taking a loose band to a tight `EDIFFG`) nor to cure a *crawl* stall. (Tiers VASP 5.4.4 / vtstcode IOPT; ranking, global-LBFGS & two-stage handoff expert-stated 2018, JCP 128 134106; ridge/soft-DOF 2021; method-independence ‚Äî optimizer ‚ä• electronic-structure level ‚Äî expert-stated 2017; optimizer ‚ä• MEP *accuracy* (efficiency/stability only) expert-stated 2015; two-stage handoff ‚Äî rough first-order ‚Üí second-order finisher once forces are small ‚Äî corroborated OLD/2007, when early-VTST `IOPT` numbering differed: verify current IOPT codes against the bundled INCAR docs.)

## `LDNEB` = **swDNEB** (switched-off double-nudging), not original Wales DNEB ‚Äî skip it for DFT-cost NEB
`LDNEB=.TRUE.` in VASP/VTST is the *switching-function* double-nudging (**swDNEB**, JCP 128 134106), **not** the Wales-group original. The original DNEB keeps a long, multi-minimum path *straight* early but **can't converge tightly** (Wales follows a single mode near the saddle); swDNEB's switching function turns the extra force *off* as the path converges, so it *does* reach a tight `EDIFFG`. Even so, **don't reach for it on a DFT band** ‚Äî double-nudging earns its keep only on *cheap empirical* potentials over *long*, many-minimum paths. Two reflexes mistake it for a fix: a **circuitous/non-converging band** ‚Üí decompose the path into elementary steps through actual intermediates (neb-diagnostics.md multi-step entry); an **oscillating non-TS image fluctuating between two configurations near convergence** the "double/variable spring" reflex hopes to damp ‚Üí a `TIMESTEP`/optimizer/`EDIFF`/geometry problem (TIMESTEP ¬∑ IOPT ¬∑ EDIFF entries). Not a band-force variant either way.
- **Rule.** `LDNEB=.TRUE.` is swDNEB (switched off ‚Üí converges tightly), not original double-nudging (which can't). Reserve it for long multi-minimum paths on cheap potentials; on a DFT NEB skip it ‚Äî decompose a circuitous band, treat an oscillating near-converged image as an optimizer/timestep/EDIFF/geometry issue. (Expert-stated 2014; swDNEB is what VASP ships. An older 2011 framing ‚Äî LDNEB "for long biological paths," bringing "instabilities as forces approach zero" ‚Äî describes the *original* DNEB that swDNEB superseded; not current. The current code is not version-specific, but one **OLD** caveat: pre-2009 VTST had a normalization bug in `neb.F`'s `vproj` helper ‚Äî the double-nudging projection divided by |v| instead of |v|¬≤ ‚Äî so DNEB runs could blow up with unphysical forces; fixed in VTST CVS Feb 2009 and irrelevant for any modern build. An exploding DNEB on a current VTST is method/inputs (above), not this; only on an *ancient* build suspect the version and upgrade before blaming inputs.)

## External stress in NEB ‚Äî `PSTRESS` is hydrostatic-only (shear needs ss-NEB `express`); and at finite `PSTRESS` the barrier is the *enthalpy*, not the OUTCAR energy
VASP's native `PSTRESS` applies only **isotropic (hydrostatic) pressure** ‚Äî no off-diagonal components, so it *cannot* impose shear/anisotropic stress. For that, use **variable-cell ss-NEB** (VTST / tsase ssneb), which carries the full 3√ó3 stress tensor as the `express` matrix, applied through the **cell DOF**, never by moving atoms. Kills four reflexes:
- *"`PSTRESS` can apply my shear/anisotropic stress."* No ‚Äî hydrostatic only; switch to ss-NEB and put the off-diagonal/anisotropic terms in `express`.
- *"Vacuum in z blocks constant-stress ssneb / I must apply stress via atomic distances."* False ‚Äî vacuum is no obstacle. Build an **anisotropic** `express`: zero the z (and unwanted off-diagonals), keep the in-plane terms ‚Äî z unchanged, in-plane sliding barrier under stress.
- Constant stress ‚âÝ rigidly shifting a layer by position. To just displace atoms, do that directly; a *constant stress* lives in the cell DOF.
- **At finite `PSTRESS` the barrier is the enthalpy H=E+PV, not the bare DFT energy.** `nebbarrier.pl` (and any script reading `OUTCAR` energies) takes bare `E` and omits the `PV` term, so endpoint/barrier energies come out inflated/inconsistent vs plain single-points ‚Äî misread as ss-NEB introducing error when it's an enthalpy-vs-energy confusion. **Fix: read the enthalpy line (E+PV) from every image's `OSZICAR`, endpoints included**, not OUTCAR energies; `nebbarrier.pl` does not auto-account for `PSTRESS`. (Thermodynamic, not a code bug ‚Äî verify whether current vtstscripts handle `PSTRESS` before trusting the script at finite pressure.)
**Where to look.** `grep -i pstress INCAR` (present ‚áí hydrostatic-only, can't deliver shear ‚Äî *and* its barrier must come from OSZICAR enthalpies, not `nebbarrier.pl`); `grep -i enthalpy */OSZICAR`; `grep express` in the tsase ssneb source. (Expert-stated 2022; enthalpy-under-`PSTRESS` / `nebbarrier.pl`-omits-`PV` expert-stated 2014; both long-standing, not version-specific.)
