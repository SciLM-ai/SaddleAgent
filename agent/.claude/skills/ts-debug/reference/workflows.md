## Convergence economy — relax geometry at *cheap* electronic settings first, then tighten to converge the *barrier*
**Symptom.** Endpoint/NEB relaxations run hundreds of ionic steps, repeatedly hit wall-time, and never converge — while the deck carries high-precision *electronic* settings (ENCUT far above the POTCAR recommendation, semi-core/extra-electron pseudopotentials, a dense k-mesh) and the endpoints are still geometrically far from their minima.
**Cause.** Over-precise electronic settings inflate the cost *per ionic step* — both wall-time *and* per-rank RAM — by orders of magnitude, so the relaxation times out (or segfaults on memory — allocation entry) before the ions settle; the NEB then inherits poorly-relaxed endpoints and can't converge the band (see neb-diagnostics us.F / partially-relaxed-endpoint entry). Production accuracy (dense k-mesh, high ENCUT, hard `_pv`/`_sv` pseudos, spin) is premature on a path search — the band only locates the saddle region; spend precision later, at the dimer/single-image refinement. A barrier is an energy *difference* along one PES, so DFT errors cancel and it's robust to absolute-energy precision: `PREC=Normal` + soft (non-`_pv`/`_sv`) pseudos + `ENCUT≈300 eV` converge the *barrier* even where absolute energies aren't, so high `PREC`/`ENCUT`/hard pseudos are genuine *overkill* for PES exploration, not merely slow — reserve them to reconverge the *important* saddle(s) once found.
**Procedure — geometry first, precision second.**
1. Relax both endpoints (and run the band) at *minimal* electronic settings: standard (not `_pv`/`_sv`) pseudopotentials, `PREC=Normal`, `LREAL=Auto`, ENCUT near the POTCAR minimum, `ISPIN=1` if non-magnetic, a loose `EDIFF`, gamma-point or the coarsest k-grid that gives qualitative physics — for an isolated molecule / gas-phase species in a box, Γ-only is *correct* (no k-dispersion to sample), not merely cheap, so run the `vasp_gam` binary.
2. Confirm both endpoints reach the `EDIFFG` *force* criterion at this cheap level (see endpoint-construction entry below).
3. *Then* tighten ENCUT, KPOINTS, and pseudopotential richness **one DOF at a time**, each time re-checking the *barrier* (not the per-step force) has converged — and, *before building the band*, that the **endpoint energy difference** (relative IS↔FS stability) is converged in those same settings: if relative stability **flips sign** when you bump ENCUT (~×1.3) or refine the k-mesh, the settings are unconverged: energies from *different* settings are never comparable — never mix settings between endpoints or between images. **On a *restart at higher accuracy*** (`PREC`/`ENCUT`/`KPOINTS`/`ISPIN`) this means **reoptimizing both endpoints at the new settings before recomputing the barrier**: a barrier that shifts dramatically across the upgrade is *stale endpoints* — left at the old accuracy, so their reference energy no longer matches the new images — **not** a method error. Relative barriers are insensitive to accuracy level *once* every image, endpoints included, shares the settings.
**Rescue a band that already stagnated/diverged.** When a band is genuinely stuck — forces frozen high (>~1 eV/Å) or *growing*, geometries going unphysical (≠ merely crawling toward `EDIFFG`, which is the TIMESTEP/walltime *continue*-the-trajectory case — parameters.md / neb-diagnostics wall-time entry) — don't chase it with optimizer switches (flipping to LBFGS only overshoots; parameters.md IOPT-tiers). Reset to minimal precision: drop to PREC=N, ENCUT≈POTCAR default, **re-relax both endpoints** there to a tight `EDIFFG≈-0.01`, **rebuild the band from the re-relaxed endpoints** (`nebmake.py`), and restart CI-NEB with QuickMin (`IOPT=3`, `TIMESTEP=0.1`) at that same cheap precision. Only once the cheap band is stable, restore production precision and re-run to *verify* the barrier.
**Rule.** Converge endpoint/NEB geometry at the cheapest electronic settings that give sane physics, then tighten ENCUT/KPOINTS/EDIFF/pseudopotentials one DOF at a time to converge the *barrier* (and first the endpoint energy *difference* — a relative-stability sign-flip across settings is unconverged, never physics; any accuracy *upgrade*, incl. an NEB restart, re-converges the **endpoints** at the new settings before the barrier is read — a barrier that jumps on the upgrade is stale endpoints, not method error) — staging this way is also *diagnostic* (shows which DOF actually moves the barrier, so precision is spent only where it matters). A relaxation that perpetually times out is usually over-precise, not genuinely non-convergent. (Contrast the neb-diagnostics wall-time entry: a *healthy* still-descending trajectory just ran out of time — extend the allocation, don't touch parameters; here each step is too costly, so *reduce* precision first.)

## Running the vtstscripts — on PATH (helpers co-located), invoked *bare* from the NEB main dir; crash-on-missing-module vs silent-empty-output
**Symptom — three distinct failures.** **(A) Crash on a missing *sibling* module** — the shell found the script on PATH, then it dies: Perl (`nebmake.pl`, `modemake.pl`, …) → `Can't locate Vasp.pm in @INC`; python (`nebmake.py`, `dymseldisp.py`, …) → `ImportError: cannot import name X from aselite` (even with `pip install ase` done). **(B) Silent empty output** — a script (e.g. `vfin.pl <subdir>`) runs with *no error* but its target dir is created *empty*. **(C) Silent *partial* output** — a script writes most of its output but one *column* is all-zero: classically `nebresults.pl`/`neb.dat`/`neb.file` column 4 (cumulative path **distance**) reads `0.000000` on every image while the force/energy columns look normal.
**Cause.** **(A)** The scripts share helper modules — Perl `Vasp.pm`, python's *bundled* `aselite` (**not** the pip `ase` package — installing `ase` does nothing) — that must sit *in the same directory* as the script; run from/copied to a split-off location, Perl's `@INC` lacks the vtstscripts dir, or python's dir is on PATH but **not** PYTHONPATH (PATH launches the script; it doesn't resolve modules at runtime). **(B)** Called with an *explicit path prefix* (`perl /path/vtst/vfin.pl …`) and/or from the *wrong CWD*, so it can't find the image subdirs (`00/`,`01/`,…) it expects in the current directory — does nothing, exits 0. (A *post-processing* script can empty its output the same way *internally*: a case-sensitive grep mismatch between the script's hardcoded keyword and the OUTCAR — a CVS-era `nebconverge.pl`/`nebresults.pl` searched `Forces: m` while VASP wrote `FORCES: m`, matched no force lines, and produced an empty `fe.dat` ⇒ gnuplot `no data point found`. Long fixed (affected CVS-era vtstscripts ~2007); but the reflex generalizes: when a vtst post-processing script runs clean yet writes *empty* output, compare its hardcoded search string against a live OUTCAR with `grep -i` before suspecting the run.) **(C)** `nebresults.pl` shells out to a co-located *helper script* to compute image-to-image distances; on a *cherry-picked* install (only `nebresults.pl` copied out, its helpers not in the same dir / on PATH) that helper isn't found, returns nothing, and the distance column silently fills with zeros — the band ran fine, only the readout is incomplete. (≠ a *2-column* `neb.dat` = unlinked binary, or a zero *force* column = patch↔scripts mismatch — both neb-diagnostics.)
**Fix.** First make the scripts executable — untar/copy can strip the execute bit, so if any fails `permission denied` run `chmod u+rx *.pl` in the vtstscripts dir. Keep the *whole* bundle in a **single** directory (every script beside its helpers) on PATH — a bare `command not found` (or empty output dirs) means the dir was never added: put it on PATH in your shell rc (tcsh `setenv PATH /path/vtstscripts:$PATH` in `.tcshrc`; bash `PATH=/path/vtstscripts:$PATH; export PATH` in `.bashrc`), then re-login/source — never copy one script *out* of the bundle, **nor *into* each run/calculation dir** (install once; scripts resolve via PATH, they are not per-directory installs); for **python** also add that dir to PYTHONPATH (`export PYTHONPATH=$PYTHONPATH:/path/to/vtstscripts` in `~/.bashrc` *and* the job script — batch jobs must inherit it; confirm `python -c 'import aselite'`). Then **invoke each as a bare command** (`vfin.pl <subdir>`, no path prefix; `which vfin.pl` confirms it's on PATH) **from the NEB main directory** holding the image subdirs — the argument is a *new destination subdir created in the CWD*, *not* the script's location nor the path to the source files (the classic mix-up: passing the dir where the VASP files already live instead of the to-be-created label). (Band construction can alternatively be driven through ASE directly.)
**Rule.** A vtstscript failing `permission denied` lost its execute bit in untar/copy (`chmod u+rx *.pl`); one that dies on a missing co-located module (`Vasp.pm`/`aselite`) was installed piecemeal / on PATH-but-not-PYTHONPATH — not a missing `ase`; one that runs cleanly but produces *empty* output was called with a path prefix or from the wrong CWD; one that fills a single column (e.g. `neb.dat`/`neb.file` path-distance) with zeros is missing a co-located *helper script* it shells out to — a cherry-picked install. Keep the full set in one dir on PATH (python: PATH *and* PYTHONPATH, interactive shell + job script), then run each *bare* from the NEB main directory whose CWD holds the image subdirs. **Two packages, two requirements:** geometry/interpolation scripts (`nebmake.pl`/`.py`, `modemake.pl`) are standalone perl/python and run on a **stock un-patched VASP** — a working `nebmake` is *no* proof the binary is vtst-patched. *Plain* elastic-band NEB also runs on stock VASP, but **CI-NEB (`LCLIMB`), improved tangent, double-nudging, the `IOPT` optimizers, and every min-mode method (dimer/Lanczos)** need vtstcode compiled *into* VASP — without it those features silently no-op and the OUTCAR-readers (`nebef`/`nebbarrier`/`nebresults`) emit all-zero/garbage (SKILL version reflex · neb-diagnostics VTST-not-linked). (The two-package split is canonical in modern VTST; script names stable.)

## Build sanity — VASP+VTST that misbehaves on *one* build but runs correctly on a reference binary is mis-*compiled*, not mis-input
**Symptom.** On a *locally-compiled* vasp+vtst, jobs hang, forces go static, or the optimizer behaves wrongly — yet the *same inputs* run correctly on a known-good reference binary (a colleague's / staff-maintained NERSC build). No error is printed, and `VTST: version` may even be present and version-matched — so it's *not* the missing-linkage or version-mismatch cases (SKILL version reflex).
**Cause.** A broken compilation silently corrupts optimizer/force state without erroring: aggressive optimization (`OFLAG=-fast`/`-ipo` instead of `-O2`), the wrong BLACS variant (e.g. OpenMPI-BLACS linked under an Intel-MPI run, or vice-versa), or incomplete ScaLAPACK linkage. The binary loads and runs — it just computes wrong. The *louder* sibling: over-aggressive optimization on **one** source file — classically `mpi.F` — instead **crashes immediately at startup** with `LAPACK: Routine ZPOTRF failed!` (a numerically-unstable build trips a Cholesky factorization on the first SCF) rather than silently misbehaving.
**Fix — rebuild from a validated reference `makefile.include`, then prove it.** Intel toolchain (`ifort`/Cray `ftn`), MKL with **ILP64** (`-DMKL_ILP64`), ScaLAPACK + the **matching** Intel-MPI BLACS, `OFLAG=-O2` (never `-fast`/`-ipo`), `fpp` preprocessor, `-Duse_shmem`. Then **validate by reproducing a test case that already passes on the reference binary** — identical inputs must give identical forces/path before you trust the new build. **Surgical alternative to a whole-build `-O2` drop:** when a *single* file is the culprit (the ZPOTRF-at-startup crash usually fingers `mpi.F`), lower just *that* file to `-O1`/`-O0` — `mpi.F` is MPI glue, off the compute hot path (BLAS/LAPACK + FFT dominate), so its opt level costs ≈nothing in throughput; recover any speed with a tuned BLAS/LAPACK (MKL/ACML/OpenBLAS/Goto) + FFTW, never by re-raising `mpi.F`'s opt level. (mpi.F trick OLD/2006 — confirm the file still exists and the trick is still needed in VASP 5/6 builds, which use separate BLAS/LAPACK flags.)
**Rule.** VASP+VTST misbehaving on one machine but not another ⇒ suspect the *build* before the inputs: diff `makefile.include` against a known-good reference — first `OFLAG` (`-O2`, not `-fast`/`-ipo`), then BLACS/ScaLAPACK linkage (the BLACS variant must match the MPI) — rebuild, and confirm by reproducing a reference test case. (Same "identical inputs, wrong only here ⇒ platform/compile mismatch" reflex as the neb-diagnostics frozen-image entry.) (VASP ~5.4 era, 2017; the flag/linkage failure modes are build-toolchain-general, not version-specific.)
**Link fails on an unresolved `flush` symbol** (`chain.f`/`lanczos.f`, building VTST under IBM `xlf` or another non-GNU Fortran)? `flush()` is a non-standard Fortran extension some runtimes don't provide; its only job is to force small output files to sync after each ionic step, so it's **cosmetic** — comment out the `flush()` calls to link, with no functional loss. (Non-GNU-compiler portability, IBM-era 2006; modern VTST may already condition/remove these — grep current source for `flush` before patching.)
**Distribution check (before the compiler).** A feature *missing* or a subroutine signature *wrong* right after upgrading a VTST source release — ssNEB absent, or `chain_force` lacking its `stress`/`TSIF` arg in a release that should include it — can be a **bad tarball**, not your edit: a newer archive accidentally re-bundling *older* `.F` files silently regresses features/interfaces. Before chasing the compile error, grep internal revision strings (`grep -i 'version\|! v' dimer.F chain.F`) and compare across downloads — a file claiming an *older* rev than a previous download (e.g. `dimer.F` v1.33 in a package advertised above one that shipped v1.36) ⇒ archive assembled wrong: re-download, re-verify, report the mismatch so it's re-uploaded. (Tarball-distributed vtstcode ~2012; largely moot under modern git, which tracks each file's revision by commit.)

## Building the initial band — `nebmake.py` is the modern drop-in for `nebmake.pl` (IDPP by default)
**Question.** "How do I use `nebmake.py`? The online manual doesn't mention it — do I run it *after* `nebmake.pl`?"
**Answer.** No — run `nebmake.py` *instead of* `nebmake.pl`, identical CLI (`nebmake.py POSCAR_initial POSCAR_final N_images`); it builds the whole band alone, so never chain the two. **Running it is what *creates and fills* every interior image dir `01`…`N` with a POSCAR** — VASP-NEB requires a POSCAR in every dir `00`…`(IMAGES+1)` at job start, so **`IMAGES` must exactly equal the interior-dir count** (interior dirs *only*, **not** the two endpoints — the classic off-by-one: endpoints `00`/`05` ⇒ `IMAGES=4`): any shortfall — *skipping* nebmake (only the endpoint POSCARs placed in `00`/`N+1`, interior dirs empty/absent), *or* setting `IMAGES` higher than the dirs you actually built — makes VASP **exit before any SCF cycle** with `ERROR: the following files does not exist NN/POSCAR` (first missing dir) — a *numbered*-image POSCAR, distinct from the bare-`POSCAR` parent-dir crash of a missing-`IMAGES` run (submit entry below). Fix by matching the two: set `IMAGES`=interior-dir count, or build the missing dirs (re-run nebmake). **All three positional args are required, in order** — omit any and the script aborts with a cryptic `vasp.pm::read_poscar, cannot open` file-error *before* it makes the image dirs, not a usage message (same trap in `nebmake.pl`). A second, *version-specific* `nebmake.pl` Perl-parser trap: older vtstscripts (`Vasp.pm` ~rev 915, 2017) misdetect a genuine VASP4 POSCAR (line 6 = atom *counts*, no element-symbol line) as `vasp5` and then abort `Illegal division by zero at Vasp.pm` on the bad atom-count parse — the tell is a `filetype: vasp5` print on a plainly-VASP4 file. It's the installed *revision*, not your POSCAR (a current vtstscripts reads the same file as vasp4 and proceeds) ⇒ update vtstscripts, or sidestep the Perl parser entirely with `nebmake.py`. (Affected ~rev 915 / 2017; fixed upstream.) A *third*, VASP4-era Vasp.pm header trap: POSCAR **line 1 a free-text/underscore name** (not space-separated element symbols) ⇒ `nebmake.pl` writes **no** per-image POSCARs, only **one stray file in the run dir named after that line** (holding the interpolated coords), image dirs `00`…`N` created but empty — Vasp.pm reads line 1 to identify species (VASP4 had no element line). Fix: line 1 = elements in POTCAR order (`C H Cr O`); VASP5 moved that role to the dedicated element line, and current vtstscripts/`nebmake.py` parse either. When nebmake.pl *silently* misbehaves with several installs present, pin **one** vtstscripts dir (`which nebmake.pl`; confirm which `vasp.pm` loads) — a stale/duplicate `vasp.pm` is the classic cause. (VASP4-era ~2008; current scripts/`nebmake.py` immune.) It **applies IDPP interpolation by default — no flag needed** — so the initial path is better-conditioned than `nebmake.pl`'s straight-line/linear interpolation. **Both scripts interpolate each atom along its *minimum-image* displacement, so an atom that wraps across a periodic boundary between IS→FS is interpolated correctly with no manual pre-wrapping — the cell boundary is not special.** **For a complex/known reaction coordinate, supply *intermediate* images as interpolation waypoints** — `nebmake.py` (and ASE) interpolate *through* provided structures, not only the two endpoints, steering toward the expected mechanism and resolving large structural changes better than an endpoint-only guess. To build a *single* intermediate geometry by hand, `interpolate.pl A B frac` writes one POSCAR `frac` of the way from A→B — `frac∈[0,1]` is a *fractional* interpolation weight (0.5 = exact midpoint), **not an absolute distance** (the recurring misread); `frac` just outside [0,1] extrapolates slightly past an endpoint (the legacy two-image dimer-axis trick, dimer-diagnostics). **A TS/saddle *guess* (prior dimer, literature, or intuition) is the prime waypoint — but *any* known intermediate (a metastable state, or a partially-converged image) serves the same: build two segments — N images IS→waypoint, Max−N waypoint→FS — then merge by *renumbering the image dirs sequentially* into one continuous `00`→`N+1` band, anchoring images near the saddle (not bunched at the barrier) for faster convergence. Should the two segments end up with very different inter-image spacings, start the merged band at a *small* TIMESTEP (old IBRION-driven NEB: small `POTIM`) so the tight-spaced/high-force region doesn't blow up** (≠ the two-step-reaction entry, which splits a real intermediate *minimum* into *separate* NEBs). Interior images are only a *starting guess* the optimizer will move, so — unlike endpoints, which must stay relaxed minima (endpoint-construction) — you *may* hand-edit interior image POSCARs toward a saner path when interpolation gives unphysical geometries; the one constraint is keeping interimage spacing roughly *equal*. (Manual silent on it as of 2023; the behaviour is in the script itself.)
**Rule.** Default to `nebmake.py` for band construction — same usage as `nebmake.pl` but IDPP out of the box (canonically ASE: `method='idpp'`/`IdppCalculator`). IDPP is *always* preferable to straight-line, not only when atoms would collide: it sidesteps the linear-overlap failure mode (neb-diagnostics) and gives **lower initial forces band-wide at no extra compute cost**, heading off high-initial-force non-convergence (`TIMESTEP`/`MAXMOVE` interim cures, parameters.md). **Interpolation and relaxation are independent** — build the IDPP band (`nebmake.py` or ASE), then relax with bare VASP *or* ASE+VASP in any combination; never *convert* a linear band — regenerate it directly, seeded with TS-guess/intermediate waypoints.

## Viewing band movies & convergence plots — `nebmovie.pl 0`/`1` = POSCAR-format frames (movie viewer); the `*.eps` are gnuplot PostScript (EPS viewer)
**Symptom/cause.** Two non-bugs, one root: `nebmovie.pl` writes **POSCAR-format** frames *concatenated* (`$xyzflag=0`, line 12 — the default, since XYZ drops the cell vectors a periodic system needs). So a "static" view (one unit cell, no animation) is a viewer not stepping through the frames, and an XYZ-only viewer (Molden) refusing to open the file at all is the same format mismatch — neither is a broken run.
**Fix.** Use a multi-frame, **cell-vector-aware** viewer — **VESTA, VMD, ASE, or TSASE** all read the POSCAR movie directly (preferred for periodic systems) — or Jmol/OVITO's animation controls; force XYZ (`$xyzflag=1`, line 12) only when the viewer demands it (then any XYZ-trajectory viewer — Jmol, VMD, OVITO — animates the resulting multi-frame `movie.xyz` directly; a lone `movie.xyz` of unknown origin is just standard XYZ, open it the same way). **`nebmovie.pl 0`** stitches the **POSCAR** (initial/interpolated) frames; **`nebmovie.pl 1`** the **CONTCAR** (converged) frames — endpoints (`00`, last image) have no CONTCAR, so flag 1 falls back to their POSCAR there. Flag 0 = the pre-submit interpolated-path check (every "eyeball the band" reflex in this skill); flag 1 = watch the converged band; no NEB change needed either way. **Atom-type labels in the movie come from POSCAR line 1's element symbols** (VASP4-era comment line, POTCAR order, e.g. `Al Ti H`): a free-text/blank line 1 ⇒ an unlabeled movie — `typeset.pl` batch-writes the symbols into every image POSCAR *and* CONTCAR at once (VASP5's dedicated element line moots it; same root cause as the `nebmake.pl` VASP4 header trap, build-the-band entry).
**The `*.eps` plots are gnuplot PostScript, not text/PNG.** `nebconverge.pl` (per-image energy & force convergence) and `nebresults.pl` (`mep.eps`, the MEP) emit their plots as a **gnuplot-generated EPS/PostScript** `vaspout.eps`/`mep.eps` — open with any PostScript/EPS viewer (ghostview, evince, Adobe Illustrator/Distiller) or a PDF-capable viewer (convert to PDF); don't expect plain text or PNG. The plot is only a convenience — the numbers live in `neb.dat`/`exts.dat`, so a missing-gnuplot / `cannot stat vaspout*.eps` warning is harmless (handoff entry) — these plot scripts (`nebmovie.pl`/`nebspline.pl`/`nebconverge.pl`/`nebresults.pl`) just shell out to **gnuplot**, a *separate system package* (`module load gnuplot` on HPC, else yum/apt), wholly unrelated to the VASP/VTST build; a `sh: gnuplot: not found` is a missing dependency, not a compile failure — install/load it only if you actually want the EPS plots. (Output format could differ in newer vtstscripts — confirm against your install.)

## Submitting & allocating a NEB — same submit script as a single-point run (just add `IMAGES`); images parallelize over MPI (don't pre-relax them); GPU needs one GPU per image
**Question.** "Do I need a special *parallel* submit script for NEB? Must it span multiple nodes? Why did only some images launch? NEB is expensive — can I pre-relax each image first?"
**Answer.** No special launcher exists or is needed: an NEB reuses the *single-point job's submit script unchanged* — the only required change is adding **`IMAGES=N`** to INCAR (N = interior images), which VASP reads to distribute the images itself across the allocated MPI ranks (the binary must itself be **MPI-compiled — `vasp_std`, not `vasp_gam`/a serial build — and launched under `mpirun`/`srun`**; a serial binary can't distribute images, silently writing no per-image OUTCAR). Don't conflate scheduler parallelism with VTST's internal image parallelism. Node spread is then free — no node-topology constraint; pack the band however the scheduler prefers. The one layout rule is the ordinary VASP one — **IMAGES must divide the total MPI rank count** — one rank/image minimum; NPAR/KPAR partition *within* an image and do **not** set the image count, so raising NPAR adds no images — only matching ranks to IMAGES does (too few ranks ⇒ surplus image dirs never populate). **GPU builds tighten it: GPU VASP binds one MPI rank per GPU, so `IMAGES ≤ #GPUs` (one image/GPU)** — more images than GPUs ⇒ surplus image dirs never start (add GPU nodes); CPU can over-subscribe ranks, GPU cannot. **No serial-image mode exists in native VASP — all images advance together every ionic step, so the whole `IMAGES`×procs/image core block must be co-allocated at once; you can't shrink the reservation by running images one-at-a-time *nor by submitting each image dir (`00/`,`01/`,…) as its own independent queue job* — the images are one tightly MPI-coupled VASP process, not separable per-image calculations.** When that allocation is infeasible, two tiers: first cut `IMAGES` and/or procs-per-image to fit the cores you have; if it still won't fit, drive the band through an **ASE or EON VASP-calculator wrapper**, which steps the images *sequentially* at ~1× procs-per-image but ~`IMAGES`× wall time (a version-independent workaround — native VASP's all-images-parallel model is unchanged).
**Per-rank RAM — plan ~1–2 GB/core; VTST adds *no* memory overhead beyond base VASP.** A 50–200-atom NEB/dimer is comfortable at ~1 GB/core (2 GB/core ample); the lone exception is exact-exchange/hybrid, which wants more per core. When per-core memory binds, spread the *same* work over **more cores** rather than buying more RAM/core — it relieves the footprint *and* speeds the run. NEB RAM scales with images × basis size × k-points, so at low parallelism each MPI rank holds too much and VASP dies `… Segmentation fault (signal 11)`. Gauge per-rank load from OUTCAR (`NIONS`/`NKPTS`/`ENCUT`). **Two opposite memory faults, separated by the node-doubling test.** *Physical exhaustion* — a `signal 11` at low core count that **clears when nodes double** (same inputs): fix by (a) adding cores/nodes, or (b) shrinking the per-rank footprint for the path search (ENCUT toward the POTCAR minimum, minimal k-mesh, soft non-`_pv`/`_sv` PAW, `ISPIN=1` if non-magnetic, `PREC=Normal` — convergence-economy entry; prefer (b), production accuracy is wasted before the saddle is located) — **and** when an NEB OOMs/FFT-crashes where the *same-system plain run* fit the same nodes (even at `IMAGES=1`), suspect a binary-flavor mismatch, not VTST overhead: the NEB binary is usually **full-k-point** while the regular run used a Γ-only build (`vasp_gam`), ~2× the planewave storage — compare `NBANDS`/`NPLWV` across the two OUTCARs to isolate which dominates (the `IMAGES` tag can itself shift band/planewave allocation), then cut per-rank memory with `NPAR`/`NSIM` or run the matching Γ-only NEB binary (modern `vasp_gam`/`std`/`ncl` split makes the flavor explicit). *Artificial ulimit* — a memory-allocation crash **at FFT init** that fires **only with `IMAGES>0`** (plain single-image VASP runs fine) and does **not** clear when nodes double: not physical RAM but a **per-process stack/data ulimit the batch scheduler reset to system default** — interactive-shell `ulimit`s aren't inherited by the job, and NEB's larger per-image FFT workspace trips the cap. Enforce it *at job-launch* — raise `ulimit -s`/`-d` (or the MPI launcher's memory flags) in the job prologue, not your login shell. (Original IBM/AIX `-bmaxdata`/`-bmaxstack` *link*-time flags 2006-era, obsolete — on AIX a shell `ulimit unlimited` was necessary-but-not-sufficient, the binary kept its own link-time ceiling; the scheduler-resets-ulimits principle holds on any batch system, e.g. SLURM. **Third, version-specific cause:** a memory-*allocation* crash that fires on one compiler/platform but not another with *identical* inputs/RAM can be an under-allocated array in old VTST source — pre-2007 `neb.F` mis-allocated `old_tangent`, UB strict compilers surfaced as a crash — so update VTST before blaming RAM/ulimit; moot on current versions.)
**Don't pre-relax images individually to cheapen the band.** Relaxing each image alone first, then seeding a full NEB from those CONTCARs/WAVECARs, does *not* lighten the NEB: an individual relax minimizes each atom's *total* force, whereas NEB converges the path-**perpendicular** projected + spring force — a *collective* property of the path's tangents, so pre-relaxed images are *not* pre-converged (neb-diagnostics convergence-metric). The real cost levers: MPI parallelism over images (above), and once a reasonable path exists, drop the band and converge the saddle with a *single-image* dimer via `neb2dim.pl` (NEB→dimer handoff).
**Tell.** Missing `IMAGES` *or* a serial binary surfaces three ways: the job *ran as one plain relaxation* (image dirs static / no forces); **or** it finished but image subdirs hold `CONTCAR`/`OSZICAR`/`WAVECAR` with **no `OUTCAR`** (only one dir — or none — has it; OSZICAR energies still readable, but `nebbarrier.pl`/`nebef.pl` need the OUTCARs); **or** — POSCARs placed *only* in `00/`,`01/`,… with none in the parent — it **crashes at launch, every rank printing its *own* VASP banner then `ERROR: the following files does not exist POSCAR`** (the launcher spawned N standalone single-points each hunting a parent POSCAR that isn't there — *either* a **serial (non-MPI) binary** that can't form image teams at all, *or* a proper MPI build run **without `IMAGES`** so ranks never partition). `grep IMAGES INCAR` (absent ⇒ add it, no script change) **and** confirm an MPI build (`vasp_std`) ran under `mpirun`/`srun` (≥ `IMAGES`×cores/image ranks); **one VASP banner per *image group* is healthy — repeated banners + repeated `POSCAR` errors = serial binary *or* unpartitioned ranks, told apart by the banner line: `serial version` in OUTCAR/stdout ⇒ recompile with MPI (an MPI build instead reads `parallel version` / `running on N cores`); `parallel version` but per-rank banners ⇒ the missing `IMAGES`**. **Mirror — MPI build launched *bare*.** An MPI binary (`-DMPI`) invoked *directly* by the scheduler (`bsub … ./vasp`, no `mpirun`/`srun`) makes `MPI_Init` see one rank: OUTCAR reads `running on 1 nodes`/`each image running on 1 nodes` and VASP **silently cuts `IMAGES` to 1** (`M_divide` clamps `NPAR` to `COMM%NCPU=1`) — only one image dir runs though the build is fine. Cheap discriminator: a *regular non-NEB* job *also* showing `running on 1 nodes` ⇒ the queue gave the job one rank, not an NEB bug — prefix the launcher (`mpirun -np N -machinefile <scheduler_file> ./vasp` / `srun`). (Explicit `mpirun -machinefile` is older LSF/PBS; SLURM `srun` propagates ranks automatically — but *an MPI binary needs its launcher or all ranks collapse to one* is version-general.) Then: launched MPI ranks vs `IMAGES`+2 in the scheduler log; `IMAGES` vs the allocation's GPU count; a `signal 11` segfault that disappears when nodes double = per-rank RAM, not a bug. **Mirror binary-flavor fault (k-point axis):** a Γ-only build (`vasp_gam`) run with a KPOINTS holding more than the Γ point self-names — VASP aborts at launch right after IBZKPT, `You are using the Gamma-point only version with more than one k-point`. Fix = relink/resubmit the full-k-point `vasp_std` (no INCAR/KPOINTS change); **never trim the mesh to Γ to silence it** (that discards k-sampling accuracy to paper over a binary mismatch).
**Rule.** An NEB needs no special submit script — reuse the single-point launcher, add only `IMAGES` to INCAR (VASP spreads images across ranks); a band that silently ran as one plain relaxation — *or* finished with image dirs missing OUTCARs — *or* (POSCARs only in `00/`,`01/`,…) crashed at launch with a per-rank `POSCAR does not exist`, each rank printing its own banner — = missing `IMAGES` **or a serial (non-MPI) binary** (run an MPI build, `vasp_std`, under `mpirun`/`srun`). No "must span nodes" rule exists — allocate to the cluster's preferred packing; the binding constraint is rank count (`IMAGES` divides total MPI ranks; GPU VASP `IMAGES ≤ #GPUs`, one rank+GPU per image). Budget ~1–2 GB/core (VTST adds no RAM overhead vs base VASP; hybrid wants more), and prefer more cores over more RAM/core; a low-core-count `signal 11` segfault that clears with more nodes is per-rank RAM (memory scales with images × basis × k-points): add nodes *or* shrink the basis for the path search. Cut an expensive band's cost by MPI-parallelizing images and/or a single-image dimer handoff (`neb2dim.pl`), **never** by pre-relaxing images individually (the NEB force is collective). Native VASP has **no serial-image mode** — when the `IMAGES`×procs/image cores can't be co-allocated, cut images/procs-per-image first, else serialize the band through an ASE/EON VASP-calculator wrapper (~1× procs, ~`IMAGES`× wall time). (Submit-script/IMAGES 2019; serial-binary/no-OUTCAR 2016; per-rank `POSCAR`-not-found crash 2016 (serial-binary variant + `serial version` banner discriminator 2007, VASP 4.6.28 — banner fundamental to all versions); GPU rule 2021; pre-relax 2019; RAM scaling 2018; RAM planning ~1–2 GB/core / VTST-no-overhead 2012; no-serial-mode/ASE-EON serialize 2012; Γ-only-binary banner / use `vasp_std` not trim-mesh 2008, VASP 4.6.21 — Γ-vs-full distinction fundamental to all versions; NEB-OOM-vs-plain-run = full-k-vs-Γ binary / `NBANDS`-`NPLWV` compare OLD/2006, diagnostic general; none version-specific.)

## Multi-step / termolecular reaction — decompose into elementary (bimolecular) steps, one NEB each, run in parallel (don't chain)
**Goal.** A reactant→product path that passes through (or is suspected to pass through) a stable intermediate minimum — i.e. two distinct reactive steps A→B→C.
**Mechanism-agnostic — you needn't know concerted vs stepwise in advance.** NEB assumes nothing about elementarity: run it with enough images spanning the *full* reaction coordinate and the converged MEP shape reveals the mechanism — a single maximum = concerted/elementary; an intermediate *minimum* between reactant and product = stepwise. So don't avoid NEB (or pre-decide the mechanism) because you're unsure a reaction is elementary; inspect the MEP first. If a stepwise minimum appears, relax it and split as below. **The one *a-priori* exception — molecularity ≥3:** a **termolecular / multi-reactant** step (A+B+C→D) has no single first-order saddle (a simultaneous 3-body collision is physically implausible), so decompose it into *sequential bimolecular* steps (A+B→E, then E+C→D) and run one NEB each *before* building any band — never set up a single termolecular NEB (forced, it just relaxes toward the nearest stepwise path anyway).
**Procedure.** Don't build one multi-step NEB spanning both steps. First relax the intermediate (B) explicitly to its own local minimum, then run the two segments — A→B and B→C — as *separate, independent* NEBs. They have no data dependency, so launch them in parallel. **That intermediate is the *shared endpoint* of both segments, so it must be a true relaxed minimum** (endpoint-construction): you cannot reconstruct the full MEP by *stitching two separately-converged segments joined at an unrelaxed interior NEB image* of a partial run — the join is no stationary point and MEP continuity across it is unenforced. No relaxed intermediate ⇒ run one full-span band, not two stitched segments.
**Why not chain.** A band stretched across two steps wastes images: they cluster around the deep intermediate and starve each saddle region — resolving both barriers *worse* at the same or higher cost than two focused segments, each cleanly resolving one saddle.
**Rule.** When a path is multi-step — an intermediate minimum, or *a-priori* a termolecular/multi-reactant reaction (no single 3-body saddle) — relax any intermediate first and run one NEB per *elementary (bimolecular)* step (A→B, B→C) as independent parallel jobs; never chain two reactive steps into one multi-step NEB, and never set up a single termolecular band. (If the intermediate instead *surfaces* only after a dense single-step band shows a mid-band dip, see neb-diagnostics MEP-topology — there segmentation is optional once the band has converged.)

## Refining the MEP in one region — child NEB from converged images, or (once a saddle exists) downhill minimizations
**Goal.** Sharpen the path in a *local region* of a converged band — resolve a poorly-sampled bump/shoulder/local-max between two adjacent images — without rerunning the whole NEB.
**Child-NEB refinement is valid.** Running a child NEB whose endpoints are two *adjacent converged parent-NEB images* is sound — those images already lie on/near the MEP, so they need **no endpoint re-relaxation** (only the original outer endpoints must have been relaxed). Recurse (a child band between any two converged images) until the feature is resolved, then a final CI-NEB pins the local maximum. Keep EDIFF/EDIFFG identical at every level.
**Once a saddle is known, prefer downhill minimizations — not recursive NEB.** If the saddle is already located and the real goal is tracing the MEP to arbitrary precision (map the path curvature, find a secondary minimum), recursive NEB is the wrong tool. Run *minimizations* that follow the path downhill: from each endpoint toward the nearest feature, and from **both sides of the saddle**. Use **steepest descent (IOPT=4) or quickmin (IOPT=3)** with a **small MAXMOVE** so each step hugs the MEP — 2nd-order optimizers (LBFGS/CG, IOPT=1/2) take large steps that leave the path. This is more efficient and more precise than ever-denser bands. For best fidelity to path curvature use **RK4**, which VASP does not implement — drive it from an external tool (vtstscripts) (as of 2023-09).
**Rule.** Converged NEB images are valid fixed endpoints for a child NEB (no re-relaxation) — recursive refinement of one region is legitimate. But once a saddle exists, trace the MEP with downhill steepest-descent/quickmin minimizations (small MAXMOVE) from both sides of the saddle rather than denser recursive bands; reach outside VASP for RK4 if path-curvature accuracy matters. (To segment a genuinely multi-well converged band into per-step NEBs instead, see the two-step-reaction entry and neb-diagnostics MEP-topology.)

## Symmetry-equivalent path — transform the converged band, don't rerun the NEB
**Goal.** You have a converged NEB for one path (A→B) and need the band/barrier for a *crystallographically equivalent* path (C→D) — the two endpoint pairs related by a space-group symmetry operation of the crystal.
**Procedure.** Don't launch a second NEB. First confirm the equivalence: check that a single lattice symmetry operation (rotation / mirror / translation / glide) maps endpoint pair (A,B) onto (C,D) — compare POSCAR/CONTCAR site labels against the operation. Then apply that *same* operation to **every** image of the converged band (all image POSCARs, or the `.xyz`/movie frames). The transformed band is itself a converged MEP for the equivalent path — energies are identical by symmetry — so its barrier is already known.
**Rule.** A symmetry-equivalent path needs no new DFT — apply the operation to *all* images for the equivalent converged band directly (same barrier, zero compute). Only paths *not* related by any space-group operation require their own NEB.

## NEB→dimer (or Lanczos) handoff — a converged CI-NEB needs no dimer; if you do hand off, `neb2dim.pl` → `ICHAIN=2` (not `IBRION=44`)
**Goal.** Decide *whether*, and *when*, to stop a (CI-/CL-)NEB and hand its saddle to a dimer/Lanczos.
**A converged CI-NEB is already done — no dimer needed.** The climbing image is driven exactly to the saddle, so once the CI-NEB meets `EDIFFG` its highest image *is* the TS; a subsequent dimer adds no information — it only confirms the negative mode and quits after one step (dimer-diagnostics "stops after 1 step"). Confirm the saddle with a frequency run instead (TS-confirmation entry). **Deliverable-dependent stop:** the CI image climbs to the saddle *independently* (inverted parallel force), so if the saddle *geometry* is the sole deliverable, stop once *it* meets `EDIFFG` even with interior images still lagging — VASP converges *collectively* (max force over *all* images), so this means watching the CI image's force and ending the run by hand. Full-band convergence is required only for the barrier *profile* / reaction-path energetics (which need every image). A handoff buys something only after a *plain* NEB (no climbing image), whose highest image is a between-images *estimate*, not the saddle.
**When you do hand off — the canonical sequence.** Converge the band to the saddle *region* first — ~0.1–1 eV/Å on the climbing image, **not** the tight final saddle criterion (the dimer does the final refinement, so over-tightening a dimer-seed NEB wastes compute); but a *still-wandering* image is a poor seed: position+mode off the saddle, so the dimer wanders and burns more compute than just finishing the cheaper NEB — fix the NEB optimizer/timestep, parameters.md; don't reach for the "fancier" method at first moderate force. Then, VTST linked (grep OUTCAR `VTST`): **(1)** `vfin.pl <subdir>` *from the NEB dir* — archives the band into `<subdir>`, staging `CONTCAR→POSCAR` across all image dirs while *preserving* the OUTCARs (restart entry); never hand-copy `CONTCAR→POSCAR` yourself. **(2)** `cd <subdir>`. **(3)** `nebresults.pl` there — creates `exts.dat` and lets you sanity-check the MEP. **(4)** `neb2dim.pl` *from inside that subdir*: it reads every image's `0?/OUTCAR` (forces+energies) to seed the dimer with the saddle geometry **and** the reaction-coordinate (MODECAR) vector inherited from the band, into VTST's dimer (`ICHAIN=2`) — **not** VASP's native `IBRION=44`, which does *not* inherit that estimate.
**Two precondition tells.** `NO EXTS.DAT IN THIS DIRECTORY; RUN nebresults.pl` = step 3 skipped, or neb2dim.pl run from the raw image dirs/parent — `exts.dat` must sit in its CWD. `gnuplot: command not found` / `cannot stat vaspout*.eps` during nebresults.pl are **harmless** — gnuplot only draws the MEP; `exts.dat`/MODCAR don't depend on it (install gnuplot only if you want the plots).
**Verify the seed.** Two failures: **(1) zero/0-byte/empty MODECAR** (with `exts.dat` energies nonsensical, >10 eV) = OUTCARs missing when it ran / dirs hand-disturbed — typically a manual `CONTCAR→POSCAR` copy *outside* `vfin.pl` (that promotion *restarts* a run, it does **not** seed a dimer). **(2) Wrong-geometry seed** = dimer first-step energy ≉ NEB peak (DIMCAR first `Energy` ≉ nebef.dat col-3 peak): it starts well *above* the peak at high force, curvature may go negative but forces never home onto the saddle = `neb2dim.pl` ran on an *incompletely converged* band or *from the wrong directory*. Either way re-seed by hand: highest-energy image's POSCAR as dimer center, regenerate MODECAR with `modemake.pl (N-1)/POSCAR (N+1)/POSCAR` (N = peak image).
**Rule.** A converged CI-NEB needs no dimer — its highest image is the TS; confirm by frequency, don't bolt on a tighter-EDIFFG dimer (and when only the saddle *geometry* is wanted, stop once the *climbing image* meets `EDIFFG` — full-band convergence is only for the barrier *profile*). When a handoff *is* warranted (plain NEB, or a genuine stall — neb-diagnostics MEP-topology), follow the sequence: `vfin.pl <subdir>` → `cd <subdir>` → `nebresults.pl` (makes `exts.dat`; gnuplot/eps warnings harmless) → `neb2dim.pl` there → VTST `ICHAIN=2` (not `IBRION=44`). Then verify: zero/0-byte MODECAR (+ nonsensical `exts.dat`) = dirs hand-disturbed (manual CONTCAR→POSCAR) so OUTCARs missing; first-step energy far off the NEB peak = wrong-image/wrong-dir seed → re-seed by hand from the peak POSCAR.

## Dimer / SSD gives only the saddle — relax along ±the negative mode to recover *and IRC-verify* the two minima
**Goal.** From a converged dimer/Lanczos/SSD saddle, get the two minima it connects (and thus their energies — the endpoint/reaction energy) — either to *discover* unknown endpoints, *plot the MEP*, or *verify IRC connectivity*: a dimer finds the saddle *nearest* its start, which need not be the TS for the intended reaction (e.g. one optimized from an unfinished NEB's max image), so the reactant↔product assignment is unvalidated until checked.
**Procedure.** A saddle search returns *only* the saddle geometry plus its negative-curvature mode (the dimer direction) — never the endpoints, nor their energies: a **static single-point** (`NSW=0`/`IBRION=−1`) at the saddle yields the *saddle's* energy, not a minimum's, and even a plain *relax* of the *unperturbed* saddle is unreliable (≈zero gradient at the stationary point ⇒ falls to *either* basin arbitrarily). Displace the saddle *slightly* along that mode in **both** directions (+mode, −mode) and run a **plain** downhill relaxation from each (`ICHAIN=0`, *not* another dimer/NEB — the displaced endpoints are *independent*, with no knowledge of each other); the two runs fall into the adjacent basins — only *those* relaxed structures give valid endpoint energies and the reaction energy. (Judge connectivity by *which* basin each relaxation reaches — **not** by `dist.pl` saddle-to-min structural distance: when atoms don't translate linearly the saddle can sit Euclidean-farther from one minimum than the other, which is *expected*, not evidence it's off-path.) `dimmins.pl` (vtstscripts) automates both ± minimizations from the saddle + its mode. **Want the *continuous* MEP/barrier profile, not just the endpoint energies?** Run an NEB between the two recovered minima; a legacy quick seed is a *damped-MD* descent off the saddle (`IBRION=3`, `POTIM≤0.1`) whose `XDATCAR` frames (`xdat2pos.pl`) become the band images (verify the script name/interface in current vtstscripts). **Already have just the 3 points (IS/saddle/FS) and only want a quick plot?** Skip the full NEB — drop each structure's `POSCAR`+`OUTCAR` into `00`/`01`/`02` and run `nebresults.pl`, *or* hand-build `neb.dat` (energies, path-force `=0`, distances) + the spline script: it plots energy vs. reaction *distance* through the 3 known points as if they were a band. The reaction coordinate **is** inter-structure distance (what NEB uses) — the only well-defined choice when reactant≠product, so don't normalize A/saddle/B to −1/0/+1. The iterative saddle-outward image-placement of the original dimer paper (JCP 111, 7010, 1999) is **not** in VASP and isn't needed for rates (harmonic-TST note below). (*Legacy two-image* dimers instead stepped the axis via `interpolate.pl 01/POSCAR 02/POSCAR ±step`, step ≈**10**× the ~0.01 Å dimer separation (~0.1 Å off the saddle): a *too-small* step (≈1) barely leaves the saddle, a *too-large* one (≫10) overshoots along the unstable mode and won't relax into the connected basins — the image pair *encodes* the mode; modern single-image dimers use `dimmins.pl` instead, as plain `interpolate.pl` between two POSCARs carries no mode. Pre-v2.0 ~2007 — dimer-diagnostics legacy-two-image entry.) **The dimer/OUTCAR energies *during* the search are optimizer steps (up-then-down), not MEP images** — get the path/endpoints only from these post-search ±minimizations (dimer-diagnostics non-monotonic-energy entry). **Only one endpoint needed** (confirm the forward product, not full IRC)? — pick the sign by chemical intuition. Use a small-MAXMOVE steepest-descent/quickmin step so each trajectory hugs the descent path (Refining-the-MEP). For an SSD solid-solid transition the relaxation must carry **cell** DOF (`ISIF=3`, or TSASE's cell-enabled relax) so the cell settles into each phase.
**Rule.** A saddle search locates the saddle and nothing else; relax along ±the negative mode (`dimmins.pl` automates it) to recover its endpoints — and IRC connectivity (the saddle is the *right* TS) holds only once both minima match the intended reactant and product (include cell DOF when the mode involves cell change, as in SSD). **Two senses of *IRC* — don't conflate:** the **connectivity** check above vs the IRC *path itself*. NEB traces the **mass-unweighted** MEP; the true IRC is **mass-weighted** steepest descent, so the two curves diverge *away* from the stationary points but share the **same saddle** (invariant to mass scaling). For harmonic-TST rates (absent tunneling) only the saddle + the normal modes *at* the critical points enter the rate — never the path *shape* — so NEB's unweighted MEP is fully sufficient and you never need the mass-weighted path for a barrier or rate. (Expert-stated 2012 & 2020; ±mode-then-NEB-to-plot-the-MEP / 3-point pseudo-NEB quick MEP plot / iterative-tracing-not-in-VASP / interpolate-step-≈10 / dist.pl-not-a-reaction-coordinate OLD/2005; mass-weighted-IRC-vs-NEB-MEP OLD/2008; not version-specific.)

## Finishing / restarting any VTST run — `vfin.pl <dirname>` archives + stages the restart atomically
**Goal.** Continue a completed-but-unfinished VASP/VTST run (wall-time kill, or needs more steps) — the friction being that a multi-image NEB restart promotes `CONTCAR → POSCAR` in *every* image dir, not the single rename a regular run needs (and emphatically **not `ISTART`** — a recurring misread: `ISTART=1` re-reads only the wavefunction, never the ionic geometry, so without the per-dir copy the band just reverts to the original interpolated path).
**Procedure.** Run `vfin.pl <dirname>` (vtstscripts) in the run dir: it archives the finished calc into a fresh subdir `<dirname>` (OUTCAR/OSZICAR/etc. preserved) and copies each `CONTCAR → POSCAR` in place — over all image dirs for an NEB — ready to resubmit. One command handles regular VASP, dimer, and NEB; never hand-touch per-image dirs.
**Endpoint-revert tell (forces *grow* on the restart).** A *second* run whose images go chaotic and forces climb — pinned on an `IOPT` switch (e.g. 7→1) — is usually a hand-reverted endpoint, **not** the optimizer: the *original unrelaxed* POSCAR was copied back into `00`/`N+1`, discarding the relaxed endpoint geometry, so a now-large `00↔01` (or `N↔N+1`) gap drives a huge spring force. Check `00/POSCAR` against the relaxed endpoint geometry (its minimized CONTCAR — a >0.5 Å shift on any atom is the tell) and the `00↔01` spacing; keep the *relaxed* endpoint, let `vfin.pl` stage the interior images, and confirm all spacings are smooth before resubmitting — the optimizer choice is innocent.
**Per-image files (self-contained dirs).** Each NEB image dir is an *independent* VASP calc reading only its *own* files, so wavefunction/charge restart/seed files (`WAVECAR`(+`.prev`), `CHGCAR`) go *inside* each numbered dir (`00/`,`01/`,…); a *top-level* `WAVECAR` initialises *no* image (same reason per-image spin seeding copies WAVECAR+CHGCAR per dir — neb-diagnostics). **The *standard* NEB restart is geometry-only (`CONTCAR→POSCAR` per dir) — reusing `WAVECAR`/`CHGCAR` is an *optional* SCF speed-up, not usual practice, which is why the cleanup script (`vfin.pl`) stages `CONTCAR→POSCAR` but doesn't leave `CHG*`/`WAV*` in the source dirs — if they were written it *compresses them into the archive subdir* (not deleted), so they're recoverable (or edit `vfin.pl` to keep copies in place). When you *do* reuse them it's the ordinary VASP charge/wavefunction restart — copy `WAVECAR`/`CHGCAR` back from the archive (un-gzip), drop one into each image dir, and set `ISTART=1`/`ICHARG=1` in the (shared) INCAR, exactly as for a single-point restart.**
**Dimer caveat.** `CONTCAR → POSCAR` is fine for a dimer — including the *converged* saddle: **CONTCAR is the best TS estimate** (consistent with CHG*/WAV*), *not* CENTCAR, which lags one dimer iteration at convergence (higher force). Use CONTCAR for frequency / TS-confirmation / continuation. A dimer *continuation* (more steps / tighter `EDIFFG`) needs the **mode** carried over too: copy **`NEWMODECAR → MODECAR`** — `NEWMODECAR` (rewritten each step) holds the latest optimized mode, so seeding it skips re-running the rotation search; **reusing the first run's original MODECAR (or omitting it) restarts the mode from scratch**, wasting the converged rotation. `vfin.pl` detects the job type and stages *both* files (dimer-diagnostics).
**Rule.** Restart any run with `vfin.pl <archive_dir>` — isolates the result and stages `CONTCAR → POSCAR` across *all* image dirs atomically (per-image `WAVECAR`/`CHGCAR` live there too — never one top-level copy), and **never hand-revert an endpoint to its pre-relaxation POSCAR**. (Wall-time-killed band still descending: resubmit *unchanged*, don't edit INCAR — neb-diagnostics wall-time entry. Diverging forces on a restart blamed on an `IOPT` switch = a reverted endpoint, above — not the optimizer. Dimer converged-saddle geometry = CONTCAR, above.)

## Accepting a converged NEB — `nebef.pl` → `vfin.pl <dir>` → `nebresults.pl`, in that order
A NEB ionic relaxation finished and you want the barrier/path — the *accept* branch (the dimer *handoff* branch is the neb2dim entry above). **Run `nebef.pl` *first***: from the NEB main dir it prints each image's energy + max force, so you decide continue-vs-accept (every image ≤ `|EDIFFG|` ⇒ accept; else continue — restart entry above; never archive a still-descending band as final) *before* touching the files. Then **`vfin.pl <dir>`** archives the OUTCARs and stages `CONTCAR→POSCAR` across all image dirs (restart entry), and **`nebresults.pl`** run inside `<dir>` writes `neb.dat`/spline + `mep.eps`/`movie.xyz` (movie-viewing & handoff entries for output formats and the harmless gnuplot/eps warnings).
**Rule.** Finish a NEB by reading it in order — `nebef.pl` (converged?) → `vfin.pl <dir>` (archive + CONTCAR→POSCAR) → `nebresults.pl` (barrier, spline, movie) — nebef.pl first so a still-descending band is continued, not frozen as final. (Script names/workflow stable across vtstscripts versions; confirm availability on your install.)

## Packaging a run for transfer / expert review — strip CHG*/WAV* first
**Goal.** Share a NEB/dimer run for debugging without hitting upload or email size limits.
**Procedure.** Before compressing, `rm */CHG* */WAV*` — CHG/CHGCAR/WAVECAR dominate run size. The rest (INCAR, POSCAR/CONTCAR, OUTCAR, OSZICAR, KPOINTS, POTCAR, scheduler stdout) carry everything needed to diagnose convergence/optimizer/geometry/scheduler issues.
**Rule.** Strip CHG*/WAV* before archiving a run for transfer — never needed to diagnose a convergence/optimizer/scheduler problem.

## Confirming the TS (CI-NEB or dimer saddle) — exactly one imaginary mode along the reaction coordinate
**Goal.** Decide whether the converged saddle search actually located the transition state. Frequency analysis after a converged CI-NEB is *optional but recommended* — it both verifies saddle character (one imaginary mode) *and* yields the hTST prefactor (Barrier→rate entry) — and runs only at the *stationary points* (the relaxed minimum and the climbing image/saddle), never at every band image.
**Procedure.** The TS *is* the highest-energy image of the converged CI-NEB (top-energy row of `neb.dat`; the geometry is *that* image's `CONTCAR`) — that is the defining criterion (a converged dimer/Lanczos saddle plays the same role). **Two preconditions: (1) the saddle is *tightly* converged — the ≲0.01 eV/Å search default is often *too loose* for the Hessian, so re-relax to `EDIFFG ≤ -0.002` (−1e-5..−1e-8 if strict) and build the dynamical matrix from the *tightened* CONTCAR — tighten *saddle-preservingly* (continue the CI-NEB or hand to a dimer), **never a plain ionic minimization (`IBRION=1/2`/any minimizing `IOPT`), which slides downhill off the one-negative-curvature saddle into a basin**; residual gradient at a loose saddle surfaces as spurious extra negative modes, and a geometry that shifts appreciably on tightening confirms the prior run was under-converged; (2) the freq calc's free-atom set matches the atoms that were free during the TS search** — a frozen-surface freq is valid only if the surface was *also* frozen during the relaxation; a DOF mismatch injects spurious extra imaginary modes (below). With those met, go straight to frequency (no optimizer switch or extra NEB steps needed). Confirm saddle character with a frequency run on *that geometry alone* (`IBRION=5/6` = finite-difference; or `IBRION=7/8` = DFPT/linear-response, *not under DFT+D* — parameters.md): a true first-order saddle has **exactly one** imaginary mode whose eigenvector lies along the reaction coordinate.
**One imaginary mode but along the *wrong* coordinate → wrong reaction, not numerics.** A cleanly converged CI-NEB (CI tangent force → ~0, one imaginary mode) proves only a *local* saddle on the *current* band, not the intended reaction. A mode whose eigenvector points along the wrong coordinate — often with a non-physical initial path (near-complete dissociation then re-adsorption, a tell an endpoint carried prior relaxation history) — means the endpoints/path encoded the wrong chemistry and CI faithfully found a *real* saddle for the *wrong* reaction. Rebuild from freshly relaxed IS/FS and visualize every interpolated image before launch (IDPP smooths the path but cannot fix wrong endpoints); never hand a wrong-mode CI to a dimer — it only refines the wrong saddle.
**Active-set size — partial Hessian for a large cell; for a small one, displace all.** A full dynamical matrix costs ~6N single-points, prohibitive at large N. Displace only atoms that move appreciably along the MEP (selective dynamics → partial Hessian); `dymseldisp.pl` (vtstscripts) picks that active subset from the band — highest-displacement atoms + their first-shell neighbors. Expand the displacement set until the imaginary-mode frequency (and the prefactor, Barrier→rate) stops shifting — the full 6N matrix is essentially never needed at large N. **But when only ≲100 atoms move, skip the shell-convergence bookkeeping and just displace them all in one (fully parallel) run — a complete small Hessian is cheaper than convergence-checking a partial one.** **Too *small* an active set is the mirror failure: it makes the imaginary mode vanish entirely — an all-real spectrum where one negative is expected. **First, free — rule out a *misread* `freq.dat`:** the vtstscripts dynamical-matrix output encodes each mode's sign as a leading **0/1 flag** (`0`=real, `1`=imaginary/negative ω²), *not* a minus sign on the value — so a genuine imaginary mode prints a *positive-looking* magnitude and is missed if you scan the column for a minus sign; a row tagged `1` **is** the negative mode. Re-read that flag column before concluding the spectrum is all-real (freq.dat 0/1-flag convention, VASP 4.6-era ~2006 — verify against your install; later vtstscripts releases may format the sign differently). A *genuinely* missing mode is then the physics case: the most-mobile atom (the migrating species) couples to its first-shell neighbors, so a Hessian over it alone can't resolve the negative curvature — enlarge the set to those neighbors (the dominant lever) — and match the relax's free-atom set: an adsorbate-only subset hides a mode with surface components. Accurate forces are usually secondary to the active set but can be the culprit — a default-loose `EDIFF` (1e-4) with a too-*large* displacement (`POTIM`=0.02 Å) buries the lone mode in force noise (set `EDIFF`≤1e-7, `POTIM`~0.002 Å, `PREC=High`, `LREAL=.FALSE`). More NEB images never recover a missing mode — the fault is the freq calc, not band resolution.**
**Spurious *extra* imaginary modes — first discard the *near-zero* ones, then: under-converged saddle, DOF mismatch, numerics, physical softness, or a *false saddle* from an under-resolved NEB; decide which first.** **Count only *significantly*-negative modes; a *near-zero* mode reading slightly imaginary is numerical noise, not a real imaginary frequency.** The expected near-zero count is **geometry-set: 6** (vacuum cluster / gas-phase fragment — 3 trans + 3 rot, kept *unprojected* in a periodic box; a molecular code like DMol3 removes them), **3** (2D slab in vacuum — 3 trans only, *or* a **fully-free periodic cell** — its 3 translational acoustic modes sit near zero and routinely read slightly *imaginary* in PBC, easily miscounted as real negatives), **0** only with a *frozen* substrate (the partial Hessian breaks translational invariance) ⇒ these read slightly imaginary, **discard them**. **More near-zero modes than that geometry count — or a *block of exact `0.000000`* in freq.dat beyond it — are not physics but an *incomplete Hessian*: displacement steps that failed / were killed / never ran leave blank force rows (`dymmatrix.pl` zero-fills them — prefactor entry), or the saddle is poor. Count the `0.000000` lines against 3N and check the Hessian for missing rows before trusting any frequency.** A true first-order saddle then has **exactly one *significantly*-negative mode**; 2+ significantly-negative ⇒ not a first-order saddle — refine the geometry (tighten the NEB or hand to a dimer) before running/trusting the Hessian. When the extra modes are genuinely significant: a converged saddle (dimer *or* CI-NEB; forces under the *search* `EDIFFG`) that still returns *2+* imaginary modes — meeting the search criterion doesn't guarantee correct vibrational character. **Cheapest first, ~free: tighten and re-relax (precondition 1) — multiple imaginary modes are *most often* just a loose saddle.** **Then a DOF mismatch:** re-run with the freq free-atom set matching the relax; if the extra modes vanish, that was it. Otherwise discriminate cheapest-first: **run a frequency calc on the stable *product/minimum*** — the *same* modes appearing *there* (where there must be none) ⇒ **physical** softness (an intrinsically soft, anharmonic mode — weakly-bound/dissociating adsorbate), not a wrong TS; modes **only at the saddle** ⇒ **numerical *or* geometric**, split by a displacement-amplitude scan (below).
- *Numerical:* step too large (outside the quadratic region) and/or loose SCF. **No universal `POTIM`/`delta`** — Cartesian displacements couple stiff+soft modes, so one value can't resolve every curvature (too large → nonlinear contamination, too small → force noise). Treat the step as a *convergence parameter*: with `EDIFF≤1e-7`, scan downward (0.02→0.01→0.005 Å; ASE `delta`≫0.002 is the usual culprit) and accept where the modes *stabilize* before growing erratically (noise floor). DFPT (`IBRION=7/8`) is displacement-independent (**blocked under DFT+D** — parameters.md).
- *Geometric — a false saddle:* extra modes that **survive the displacement scan unchanged** and are **absent at the product minimum** are neither numerics nor softness — the geometry is genuinely **not a true first-order saddle**, because the **NEB that seeded the dimer used too few images**: the coarse MEP started the dimer (a *local* min-mode optimizer) off the real saddle, so it settled on a higher-order one with residual *real* curvature. `grep IMAGES INCAR`/OUTCAR `NIMAGE` ⇒ **add images, re-converge the band, re-seed the dimer (`neb2dim.pl`)**; retuning `POTIM`/`DISPLACECAR`/the freq step never fixes it. *But* not every higher-order saddle is under-resolution: a too-*symmetric* path can cross a **genuine monkey / 2nd-order saddle** (≥2 true negatives *after* dropping the acoustics) that *no* image count removes ⇒ **break the path symmetry / choose a different mechanism** (neb-diagnostics symmetric-slab 2nd-order-saddle entry), don't densify the same band.
- *Physical softness:* opposite lever — a soft mode's curvature can sit *below* the noise, so the default step (~0.001–0.002 Å) is too *small*. **Raise** it to ~0.005–0.01 Å (`EDIFF≤1e-8`, since curvature ≈ residual force noise); a mixed stiff+soft system needs a *compromise* step — converge it by the same scan. Very anharmonic modes (molecular desorption) hit a **fundamental harmonic-method limit** — uncapturable at any step; once the product is confirmed stable, accept it. **Downstream, the use-case sets their treatment** (~100 cm⁻¹ frustrated adsorbate translation/rotation): negligible for **ZPE** but can **dominate the TST prefactor/entropy** — never blanket-discard the lowest modes. On a surface, hindered translation/rotation needs a beyond-harmonic model (mobile-adsorbate / 2D free-rotor) for rates; in a gas-phase saddle, free rot/trans modes may instead drop from the prefactor when their entropy cancels reactant→TS.
**A tight `EDIFF` only helps if the SCF *reaches* it within `NELM`:** any displacement step that hits `NELM` (default 60) feeds the Hessian a non-self-consistent force, and *one* poisoned step corrupts the whole spectrum — **displaced geometries converge harder than the equilibrium minimum, so a `NELM` that sufficed for the relax can silently fail at the Hessian steps**. **Discard every frequency and rerun**, raising `NELM`→200+ and/or improving the mixer (`ALGO=Fast`). Grep the vib OUTCAR for `aborting loop … EDIFF was not reached` (the NELM-hit tell) + the vib INCAR for `EDIFF`/`NELM`/`IVDW`.
**Rule.** Confirm the TS by a single reaction-mode imaginary frequency at the saddle (highest-energy image or dimer) — a purely *local* test (zero force **and** exactly one *significantly*-negative mode there — *near-zero* imaginary modes, e.g. an isolated molecule's 6 trans+rot, don't count), so intermediate images and imaginary-frequency *magnitude* never rank a TS (magnitude matters only *at* the saddle). One clean mode needs a *tightly-converged* saddle (`EDIFFG ≤ -0.002`, not the ≲0.01 search default), the same free-DOF set as the relax, a convergence-scanned step, and a tight `EDIFF` the SCF actually reaches (or DFPT) — else extra modes are ghosts (above, loose saddle most common), *unless* they're step-invariant and absent at the product minimum, the false-saddle/too-coarse-NEB case (add images, re-seed the dimer); a *missing* mode (all-real) is the inverse — first confirm you didn't misread the freq.dat **0/1 sign flag** (`1`=imaginary, prints a positive-looking magnitude), then suspect an active set too small, enlarge to the moving atom's first-shell neighbors. IRC to the right minima is useful, not required.

## Barrier → rate / diffusivity — a bare NEB barrier is neither a rate, a full diffusion E_a, nor a quantum-corrected barrier
**Question.** "I have the migration/NEB barrier — how do I get the hop rate, or the diffusion activation energy?"
**Rate — needs the prefactor.** The barrier gives only the Boltzmann factor; the rate needs a *prefactor* from a second calculation. In the harmonic approximation (hTST/Vineyard), `k = ν*·exp(−E_mig/k_BT)` with attempt frequency **ν\* = Π νᵢ(reactant) / Π νᵢ(saddle)** — the product of *all positive* normal-mode frequencies at the **reactant minimum** over those at the **saddle** (`ν = ω/2π`). So run a normal-mode frequency calc at *both* geometries: the saddle freq is the same `IBRION=5/6` run that confirms the TS (TS-confirmation entry) — drop its one imaginary mode, keep the rest for the denominator; the reactant minimum supplies the numerator (one more positive mode than the saddle, giving ν\* units of frequency).
**Computing ν\* in practice — `dymseldisp.pl`→`dymmatrix.pl`→`dymprefactor.pl` (vtstscripts), four checks.** Displace the active atoms (`dymseldisp.pl` picks them), assemble the (partial) Hessian (`dymmatrix.pl`), form the ratio (`dymprefactor.pl`). **The prefactor scripts read `freq.dat`** — which this VTST dynamical-matrix route writes, but a *VASP-native* `IBRION=5/6` freq run does **not** (its modes land in the OUTCAR `f =`/`THz` lines): to feed an `IBRION=5/6` spectrum into `dymprefactor.pl`, grep those lines and assemble `freq.dat` by hand (older vtstscripts ~2.03d had no automated extractor — verify the current install; newer releases may include one). **(1) All-zero/imaginary frequencies right after enlarging the displacement shell (or NSW) = an *incomplete* VASP run, not bad physics:** `dymmatrix.pl` silently zero-fills missing displacement steps → a pathological Hessian; confirm every displaced-atom step finished (XDATCAR/OUTCAR step count — a **wall-time kill** mid-sweep is the usual cause; budget walltime ≥ N_displacements × per-SCF cost, EDIFF tight enough no step exits early) and the Hessian has no zero/NaN rows before blaming the modes. *A distinct, since-fixed cause of **all-exactly-zero** frequencies when the run **did** finish every step:* an ancient VTST build (≤3.0b) left the `chain.F` early-return guard `IF (images==0) RETURN` *uncommented*, so a standalone Dynamical Matrix / `ICHAIN=1` run (`images==0`, no NEB) exits *before* the Hessian loop — every frequency comes back identically zero, no error; fixed in 3.0c by commenting that line. Confirm the `VTST: version` banner and grep `chain.F` for that guard *by name* (never the quoted line number — source lines drift), then update the build rather than hand-patching it. **(2) Converge the prefactor against the shell** — expand the displacement sphere and re-run until the prefactor *ratio* stops changing; extra shells add near-identical modes to numerator+denominator so they nearly cancel (cheap, mandatory for a *partial* Hessian — skip only when the system is small enough to displace all atoms, TS-confirmation; same shell scan as the imaginary-mode convergence there). **(3) The 3 rigid-body translational zero modes** (present whenever no atom is frozen) must drop out of the ratio — numerical noise renders them slightly imaginary/positive, and a `freq.dat` still carrying those imaginary entries makes the prefactor script *refuse to run*; strip the near-zero modes from **both** the saddle and reactant `freq.dat` (or confirm `dymprefactor.pl` removes them) before forming the ratio. **(4) Sanity prior:** a solid-state-diffusion prefactor lands ~10¹²–10¹³ /s (THz lattice-vibration scale); far outside that range flags a method error (wrong modes kept, unconverged Hessian, units), not exotic physics — re-examine the dynamical matrix before reporting it.
**Diffusion E_a — intrinsic defects add the formation energy.** The NEB barrier is only the *migration* (hop) energy. When the diffusing defect is *thermally generated* (**intrinsic** — no pre-existing/quenched-in defects), diffusivity = thermal defect *concentration* (`∝ exp(−E_form/k_BT)`) × hop *rate* (`∝ exp(−E_mig/k_BT)`), so the measured Arrhenius activation energy is **E_a = E_form + E_mig**, not the migration barrier alone. Only for **extrinsic** defects (pre-existing, fixed/temperature-independent concentration) does `E_a ≈ E_mig`. Compute the defect formation energy separately and add it to the NEB migration barrier.
**Quantum (ZPE + tunneling) corrections — usually negligible; `dymeffbar.pl` only if a reviewer insists.** The classical barrier omits zero-point energy and tunneling, but both are physically negligible at most catalytic temperatures: above the crossover temperature the quantum partition function → the classical limit. They matter only for *light* atoms (H, Li) at *low* T (below the crossover); when demanded, run `dymeffbar.pl` (vtstscripts) on the converged band — it returns the effective barrier (ZPE + tunneling) vs temperature; report the crossover T above which tunneling is negligible. *Deep-tunneling regime (well below crossover):* the harmonic `dymeffbar` picture breaks down and a **VTST FPI instanton** (Feynman path-integral chain of images) gives the rate directly — but expect its convergence to *look* N× slower than an equivalent NEB: **forces on each image are divided by the chain image count by design**, so the per-image force magnitude is inherently small and a healthy optimizer still creeps. Allow proportionally more ionic steps / relax EDIFFG expectations — slow convergence here is normal, **not** a code bug. (FPI force-scaling per ~2012 VTST instanton; verify against current vtstscripts source before relying on it.)
**Rule.** A bare NEB barrier is neither a rate, a full diffusion activation energy, nor a quantum-corrected barrier. (1) *Rate:* the hTST attempt frequency is the ratio of the products of positive normal-mode frequencies at the reactant vs the saddle — compute frequencies at *both* the reactant minimum and the saddle, never the saddle alone; soft/anharmonic modes that dominate the prefactor get the beyond-harmonic treatment in the TS-confirmation entry. Build it via `dymseldisp.pl`→`dymmatrix.pl`→`dymprefactor.pl`, converge vs the displacement shell, drop the 3 translational zero modes, and sanity-check ~10¹²–10¹³ /s (all-zero/imaginary output = an incomplete run zero-filled by `dymmatrix.pl`, not physics). (2) *Diffusion E_a:* for intrinsic (thermally generated) defects E_a = E_formation + E_migration (concentration × hop); only extrinsic, pre-existing defects give E_a ≈ E_migration. (3) *Quantum corrections:* ZPE/tunneling are negligible at most catalytic T (classical limit) — only light atoms (H, Li) at low T need them; `dymeffbar.pl` applies both vs temperature, and the deep-tunneling regime needs the FPI instanton chain (image forces scale 1/N → ~N× slower convergence than NEB, by design — not a bug). (Expert-stated 2017–2019; statistical thermodynamics / harmonic TST, not version-specific; `IBRION=5/6`→`freq.dat` extraction gap MID/2011, VTST 2.03d — newer releases may automate it, verify the install.)

## NEB endpoint construction — both endpoints are fully relaxed minima, never manually-displaced geometries
**Goal.** Prepare the initial (reactant) and final (product) POSCARs the band will interpolate between.
**Anti-pattern.** Don't nudge a relaxed structure — small manual bond-length displacements, a hand-perturbed geometry — and feed it straight into `nebmake`. A displaced-but-unrelaxed endpoint isn't an energy minimum, so the interpolation is ill-posed: images converge slowly, settle onto the wrong path, or run **hundreds–thousands of ionic steps without ever converging**, and the barrier is meaningless (the neb-diagnostics us.F entry is the crash version). Any bad starting band — unrelaxed endpoints, or a structural discontinuity like an atom-count/ordering mismatch (an atom "removed" from the final image instead of moved far and relaxed; dissociation entry below) — **cannot converge no matter how many ionic steps (NSW) you give it**: fix the band (re-relax, rebuild, re-inspect), don't add steps. And don't hand-set a *specific* endpoint bond/inter-fragment distance: those are *outputs* of the relaxation, not inputs — place a sensible guess and let the optimizer find the minimum.
**Procedure.** Relax *each* endpoint to its own local minimum as an independent optimization (`IBRION=2` CG to the `EDIFFG` *force* criterion; confirm `reached required accuracy`) *before* interpolating — this is a **separate run from the NEB**, so its optimizer is a free choice that need **not** match the band's (`IBRION=2` is standard and *fine* here though forbidden on the band, which runs `IBRION=3`/`IOPT` — parameters.md); never manually adjust the geometry *between* the two relaxed endpoints. Then interpolate and **eyeball the interpolated band** (`nebmovie.pl`/VESTA) for a sane, collision-free path *before* submitting the NEB (see neb-diagnostics linear-overlap & endpoint-ordering entries for what to catch).
- *Reactant:* relax the starting structure to its minimum.
- *Product, geometry known:* use chemical intuition to place the fragments in favourable binding sites, then relax to the product minimum.
- *Product, geometry unknown:* discover a candidate first — run a **dimer** search from the reactant to step over the barrier, *or* manually break the reactive bond *far enough* and allow a *full* relaxation to fall into a product basin. Relax whatever it finds to a true minimum, then use that as the final endpoint.
- *Solid-solid transformation:* an **ssneb** needs a one-to-one initial→final atom map — so the two phases must carry *identical total atom counts*: to connect *different polymorphs* whose unit cells differ in atoms-per-cell, build commensurate supercells that equalize the count first (e.g. 2×1×1 of the smaller-cell phase vs 1×1×1 of the other). **The final-state POSCAR alone picks the mechanism: nebmake interpolates *purely geometrically* (it knows no chemistry), so the path is fully fixed by the final state's (1) cell shape and (2) atom identity/order** — an assumed atom ordering that differs from the true mechanism yields a spurious, atoms-crossing path (cf. neb-diagnostics endpoint-ordering). So when *several* mechanisms connect the *same composition* endpoints (classically **cell-dominated vs atom-dominated**), nebmake cannot choose between them: construct *one* final-state POSCAR per mechanistic hypothesis — each encoding that mechanism's cell shape **and** atom arrangement — and run a *separate* NEB for each (one mechanism → one final state → one band). *Mechanism unknown:* run **ssdimer** first — it needs *no* final state and reveals the migration mechanism; then encode that mechanism in the final-state POSCAR and run ssneb to refine the barrier. **ssdimer = TSASE's solid-state dimer (SSD), not in vtstcode** — drive it through an ASE-VASP calculator (EON-boundary entry below), and **prototype the search on a cheap empirical potential (LAMMPS/EMT) before committing DFT** — an SSD search is expensive. (An older 2015 note called SSD vtstcode-linkable into VASP; the current TSASE-only split is as of 2016 — defer to it; re-check a newer distribution.)
**Rule.** NEB endpoints must be fully relaxed energy minima, not hand-displaced geometries — construct each by an independent optimization and never hand-tune bond lengths or inter-fragment distances (the relaxation sets them). When the product minimum is unknown, find it first (chemical-intuition placement + relax, a dimer search, or break-the-bond-and-fully-relax); never interpolate from a guessed, unrelaxed final state. Interpolate, visually sanity-check the band, then run. **Scanning a thermodynamic variable (pressure / volume / electric field): re-minimize *both* endpoints at *each* new value — an endpoint that is the ground-state structure at one value can cease to be at another — and inspect the profile for an emergent mid-path minimum or a now-non-monotonic band: that is the fingerprint of a landscape-topology change invalidating the old endpoint pair (adopt the new minimum as an endpoint or split into segments — neb-diagnostics MEP-topology). Seed each value's band from the *previous* converged value's path, not a fresh linear interpolation.** (Expert-stated 2012; not version-specific.) (Special cases below: a far dissociated/desorbed product; a metastable endpoint that turns out above its true minimum — neb-diagnostics MEP-topology.)

## Dissociation / desorption / adsorption / removal barrier — endpoint construction (gas-phase fragment needs no infinite separation)
**Goal.** Activation barrier for a species leaving — or arriving at — a binding site: adsorbate desorbing, bond breaking by removal, or a path whose reactant/product is a nominally *gas-phase* fragment.
**Setup.** Build the far endpoint by displacing the departing atom *far* from the binding site so residual interaction is negligible, then relax to its own minimum like any endpoint (above) — no special geometry beyond non-interacting. Keep it in the cell at the same index — "removed" means moved far, **not deleted**; atom count/order must match the other image or interpolation breaks (neb-diagnostics endpoint-ordering rule).
**A gas-phase endpoint is finite, not infinite — and forces no min-mode method.** A nominally gas-phase reactant/product needs *no* infinite separation, and its energy is *not* "ill-defined" — no reason to drop NEB for a dimer. Place the fragment ~5 Å above the surface and relax: it falls into the nearest local minimum (a near-surface, *physisorbed* gas-phase minimum when adsorption is activated; straight onto the adsorbed site if it isn't). That finite-separation minimum is a valid endpoint, and a *regular NEB* between it and the adsorbed state gives the ad/desorption barrier — no dimer. The endpoint need **not** be the *global* minimum — when the true minimum lies far (spanning the full near→far path would need prohibitively many images), use the relaxed *boundary-region* local minimum as the endpoint and reference the barrier to a **separately-computed** asymptotic/true-minimum energy rather than forcing the band to reach it (NEB imposes no constraint keeping images from drifting past an endpoint — reference energies explicitly). (Boundary-endpoint + separate-reference OLD/2007; the endpoint still relaxes to a *local* minimum per above — just not necessarily the global one.)
**Rule.** A dissociated / far / gas-phase state is a valid NEB endpoint — the sole requirement is each endpoint relaxes to a *local minimum at finite separation* (departing species far enough for negligible interaction, or ~5 Å above the surface relaxed to its near-surface minimum). Infinite separation is never required and a min-mode method never forced; a regular NEB finds the barrier. (Gas-phase endpoint expert-stated 2017; not version-specific.)

## EON capability boundary — no cell relaxation; fix the cell in VASP, or use TSASE SSD for concerted cell changes
**Symptom.** A documented EON optimizer option for cell/box relaxation (`opt_method = box`) is rejected at parse time — `Unknown optMethod: box` — and grepping the source (`Optimizer.cpp`) shows the option isn't there. (Or: you want a solid-solid phase transition whose mechanism *requires* cell shape/volume change, and EON offers no way to relax the cell.)
**Cause.** Documentation–code divergence: EON has no cell-relaxation support — the docs list a feature that was never implemented. The source's absence is authoritative over the docs. EON models phase transitions purely by **atomic motion within a fixed cell**.
**Fix.** If the cell does *not* change during the reaction: relax it externally in VASP (`ISIF=3` for full cell+ions, or the appropriate `ISIF`) first, then hand EON the resulting fixed-cell structure — EON optimizes atom positions only. **But for a *concerted* / martensitic solid-solid transition where the cell change is part of the reaction coordinate, the cell can't be held fixed and EON cannot represent it** — use TSASE's **solid-state dimer (SSD)** instead, which carries full cell degrees of freedom and drives any ASE calculator (VASP or LAMMPS). (Recover the two end phases from the SSD saddle by relaxing along ±the negative mode with cell DOF — see the dimer-saddle→minima entry above.) (As of EON rev 2470, 2022-09; re-check the source if a much newer build claims cell support.)
**Rule.** When EON (or any code) rejects a documented option with `Unknown X`, grep the source before debugging config — if it's absent it's unsupported, not your typo. EON = atom motion in a fixed cell: fine when the cell stays fixed (pre-relax it in VASP via `ISIF`), but for concerted cell transformations reach for TSASE SSD, which handles cell DOF.

## EON run model & scheduler integration — run the `eon` *server* (it owns dispatch); the wrappers it calls need a clean `#!/bin/bash` + full-path binaries
**Entry point — run the server, never `submit_job.sh`.** Start `eon` (the server) from the main working dir: it reads `config.ini`, builds client work packages, and *internally* calls `submit_job.sh` to dispatch workers to the queue. The user **never** runs `submit_job.sh` (or an eon client) by hand — the client is a worker with no task until the server assigns one. **Tell:** the client prints `error: Unknown job: akmc` (or whatever `job =` names in `config.ini`) and dies at once though `submit_job.sh` returned a jobid — it launched standalone with no server-issued work package, so it can't resolve the job type. Fix = run the server, not the wrapper.
**Scheduler rejects the wrapper.** When the server *does* call the wrapper and the scheduler bounces it — `sbatch … does not look like a batch script` / `must start with #!` *even with a visible shebang*, or `sbatch`/`qsub: command not found` — it's an env/setup gap, not an EON bug (EON bridges to the queue via per-cluster wrappers `submit_job.sh`/`queued_job.sh`/`cancel_job.sh`). "Not a batch script", ranked: **(1)** Windows-edited/transferred → invisible CRLF (`\r`) or a UTF-8 BOM before `#!` makes line 1 a bad interpreter path despite a correct-looking shebang — **most common** → `dos2unix submit_job.sh`. **(2)** after a scheduler port (SGE `qsub`→SLURM `sbatch`) the wrapper truly lacks `#!/bin/bash` first → add it. **(3)** no execute bit → `chmod +x`. `command not found` = scheduler binary off the *minimal* PATH of a non-interactive launch → full path. Verify: `file submit_job.sh` (reports CRLF) + `sbatch --test-only`.
**Rule.** EON: the user runs the `eon` *server* from the project root (it owns dispatch and calls the wrappers); `submit_job.sh` is an internal hook, never a user entry point — `error: Unknown job: <method>` = you launched the client/wrapper standalone, so run the server. When the server's wrapper is rejected "not a batch script" but the shebang *looks* right, suspect invisible CR bytes first (`file`→`dos2unix`), then `chmod +x`, then a truly missing `#!/bin/bash`; `command not found` → full-path scheduler binaries. Read scheduler stderr first (same launch-env reflex as the VASP no-OUTCAR/sourcing entry in SKILL.md). (Expert-stated; run-model + scheduler-port infrastructure, not version-specific.)

## vtstscripts scope boundary — saddle/path tools only, not an electronic-structure post-processor
**Ask.** "Any vtstscripts/VTSTTools script for effective masses (or band structure / DOS / optical props)?" — **No.** vtstscripts is a TS toolkit (NEB/dimer/Lanczos saddles + MEP/barrier analysis), nothing electronic-structure. Carrier effective masses come from electronic-*band curvature*, so they need a full k-path band-structure calculation analyzed by a dedicated band-structure post-processor (expert's pointer: a phonopy-style band tool) — redirect there, don't hunt for it in this toolkit. (Lone legacy exception: `doslplot.pl`, a thin DOSCAR-column extractor hard-coded for `LORBIT=10`; `LORBIT=11`'s extra orbital columns shift the indices, so it emits nothing/wrong columns until you bump the `$col` variable in the script — affected ~2014 vtstscripts; verify `$col` in your copy, and still prefer a real DOS tool.)
**Rule.** Out-of-scope analysis (effective masses, band structure, DOS, optics) → vtstscripts doesn't do it; redirect to the right electronic-structure post-processing code (effective masses = band curvature from a full band structure). (Expert-stated 2021; not version-specific.)

## Running the saddle method on an empirical potential (non-DFT) — drive ASE/TSASE; the in-source `EAM_FORCE` is a legacy benchmarking hook, not an interface
**Ask.** "I want the NEB/dimer *method* but *not* DFT — can I drive it with an empirical potential? The VTST source calls `EAM_FORCE`; is that it, and is it part of VASP?"
**Answer.** Supported route: skip VASP — drive the band/dimer through **ASE or TSASE** with an empirical calculator (LAMMPS, EMT, …); the optimizer is force-method agnostic (SKILL), so any calculator works (this is also how you cheaply prototype an expensive SSD/NEB before committing DFT — endpoint-construction · EON-boundary entries). `EAM_FORCE` is **not** that interface and **not** part of VASP: a compile-flag-gated trick that *replaced* the DFT force with an empirical (EAM) force so an exact, error-free force could be compared against another code — enabled only by a build flag + a linked external EAM library, off by default. A legacy benchmarking relic, not a general non-DFT NEB driver.
**Rule.** To run the saddle-search *method* on an empirical potential, drive ASE/TSASE with a LAMMPS/EMT (etc.) calculator — not VASP; the in-source `EAM_FORCE` hook is a compile-flag force-replacement for exact code-to-code force benchmarking, not a supported interface, and may be removed/restructured in current builds (grep the source for `EAM_FORCE` *by name* before relying on it, per the SKILL source-patch reflex). (Source feature ~2009.)

## Saddle-search / min-mode methods are the wrong tool — a *rough* landscape (→ sampling) or a *non-equilibrium high-energy* process (→ AIMD)
**When.** The saddle-search class (NEB/CI-NEB, dimer, Lanczos) presupposes a *smooth single-pathway* PES **and** a reactant in *thermal equilibrium*. Two distinct violations:
- **(a) Rough landscape, many competing pathways** — amorphous/glassy/disordered diffusion, or a solid-solid transformation with many possible mechanisms. Tell: NEB won't converge, or "converges" to one arbitrary path among many.
- **(b) Non-equilibrium, high-energy process** — barriers of *tens of eV* driven by *kinetic-energy deposition*: ion bombardment, radiation damage, collision cascades, threshold displacement energies (TDE). Tell: a target barrier far above any thermal scale (≫ a few eV) for a collision/irradiation event, not a thermally activated hop.
**Why each fails.** **(a)** NEB tolerates many atoms *moving* but not landscape *roughness*: on a rough PES it finds one arbitrary path, not the dominant one, and the rate needs averaging over the distribution of paths — even a clean barrier from that one path doesn't represent the kinetics. **(b)** TST/min-mode methods assume the reactant is *thermally equilibrated*, so the crossing probability ∝ the Boltzmann factor exp(−ΔE/kT); a high-energy collision starts *far* from equilibrium and deposits energy kinetically, so the system does *not* react by crossing near a saddle (the incident energy, tens of eV, far exceeds a typical barrier of a few eV). Invalid *by construction*, not a convergence problem.
**Instead.** **(a)** A sampling/ensemble method — **adaptive kinetic Monte Carlo (aKMC)**: the *automation layer* (`akmc.pl`, now EON) driving VASP that, *from each minimum*, auto-generates random displacements, runs the **many dimer searches**, aborts bad jobs, minimizes from found saddles, and steps the KMC clock — so a *systematic* survey (many dimers from one minimum) is never hand-crafted displacement-by-displacement — for disordered-diffusion event tables; metadynamics / basin hopping for rough transformations (high-T MD extrapolated down is riskier — T can shift the mechanism; an ML potential can accelerate, re-verify key barriers with DFT). **(b)** Ab-initio molecular dynamics (AIMD) for the non-equilibrium collision dynamics — a saddle search is valid only when thermal equilibration holds (kT comparable to the barrier).
**Rule.** Check both premises before trusting any saddle search — a smooth single-pathway PES *and* a near-equilibrium reactant. Many competing pathways ⇒ sampling (aKMC/EON event tables; metadynamics/basin hopping); a high-energy non-equilibrium event (radiation damage, TDE, collision cascade, barriers ≫ kT) ⇒ AIMD — the Boltzmann-factor picture min-mode methods rest on has broken down. (Non-equilibrium boundary expert-stated 2014; aKMC-automation-layer OLD/2007 — `akmc.pl` predates the current EON/vtsttools-akmc interface, but use-the-automation-layer for systematic dimer surveys stands; method-applicability, not version-specific.)
**AIMD caveats (route (b)).** Get right / disclose two things. **(1) Thermostat = phonon sink, not ensemble accuracy.** A small explicit slab has no real heat bath, so energy that should flow into the bulk reflects off the periodic boundary and artificially reheats the surface. Apply *any* reasonable thermostat (Nose-Hoover, `MDALGO=2`/`3`); the thermostat *type* barely matters — what matters is a time constant (`SMASS`) matched to the material's phonon timescale so it absorbs that heat. **(2) No electronic dissipation.** Born-Oppenheimer DFT-MD moves nuclei on one adiabatic surface, so impact ionization, hot-electron redistribution, and non-equilibrium electron-phonon coupling are absent — these need TDDFT/Ehrenfest or empirical two-temperature / electronic-stopping corrections (not in standard VASP). A plain-AIMD TDE/bombardment energy transfer is therefore a *lower bound*; state this limitation explicitly. (Expert-stated 2014; physics, not version-specific.)
