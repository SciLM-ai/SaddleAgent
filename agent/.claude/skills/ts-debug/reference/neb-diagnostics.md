## Before blaming PBC — name the mechanism or drop it
"Periodic boundary conditions" is not a diagnosis. PBC is almost always handled correctly; if you
invoke it, identify the **specific** mechanism + the discriminating check: too-little vacuum → image
interaction (measure the gap; converge E vs vacuum); uncorrected slab dipole (`LDIPOL`/`IDIPOL`/`DIPOL`
— parameters.md) → per-image energy drift; an atom wrapping a cell face so linear interp makes a huge
displacement (a fractional coord jumps ~1 — usually a *valid* min-image path, see the VESTA/PBC-artifact
entry, **not** a bug); spurious bonding to a periodic image (check the minimum-image distance); a charged
cell needing a compensating background (`NELECT` — parameters.md). If you can't name a mechanism + the
check that confirms it, it is **not** PBC — keep looking.

## NEB converges on a rigid analogue but stalls on a *flexible* substrate — soft modes ill-condition the Cartesian band; stage rigid→flexible
**Signature.** The same deck converges cleanly on a rigid/constrained version but, on the fully *flexible* substrate (a soft nanowire, floppy molecule/ligand, mobile adsorbate complex), forces *plateau and never drop* (`F=`/RMS flat across ionic steps, no crash); XDATCAR images drift along the soft rotation/translation/low-frequency mode instead of advancing the reaction coordinate. (Vs a single freely-translating species — SKILL "freeze one atom" — whose forces *oscillate*; here a whole flexible region carries *many* soft modes one frozen atom can't damp. Not the post-convergence soft-mode *spline* artifact below, where forces already met `EDIFFG`.)
**Likeliest cause (ranked).**
1. **Soft internal modes ill-condition the Cartesian optimizer.** VASP/VTST NEB optimizes in Cartesian coordinates; the low-frequency rotation/translation modes of a flexible substrate mix into the reaction coordinate (internal coordinates would decouple them), so the projected force plateaus while images wander the soft subspace — and the elastic band *spuriously lengthens* along the soft directions, degrading image spacing along the true MEP.
2. (Rule out) wrong/bundled path or too-coarse images (entries below) — but here the *rigid analogue converges*, isolating flexibility as the cause.
**Discriminating look.** Does the *rigid/constrained* analogue converge on the same deck? If yes, flexibility is it. Tell: `F=`/RMS flat over many steps + XDATCAR displacing along the floppy mode, not the reaction coordinate.
**Fix family. Stage the relaxation.** Converge the NEB on the **rigid/constrained** system first (freeze the flexible DOF, or use the rigid analogue), then **transfer those converged images as the starting band** for a second NEB on the fully flexible system — seeding it near the true MEP so the optimizer never gets lost in soft-mode space. (Freezing one atom cures only a single rigid-translation zero mode; a broadly flexible substrate needs the staged transfer.)
**Rule.** NEB that converges on a rigid analogue but stalls on the flexible substrate (forces plateau, images drift along a soft mode) is a Cartesian band ill-conditioned by soft internal modes, not an optimizer mis-tune: stage it — converge the constrained/rigid NEB, then restart the flexible NEB from those images. (Expert-stated 2014; not version-specific.)

## Over-constrained selective dynamics — a frozen DOF lies *along* the migration path (interface/defect NEB)
**Signature.** A reflex freeze of a whole plane on *all* atoms (e.g. fix x-y for an interface/slab, leaving only z) then a stalled or wrong-looking band — while an atom near the vacancy/defect needs to move in a *locked* direction. (Converse of the flexible-substrate entry above — there too *much* freedom stalls via soft modes; here too *little* blocks the reaction coordinate. Not the recycled-intermediate `F F F` *mismatch* across images of SKILL triage — there the flags differ between images; here the *right* atoms are wrongly frozen in every image.)
**Cause → look → fix.** Selective dynamics locks a DOF the migration uses, so no valid MEP exists in the constrained subspace → inspect image-to-image displacements (band movie / diff endpoint POSCARs): any atom near the defect moving in a constrained direction confirms it → release selective dynamics on atoms in the defect/vacancy neighbourhood, keeping `F F F` only on truly bulk-like far-field atoms.
**Rule.** For interface/defect NEB, constrain only far-field bulk atoms; atoms adjacent to the vacancy/defect must be unconstrained — verify no frozen DOF lies along the path (image-to-image displacements) before submitting. (Expert-stated 2010; not version-specific.)

## Symmetric-slab NEB — barrier ~2× too high, *or* the band breaks the designed symmetry (one face reacts before the other): both correct physics for a both-face setup, not an error
**Signature.** A both-face-symmetric slab (built to cancel the dipole, the *same* adsorbate/reaction on **both** faces) gives one of two surprises on a clean, converged NEB (forces met `EDIFFG`, sane profile): **(a)** the barrier is roughly *twice* the expected/literature single-reaction value (`ISYM` left active); **(b)** the band *spontaneously breaks* the designed symmetry — the two faces' events no longer happen at once, one completing before the other (two distinct peaks in the per-image profile, images no longer mirror-symmetric) — read by the user as a numerical artifact/bug.
**Likeliest cause (ranked).**
1. **The symmetric both-face setup is a *second-order* saddle** — two equivalent reaction coordinates maximized simultaneously — but NEB seeks minimum-energy paths over *first-order* saddles. **(a)** With `ISYM` active the symmetry is *held*, so the climbing image carries *both* reactions cleanly and the barrier doubles. **(b)** Without that constraint NEB *correctly* breaks the symmetry into two **sequential first-order saddles** (one face reacts, then the other), each with the true single-event barrier — correct physics, **not** a numerical artifact (related: the multi-step-path entry below).
2. (Rule out) inputs-inconsistent / wrong-endpoints — those give *unphysical* spiking/inverted barriers, not a clean ~2× or a smooth sequential split.
**Discriminating look.** POSCAR/CONTCAR: same adsorbate on *both* slab faces ⇒ both-face setup. Clean 2× barrier with `grep ISYM` active ⇒ (a); a converged band whose two faces react in sequence (two profile peaks, geometries no longer mirror-symmetric) ⇒ (b). Decisive cross-check for either: rebuild on **one** face with a dipole correction (`LDIPOL=.TRUE. IDIPOL=3` for a z-slab) — its single-event barrier should equal *half* the symmetric-2× value.
**Fix family.** **(a)** Both faces genuinely reacting under imposed symmetry ⇒ **divide the reported barrier by 2**. **(b)** Accept the symmetry-broken path — it *is* the physical MEP (two sequential single-event saddles). Cleanest for either: if only one event is wanted, **model one event per NEB** — asymmetric slab or adsorbate on one face only — which gives the single-reaction barrier directly and validates the ÷2; **dipole corrections do not require a symmetric slab**.
**Rule.** A symmetric slab with the same adsorbate on *both* faces encodes *two simultaneous* reactions = a second-order saddle. With `ISYM` active NEB reports the doubled barrier (halve it); without it the band *correctly* breaks the symmetry into two sequential first-order saddles (one face finishes first) — that is the right MEP, **never** a numerical artifact, and you must **never constrain symmetry to force the second-order saddle**. For a single-event barrier, model one event per NEB (asymmetric / one-sided slab; a dipole correction needs no symmetric slab) or halve the symmetric value, cross-checking the two. (Barrier-doubling expert-stated 2016; symmetry-breaking-is-physical 2014 — consistent; not version-specific.)

## NEB won't converge with *any* optimizer + a large endpoint IS↔FS energy gap — the endpoints' DFT setup is wrong, not the optimizer
**Signature.** Neither VASP's `IBRION=1` nor the VTST optimizers reach `EDIFFG`: forces drop a little then *grow* into an unphysical geometry, max force pinned on the migrating atom; cycling `IOPT`/timestep doesn't help (LBFGS may also crash `RSPHER: NONLR_S%LMMAX is wrong` — a USPP+high-precision tell, parameters.md). The expert's *first* read: a **large IS↔FS energy gap** — e.g. ~10–15 eV across the band (`grep E0 00/OSZICAR */OSZICAR`, or final `TOTEN`) — bigger than the reaction thermochemistry justifies, especially on a migration/diffusion path whose endpoints should be near-isoenergetic. (Unlike inputs-inconsistent below, both endpoints here share the *same* wrong settings — the gap is real, the physics wrong; unlike ridge-trapped/LBFGS-overshoot, those have sane endpoints.)
**Likeliest cause (ranked).**
1. **Endpoints converged with unphysical DFT settings**, so their energies aren't comparable and there's no real MEP to find: **constrained spin** (`NUPDOWN` set) pinning the total moment so an endpoint can't reach its natural ground state → artificial asymmetry; **ultrasoft pseudos** (`US` POTCAR) not PAW, worsened by **over-high `PREC`/`ENCUT`** for the pseudo; an **awkward/non-orthonormal cell**. No optimizer fixes a band whose endpoints sit on different wrong electronic states.
2. (Rule out) genuinely unconverged endpoints (us.F entry below) or a path bundling several processes (entry below) — those have their own tells, not a large-but-clean endpoint gap with forces merely growing.
**Discriminating look.** Compare the two endpoints' final energies — a ≥~5 eV gap unjustified by chemistry = setup error; `grep TITEL */POTCAR` for `US ` (ultrasoft) vs `PAW_PBE`; `grep -iE 'nupdown|prec|encut' INCAR`.
**Fix family.** Reconverge *both* endpoints sanely *before* touching the optimizer: drop `NUPDOWN` (let each relax its own spin; `ISPIN=2`+`MAGMOM` if magnetic); switch to **PAW** (`PAW_PBE`); `PREC=Normal` with `ENCUT` near the PAW-recommended value (~300 eV for light alkali/alkaline-earth); an orthonormal cell; a reasonable k-mesh. Near-isoenergetic endpoints ⇒ rebuild and the band converges quickly.
**Rule.** A large, chemically-unjustified endpoint energy gap (≥~5 eV, esp. on a should-be-isoenergetic migration path) is a DFT-setup error in the endpoints, never an optimizer/convergence problem: diagnose endpoint settings — spin constraints (`NUPDOWN`), pseudopotential family (PAW not ultrasoft), precision/cutoff, cell — before adjusting `IBRION` or switching VTST optimizers. (Expert-stated 2017, VASP 5.4.4; not version-specific.)

## NEB won't converge, optimizer-insensitive — the *path*, not the optimizer (endpoints bundle several processes, OR the mechanism is genuinely multi-step)
**Signature.** Forces stall above `EDIFFG` and cycling the optimizer (`IOPT=1/2/7`, smaller timestep) never helps — no crash, no short contact. Two faces: **(a)** the band movie shows motion *beyond* the intended coordinate — target atom hops *and* substrate reconstructs *and* layers slip at once (the two endpoints differ in more than one process); **(b)** the *middle* images' forces *rise* while the outer images converge (or read low early then *grow* on restart), and the top image is an *inflection point*, not a clean single-maximum saddle — the band spans more than one saddle; **or** a CI-NEB *pre-converges close* (forces ~0.1 eV/Å) then **stalls** short of a tight `EDIFFG` (~0.05) with the **climbing image oscillating between two interior images of similar energy** — a *double-barrier* MEP whose two competing saddles a single CI cannot both seat. (Cell-deform entry below: reaction atom moves *correctly* while the lattice shears — a cell artifact, `ISIF=2`. Too-few-images: motion *sane*, just under-resolved.)
**Likeliest cause (ranked).**
1. **The IS→FS path is not a single concerted saddle.** (a) endpoints *accidentally* bundle several processes — IS and FS relaxed independently into different reconstructions/registries on top of the intended reaction; or (b) the transformation is *genuinely* multi-step — an assumed *concerted/symmetric* mechanism that actually breaks symmetry and proceeds *stepwise* (incl. a **multi-species adsorption/dissociation** taken as concerted but truly **sequential** — one species adsorbs/leads, the other follows), or a high-DOF system (extended solid, 2D/surface) rearranging through sequential local moves — most starkly a **solid-solid phase transformation** whose path threads metastable intermediate phases — so no single saddle/MEP exists (a concerted path is often a metastable approximation of a stepwise one). Either way the band can't converge and the optimizer is innocent.
2. (Cheaper, rule out) wrong optimizer/timestep, recycled intermediates, loose `EDIFFG`, unrelaxed endpoints (SKILL triage) — but if IOPT cycling did nothing, suspect the path.
**Discriminating look.** *Cheapest first:* after **one** ionic step the max force should be sane (**<~20 eV/Å**) — far higher flags a bad path / atom clash to fix before any optimizer tuning. *Then visualize before re-tuning:* play the band movie (VESTA/OVITO). Motion beyond the target atom(s), or two endpoint POSCARs differing by more than one change ⇒ (a). Middle images worsening while ends converge, an emerging mid-band minimum/shoulder in `neb.dat`, large inter-image spacing (~2–3 Å) or consecutive-tangent angles >90° (the band skipping over intermediate valleys), or — `dist.pl 00/POSCAR 0N/POSCAR` — a single species translating a *long* distance to a far site past a nearer accessible bonding partner ⇒ (b).
**Fix family.** **(a)** Rebuild to isolate one process: displace *only* the target atom to its final site, fully relax at the same level, confirm locally stable with no other DOF changed, re-interpolate, re-view the movie. **(b)** Decompose the mechanism — don't tune: first, if a transferred species reached a *distant* site past a nearer partner, rebuild the far endpoint at its *closest* chemically reasonable binding site (and **relabel equivalent transferred atoms — e.g. *which* H moves — to find the lower-barrier mechanism**); then relax the lowest mid-band image (or the suspected inflection geometry) to pin the intermediate minimum and **split there into one NEB per barrier** (a single CI climbs only the **one global** maximum, so cut a double-barrier path at the intermediate minimum and run each sub-NEB independently, no recombine — *or* seed a dimer/Lanczos from the secondary-max image to capture the non-global saddle CI never climbs), and set `ISYM=0` so symmetry-breaking rearrangements aren't suppressed (more images + conservative `IOPT=3 POTIM≈0.02` buy resolution, not a saddle that isn't there).
**Rule.** Optimizer-insensitive NEB divergence is the band chasing a path that isn't a single saddle — a wrong *path*, not a wrong optimizer. *Before* tuning IOPT, ask whether IS→FS is one concerted saddle: extra atoms moving in the movie ⇒ endpoints bundle processes (redesign to differ by exactly one); middle-image forces rising while ends converge — or a CI near-converging then stalling as it oscillates between two similar-height saddles ⇒ genuinely multi-step / double-barrier (split at the intermediate minimum into one NEB per barrier, no recombine, `ISYM=0`); a species moving a geometrically long distance past a nearer partner ⇒ a missing intermediate (rebuild the endpoint to its nearest accessible site, relabel equivalent transferred atoms, or split). Don't reach for a fancier band method (DNEB/`LDNEB`) on a DFT NEB to rescue this — decompose the mechanism instead (parameters.md `LDNEB` entry).

## Images on inconsistent magnetic/electronic states — stuck force, a non-physical (negative) barrier, *or* a spin-inflated (too-high) barrier, not geometry
**Signature.** Two faces of one *electronic* root (geometry sane, no short contact): **(a)** on a path whose endpoints differ in spin, one interior image's force sits orders of magnitude above its converged neighbors (`nebef.pl`, e.g. ~0.6 vs <0.01 eV/Å), near the spin-crossing point; **(b)** a converged band puts the TS (or any interior image) *below* the endpoints → negative/non-physical barrier or KRA, appearing **sporadically across a batch of identically-set-up runs**, the offending image's magnetization off the smooth IS→FS trend; **(c)** flipping `ISPIN=1→2` turns a previously *symmetric* barrier strongly *asymmetric* on a **charge-polaron / mixed-valence hop** (Fe²⁺↔Fe³⁺) whose per-image moments don't transfer symmetrically about the path midpoint; **(d)** the *whole* barrier comes out ~0.1–0.5 eV *higher* under `ISPIN=2` than `ISPIN=1` while **both** endpoints converge to the *same* closed-shell singlet under either setting — only the saddle/interior images carry a spurious non-zero moment, so a **closed-shell** ground state means the magnetism is the error, not the physics. Most common on **DFT+U / high-spin oxide (perovskite, TM-oxide)** systems. Distinct from linear-overlap (sub-1 Å pair) / index mismatch (cross-overs) — geometric; from inputs-inconsistent (below) — there the inputs *differ* across dirs, here they're *identical*; and from a structural basin-crossing (entry below) — there one image's force spikes *mid-run* from an off-path *geometry* change, magnetization on-trend.
**Likeliest cause (ranked).**
1. **Different images relax to different electronic/magnetic minima.** VASP starts from a *very high* default moment and relaxes it down; an image can stick in a different local minimum than its neighbors. Differing endpoint spins guarantee a crossing where an image locks onto the wrong branch (a); even without, the high-moment start scatters images onto different minima so their energies aren't comparable and the barrier inverts (b); and a uniform endpoint-copied (or spin-neutral) `MAGMOM` on a polaron/mixed-valence hop pins every image on one localized spin, so the moment never transfers at the midpoint where the charge does → unphysical intermediate spin → spurious asymmetric barrier (c); and the same high-moment start can trap the *saddle* in an excited high-spin state when the true ground state is **closed-shell**, so only the barrier-top images magnetize and the whole barrier inflates (d). The *sporadic* batch-to-batch variability is the fingerprint of (a,b); not a geometry/optimizer fault.
2. (Rule out) atom overlap / index mismatch (geometric tell), or inputs-inconsistent across dirs (diff INCAR/POTCAR/KPOINTS — below) before invoking electronic minima.
**Discriminating look.** `grep magnetization */OUTCAR` — the per-step *total* moment (the "number of electrons" line, printed regardless of `LORBIT`; per-atom decomposition needs `LORBIT≥10`) — an outlier off the IS→FS trend with sane geometry (for (d): a non-zero *saddle* moment over *zero*-moment endpoints). Decisive confirmation: re-run the offending NEB with **spin off (`ISPIN=1`) and/or `U=0`**; if the barrier turns physical — for (d), drops below the `ISPIN=2` value — the magnetic state is the culprit.
**Fix family.** Start every image from the *same* physically-motivated state, electronically consistent with the relaxed endpoints: set explicit `MAGMOM` *interpolated from the endpoints' converged moments* (per-image where endpoints differ in spin) — and match `ISPIN`/`LDAUU/J/L` to the IS/FS — instead of the high default, and do the handwork to find the *lowest-energy* magnetic solution (try a few initial moments, keep the lowest) before trusting the production NEB. For a polaron/mixed-valence hop, set the per-image `MAGMOM` so the local moment *switches at the path midpoint* (transfers discretely where the charge does — not endpoint-copied, not linearly smeared) and confirm the per-image per-atom moments come out symmetric about image N/2. A single image still trapped after MAGMOM ⇒ run a standalone single-point at the right spin state and copy its `WAVECAR`+`CHGCAR` into that image dir so the restart inherits the correct density. For the **closed-shell** case (d) the consistent interpolated moment is *zero*: restart the `ISPIN=2` band from the converged `ISPIN=1` CONTCAR (and WAVECAR) with `MAGMOM=0` on all ions, seeding a near-zero spin density so SCF relaxes to the singlet rather than an excited triplet.
**Rule.** On magnetic / DFT+U NEBs never trust VASP's default high-moment start — it lets images fall into different electronic minima: one image stuck at high force (spin-changing path), or, sporadically across a batch, an interior image below the endpoints (negative barrier/KRA). Confirm by toggling `ISPIN=1`/`U=0`; fix with explicit `MAGMOM` on all metals converged to the lowest-energy state, seeding `WAVECAR`+`CHGCAR` from a correct single-point for any stubborn image. On a polaron/mixed-valence hop the per-image `MAGMOM` must transfer the moment at the path midpoint (verify symmetry about N/2) or `ISPIN=2` yields a spurious asymmetric barrier where `ISPIN=1` was symmetric. A *uniformly higher* `ISPIN=2` barrier than `ISPIN=1` with a spuriously magnetic saddle over closed-shell endpoints is the same trap *inflating* (not inverting) the barrier ⇒ restart the band from the `ISPIN=1` CONTCAR/WAVECAR with `MAGMOM=0` (not a positive default) to recover the low-spin saddle. (Expert-stated 2019; polaron-midpoint sub-case 2014; closed-shell-saddle inflation MID/2011 — consistent; not version-specific.)

## Forces *rise* after a healthy NEB (many good steps) — one image into a new geometric basin, the band sliding off a shoulder onto a lower path, or LBFGS memory gone bad; check `IOPT` + geometries before calling it unconverged
**Signature.** A force-based NEB descends cleanly for ~50–300 steps, then forces *climb* instead of fall and the energy profile turns noisy/erratic. **First rule out bad basics** (this entry assumes the band was *ever* healthy): if instead the geometry "goes crazy" early — an adsorbate burrowing *into* the slab, relaxed top layers scrambled, atoms drifting unphysically — and the user blames *fictitious forces*, that's the simpler failure: **unrelaxed endpoints or a bad initial band**, not fictitious forces. Confirm both endpoints hit `reached required accuracy` and visualize the *initial* band (`neb_movie.pl`→xyz) before chasing anything below. Scope tells the (post-healthy) cause apart: **(i)** *one* interior image's max force jumps ×10–100 its neighbors (e.g. ~40 eV/Å), neighbors still near `EDIFFG`; **(ii)** forces rise *band-wide* and image geometries have drifted off the MEP toward a distinct lower-energy region/valley; **(iii)** forces rise band-wide with `IOPT=1` (LBFGS) set, geometries still near the MEP. (The magnetic-minima entry above is *electronic* — off-trend magnetization, present throughout; here it *develops* mid-run. Not ridge-trapped, where forces *plateau* with the barrier stuck high; not too-few-images, where it merely crawls.)
**Likeliest cause (ranked).**
1. **(i) One image fell into a new geometric basin** — an intermediate minimum, local structural/phase transition, or alternative final state unlike the interpolated guess; its new curvature overwhelms the LBFGS/quickmin step-size estimate and that image diverges. Band fine; one image left the assumed path.
2. **(ii) The band slid off a PES shoulder onto a lower-energy pathway** — a shoulder adjacent to the intended MEP gives images a downhill gradient, so they drift into a lower basin/valley and forces rise. *Physical* band migration, not a solver fault.
3. **(iii) LBFGS (`IOPT=1`) inverse-Hessian memory corrupted** — on a curved NEB landscape the accumulated curvature estimate goes bad and emits destabilizing step directions after a long healthy run. *Numerical*; QuickMin/FIRE carry no memory to corrupt. (Distinct from LBFGS *overshooting on switch-in* to a high-force, far-from-basin band — SKILL triage; here it ran healthy first.)
**Discriminating look.** *Free test for (iii):* `IOPT=1` set ⇒ switch to `IOPT=3` (QuickMin) or `IOPT=7` (FIRE), continue from CONTCAR — re-stabilizes ⇒ it was LBFGS memory. *Then inspect image geometries at the onset step* (where forces begin rising): one image ×10–100 above neighbors settling into a *new* low-energy structure (copy `image_XX/CONTCAR`→POSCAR, relax standalone) ⇒ (i); the *band* drifted off the MEP toward a lower-energy region ⇒ (ii).
**Fix family.** **(iii)** swap LBFGS → conservative `IOPT=3`/`7` as the *first* diagnostic (robust on flat/curved PES, no memory); drop `TIMESTEP` if the step is still hot. **(i)** relax the offending image alone — a new minimum is an intermediate or better final state the path must treat explicitly (re-interpolate as two segments / adopt as the new endpoint — MEP-topology entry); damp the destabilized step with a smaller `TIMESTEP` meanwhile. **(ii)** re-interpolate the band more directly (or add constraints holding images near the intended path); if the lower path is the *real* MEP, re-derive endpoints and follow it.
**Rule.** Rising forces after a *healthy* NEB are not automatically "unconverged" — check INCAR `IOPT` and inspect image geometries at the onset step before grinding convergence. With `IOPT=1`, swap to `IOPT=3`/`7` as a free test (LBFGS memory corrupts on curved bands; QuickMin/FIRE don't). If that doesn't help, the band physically left the MEP: one image into a new geometric basin (relax it standalone) or the whole band sliding off a shoulder onto a lower path (re-interpolate/constrain). But *before* any of that, when geometry explodes / atoms drift unphysically and the user blames *fictitious forces*, check the basics first — fully relaxed endpoints and a visualized, sane initial band (`neb_movie.pl`); that's the cause far more often than fictitious forces or any algorithmic artifact. (Single-image basin expert-stated 2017; LBFGS-memory + shoulder-slide 2013; basics-first/not-fictitious-forces reflex OLD/2007 — consistent, not version-specific.)

## SS-NEB `ISIF=3` forces won't converge / `constr_cell_relax.F` ignored — drop to fixed-cell `ISIF=2`; patch `neb.F` only if the cell must vary
**Signature.** An `ISIF=3` SS-NEB whose forces never fall after many steps (smaller `TIMESTEP`/tighter `EDIFFG` change nothing) while cell *angles* (z-x, z-y) drift — often the user then patches `constr_cell_relax.F` to hold a direction (e.g. Z), rebuilds, and it has *no effect*: every lattice parameter still drifts. User blames the patch.
**Likeliest cause (ranked).**
1. **The cell shouldn't be relaxing.** Full cell freedom lets the shape change in ways incompatible with the band, so forces never converge — and usually the reaction doesn't need a variable cell at all (parameters.md ISIF entry).
2. **SS-NEB integrates the cell from a stress tensor *inside* `neb.F`, not the VASP `constr_cell_relax` hook** — patching the VASP-side file never masks the stress SS-NEB uses, so all components stay free. (Rule out: patch wrong/not built — but then the *plain* `ISIF=3` relax fails too; here it works there, fails only on the band.)
**Discriminating look.** Diff CONTCAR vs POSCAR cell vectors (OUTCAR `direct lattice vectors`/`volume`) — angle/length drift under `ISIF=3` is the tell. Patch holds in a plain `ISIF=3` relax but not SS-NEB ⇒ hook bypassed; grep `neb.F` for the stress-update loop *by name* (`stress(i,j)=0._q` lines), never a quoted line number (drifts across versions; a VASP 5.3.5-era report puts it near line 236).
**Fix family.** **First, if the cell needn't vary, run fixed-cell NEB (`ISIF=2`, ions/layers only)** — the expert default and the cure for the non-convergence above; don't source-patch to hold one direction when you can fix the whole cell (parameters.md). Only when a variable cell is genuinely required, hold a direction *inside* `neb.F`: its stress is **upper-diagonal** (lower triangle `(2,1)`,`(3,1)`,`(3,2)` pre-zeroed), so the six upper elements (`(1,1)`,`(1,2)`,`(1,3)`,`(2,2)`,`(2,3)`,`(3,3)`) are the active cell DOF — zero all but the one(s) to keep free (e.g. only `stress(3,3)` for Z-only), rebuild, verify on a short run. Or run SS-NEB through **TSASE** (ASE-native cell-filter/mask, no Fortran patch).
**Rule.** `ISIF=3` SS-NEB forces stalling with the cell drifting ⇒ almost always **drop to fixed-cell `ISIF=2`** (atoms only): `constr_cell_relax.F` is bypassed on the band, and source-patching to hold one direction is complex and unreliable. Reserve cell freedom for reactions that truly change the cell, then mask the unwanted **upper-triangle** `neb.F` stress (lower pre-zeroed) or use TSASE. (Reported VASP 5.3.5 + VTST; fixed-cell recommendation expert-stated 2015.)

## IDPP band looks identical to the linear one — loose default IDPP `fmax` exits early, not broken / not a silent fallback
**Signature.** `nebmake.py` (IDPP) and `nebmake.pl` (linear) give visually indistinguishable initial bands for the same endpoints — typically a clean bond-breaking/forming path with no atom overlap, bond lengths already varying ~linearly. The user suspects IDPP is broken or silently falling back to linear. (Contrast the linear-interpolation-overlap entry below, where the two differ *sharply* because the straight line collides atoms.)
**Likeliest cause (ranked).**
1. **The IDPP optimizer's default `fmax` stop is loose, so it quits (~step 0) before reshaping the band.** IDPP minimizes an image-pair-distance cost; when bond lengths already change ~linearly the straight path sits at/below the default `fmax` (≈0.05 eV/Å), so the two *look* identical even though IDPP ≠ linear in principle.
2. (Rule out) IDPP genuinely not running — but that prints *no* IDPP iterations at all, vs converging on step 0.
**Discriminating look.** Read the IDPP force `nebmake.py` prints before it stops; then lower the `fmax` passed to the ASE IDPP method in `nebmake.py` and rebuild. The path now shifting = the default masked a better band; barely moving = linear was genuinely near the IDPP minimum for this geometry.
**Fix family.** Tighten IDPP `fmax` in `nebmake.py` to force the true minimum before trusting linear ≈ IDPP. In the fmax→0 limit IDPP *always* differs from linear except in highly symmetric cases — so no benefit at the loose default isn't proof the path is optimal; only a tightened-`fmax` rebuild that still barely moves is.
**Rule.** IDPP and linear interpolation differ in principle; a loose default `fmax` lets the IDPP optimizer exit early and counterfeit agreement. If IDPP gives no benefit, tighten its `fmax` (toward 0) and rebuild before concluding the linear path is optimal — they coincide only in highly symmetric geometries.

## NEB barrier-less *or* the slab deforms through the band — endpoint cells differ (orientation, shape, or size), linear interpolation invalid
**Signature.** One root cause, two tells — either alone suffices. **(a) Energy:** every image energy falls monotonically IS→FS — no hump, no saddle — *insensitive to the optimizer* (stock VASP, `IOPT=2/3` all give the same featureless slide); vs a real downhill reaction, *tightening* convergence pushes the apparent TS *further below* IS. No crashes/"too close"/spikes — clean but barrier-less. **(b) Geometry:** in the band movie the *reaction* atom moves correctly (e.g. toward the vacuum) but the **slab simultaneously compresses, tilts, or rotates** — collective motion orthogonal to the reaction coordinate. (Distinct from index-mismatch/linear-overlap — cross-overs/100s-eV spikes — and inputs-inconsistent — DFT-drift spikes/offsets the band.)
**Likeliest cause (ranked).**
1. **The endpoint POSCARs have different cells** — lattice vectors differing in orientation (rotated/permuted) *or* in shape/size (one tilted/off-diagonal, the other orthogonal) — usually each endpoint relaxed *with cell freedom* (`ISIF≥3`) and settled differently, or a final cell hand-edited. `nebmake.pl`/VASP linear interpolation interpolates the **lattice vectors as well as the atoms**, so any cell difference smears across the images as cell deformation: the slab shears through the band (b), and the straight line between two cells has no real MEP → every image slides to its lowest point → barrier-less (a).
2. (Rule out) a genuinely activationless (downhill) reaction — but that shape is *stable* under tighter convergence with identical endpoint cells; here the TS sinks *further* below IS as you converge harder.
**Discriminating look.** Diff the cell-vector block (lines 3-5) and atom ordering of image-00 vs image-NN POSCAR — any lattice-vector mismatch is the diagnosis. Play the initial-band movie: a valid path is a smooth slow-motion of *only* the reacting atoms; cell shearing/rotating or a deforming slab confirms it. *Cells match and the movie is clean (no slab deformation) yet still barrier-less?* Suspect **symmetry-pinned endpoints** — relaxed with `ISYM≠0`, settling on a symmetric non-minimum so the band slides downhill from a false IS/FS; grep the *endpoint-relax* INCAR for `ISYM` — the band's own `ISYM=0` doesn't protect the endpoints (parameters.md ISYM entry). If `ISYM` was *already off* in the endpoint relax and all inputs match, the reaction is likely **genuinely barrierless** (direct adsorption/capture from vacuum) — NEB is the wrong tool: confirm no local maximum in the `neb.dat` energy column, then use a constrained or relaxed scan instead of investing in a full band.
**Fix family.** Force both endpoint POSCARs to share *identical* lattice vectors before interpolating: **(a)** re-relax both in one fixed cell — geometry only, `ISIF=2`, **never relax the cell during endpoint optimization for an NEB**; or **(b)** copy the lattice-vector block from one relaxed endpoint into the other and re-relax *ions only* (`ISIF=2`). Re-interpolate; confirm the movie before resubmitting.
**Rule.** Different endpoint cells — orientation *or* shape/size — break NEB interpolation (it acts on the lattice vectors too): the slab deforms through the band and/or the profile goes barrier-less and optimizer-insensitive. Endpoints must share identical lattice vectors (relax `ISIF=2`, never `ISIF≥3` — Fix above). Treat the initial-band movie as a mandatory pre-submission check (slab shears/tilts/rotates/jumps ⇒ won't converge); a barrier-less profile with matching cells *and* a clean movie is symmetry-pinned endpoints — or a genuinely barrierless reaction (verify no `neb.dat` local max before committing a full NEB), not cells (Discriminating look).

## Variable-cell NEB (GSSNEB / SS-NEB) won't converge from a linear-interpolation start — under-constrained shear DOF (starkest for 2D/vdW); bootstrap from a fixed-cell path
**Signature.** A variable-cell NEB (GSSNEB / SS-NEB: `LNEBCELL=.TRUE.` + `ISIF=3` on the band — parameters.md for the tag mechanics) oscillates or won't reach a tight `EDIFFG` (~0.05 eV/Å) from a *linearly-interpolated* start, though coarser criteria do; the two endpoints have *genuinely different* cells (a real phase transition — so the variable cell is the right tool, not the endpoint-cell *mistake* of the barrier-less entry above). Starkest for 2D/layered (van der Waals) slabs with a vacuum (z) direction. Often paired with "I ran *fixed-cell* NEB (`LNEBCELL=.FALSE.`, `ISIF=2`) as a workaround because the cells differ — is that path even valid?"
**Upstream trigger — is SS-NEB even warranted?** *Atoms-only* NEB that seats a clean saddle at ambient pressure but, under *significant* pressure (≥~30 GPa), gives an irregular/asymmetric MEP with **no image at a clean energy maximum** is the positive tell that cell DOF is *essential*: under pressure the cell shape/volume relaxes *along* the mechanism, so an atoms-only band cannot represent the path (more images / tighter EDIFFG won't fix it). Switch to SS-NEB (`LNEBCELL=.TRUE.`/`ISIF=3`, or TSASE's cell-enabled NEB), then heed the convergence caveat below. (MID/2012; SS-NEB long available in TSASE.) **And build the under-pressure barrier from *enthalpy*, not energy** — a *separate* cause of the same "splined peak won't sit on the saddle / unsmooth MEP under pressure" symptom: at non-zero pressure the *pV* term is significant, so a barrier/MEP built from bare DFT energies is wrong. Construct it from per-image **enthalpy** `H = E + pV` (TSASE already does); manual fix — read each image's enthalpy from its `OSZICAR`, normalize per atom, overwrite the energy column in `nebss.dat`, re-spline. *(nebbarrier.pl read energy, not enthalpy, as of early April 2012 — a since-patched VTSTscripts bug; verify your nebbarrier.pl reads enthalpy from OSZICAR and re-run nebresults.pl after updating.)* **The *small*-cell-change converse:** cell shifts only slightly between IS/FS (e.g. an interstitial whose two sites have near-but-not-identical cells) — no SS-NEB needed, relax the cell inside *standard* NEB (`ISIF≥3`), but two correctness reflexes apply because differing cell shapes give each image a *different effective plane-wave basis* (the G-vectors inside the ENCUT sphere depend on the cell ⇒ energies aren't directly comparable, a basis-incompleteness/Pulay error growing with the shape difference): **(1) re-run the converged band once from CONTCAR so every image rebuilds on the *same* basis**; **(2) use a higher ENCUT and denser k-mesh** than a fixed-cell run needs. (MID/2011.)
**Likeliest cause (ranked).**
1. **GSSNEB is formally under-constrained.** A change in an off-diagonal cell angle (z-x, z-y *shear*) can be reproduced by atomic displacements, so the optimizer sees degenerate DOF and, from a poor (linear) start, oscillates or crawls without reducing force. Worst for layered/vdW slabs, which pay almost no energy to change z-length or tilt — cell and atoms relax in a *redundant* coordinate space (layers rigidly rotate/expand). Not an optimizer-tuning or endpoint-convergence fault.
2. (Rule out) plain endpoint/optimizer issues — they don't key on a variable-cell band whose coarser-criterion or fixed-cell version already converged.
**Discriminating look.** `grep -iE 'lnebcell|isif' INCAR` confirms the variable cell is active; OUTCAR cell vectors (esp. tilt angles) drifting *without* force reduction is the degeneracy. Diff the endpoint POSCAR cell blocks — differing *only* in z-length and/or tilt confirms the soft-DOF sub-case.
**Fix family. Bootstrap; don't grind GSSNEB from scratch.** **(1)** First pre-converge a **fixed-cell** NEB (`LNEBCELL=.FALSE.`, `ISIF=2`, atoms only) for a reasonable path geometry, then re-enable the cell (`LNEBCELL=.TRUE.`, `ISIF=3`) restarting from those CONTCAR images — the already-relaxed atoms sidestep the worst shear degeneracy and smooth convergence. **(2)** If the cells differ *only* in soft DOF, instead map both endpoints onto one shared cell and keep the fixed-cell NEB (`ISIF=2`; the share-one-cell construction of the barrier-less entry above), removing the redundant coordinates entirely. **(3)** Forced to stay in GSSNEB from a rough start, use a *small* optimizer timestep — it still converges, looking rough along the path. **Validate, don't accept the shortcut:** a fixed-cell result is *not* the final answer for a formally variable-cell problem — normalize the endpoints to a common cell, confirm relative energies are unchanged, then bootstrap GSSNEB from the converged fixed-cell path to confirm the barrier. **Bounds (variable cell unavailable, or you want a cheap bracket):** a frozen-box NEB is a rigorous *upper* bound on the barrier and the relaxed-/variable-cell NEB the *lower* bound — run fixed-cell in the IS cell *and* in the FS cell to bracket it. Best one-shot compromise: a **per-image linearly-interpolated cell** (image 00 keeps the relaxed IS cell, image NN the FS cell, interiors interpolated; ions relaxed `ISIF=2` in each image's frozen cell) — the endpoints then carry their *correct* relaxed-cell energies, so the extracted barrier is meaningful, unlike forcing every image into one endpoint's cell (which mis-relaxes the other endpoint) — *exactly* what early `nebmake.pl` (~2007) did **silently**: it interpolated atomic positions only, no cell-matrix code, so it copied POSCAR1's cell onto every image with no error/warning when endpoint cells differ, leaving a variable-cell path with no cell change at all. **Diff the lattice-vector block (lines 3-5) across the image POSCARs — identical vectors despite differing endpoint cells is the tell.** (Pre-NEBCELL workaround; modern `nebmake.pl`/`LNEBCELL` interpolate the cell — verify your version, the barrier-less entry above assumes the cell *is* interpolated — but prefer the variable-cell NEB itself for the lower bound.)
**Rule.** Variable-cell NEB (GSSNEB/SS-NEB) is under-constrained — off-diagonal shear ≡ atomic displacement — so a linear-interpolation start oscillates or crawls (starkest for 2D/vdW soft DOF), not an optimizer fault. Don't grind from scratch: pre-converge a **fixed-cell** NEB (`LNEBCELL=.FALSE.`, `ISIF=2`), then bootstrap GSSNEB (`LNEBCELL=.TRUE.`/`ISIF=3`) from those CONTCAR images; or, if cells differ only in soft DOF, share one cell and stay fixed-cell. A fixed-cell barrier is an *upper* bound and the variable-cell one a *lower* bound — bracket with both; never accept a fixed-cell shortcut as final for a formally variable-cell problem (validate by bootstrapping GSSNEB — Fix above). (Expert-stated 2015; bounding + per-image linear-interp cell OLD/2009, pre-NEBCELL; tag spelling per parameters.md, varies across VTST versions.)

## SS-NEB `nebss.dat` energy ≠ `nebefs.dat` 'Rel Energy' — `nebss.dat` is *per-atom-normalized* (E/N), `nebefs.dat` is the *raw* barrier; neither is enthalpy
**Signature.** On an SS-NEB the energy in `nebss.dat` disagrees with `nebefs.pl`/`nebefs.dat`'s last column for the *same* band — user asks which is the enthalpy and which is the real barrier height.
**Likeliest cause.** Two outputs on **different scales**, both energies (neither is enthalpy — for the under-pressure `H=E+pV` correction see the GSSNEB entry above): **`nebss.dat`** prints a *normalized* path — `Image, Distance/sqrt(N), (Energy−E0)/N, force/sqrt(N), dir` — dividing energy by N (atoms) and the path coordinate by sqrt(N) so the *same* saddle in a *larger* supercell gives the same curve; **`nebefs.dat`** (written by `nebresults.pl` from `nebefs.pl`; cols `Image Force Stress Volume Magnet RelEnergy`) gives the *raw* energy relative to image 1. They differ by ~factor N and are **not** interchangeable.
**Discriminating look.** Compare the two files' energy columns for the climbing image — a clean ~N× ratio confirms it's the normalization, not a bad run.
**Fix family.** Read the **physical barrier height from `nebefs.dat`** 'Rel Energy' (last col); use **`nebss.dat`** only to compare the *same* mechanism across *different supercell sizes* (its per-atom normalization makes barriers cell-size-independent). Don't mistake the normalized number for the barrier.
**Rule.** SS-NEB reports the barrier on two scales: `nebss.dat` is per-atom-normalized (E/N, dist/sqrt(N), force/sqrt(N)) for cross-cell-size comparison; `nebefs.dat` 'Rel Energy' is the actual unnormalized barrier vs image 1 — they differ by a factor of N, both correct for their purpose, neither is enthalpy. (Stated 2011; nebss.dat column layout may vary across VTST revisions — verify the header.)

## NEB convergence metric — the NEB-projected (path-perpendicular) force (OUTCAR `FORCES: max atom, RMS`; std-NEB also `CHAIN+TOTAL`), not image energy, not the full per-atom `TOTAL-FORCE`/`vef`, not the `NEB: Tangent` unit vector
**Signature.** "Is it converged?" confusions, one root — *which* number must meet `|EDIFFG|`. **(a)** an image energy fails to drop (non-monotonic/*rising*) while force norms keep falling (`imageN-fe.dat`: col 2 up, col 1 down) — *or* a beginner reads each image's OSZICAR, sees the *absolute* energies look "all the same", and concludes the NEB did nothing (the tiny SCF differences are not the barrier). **(b)** a converged NEB shows *much larger* forces from the wrong row — the **full per-atom force** (`grep TOTAL-FORCE OUTCAR`, or `vef.pl`/`vef.py`) puts individual atoms several× `|EDIFFG|`, *or* the per-image **`NEB: forces: par spring, perp REAL, dneb`** line's `perp REAL` column (a whole-image *norm*, not a per-atom max) reads above `|EDIFFG|`, while `nebresults.pl`/`FORCES: max atom` sits below. **(c)** which OUTCAR row meets the threshold? std-NEB labels the projected force `CHAIN+TOTAL` (`CHAIN`=spring, `TOTAL`=ionic) but **CI-NEB prints no `CHAIN+TOTAL`**. **(d)** large components under the `NEB: Tangent` header alarm the user. **(e)** across *restart cycles* the barrier creeps *upward* and the climbing image lands on a *different* image each restart, misread as "converging toward the value." (Distinct from the ridge-trapped entry below: there forces *plateau above* EDIFFG with the *barrier* stuck wrong-high; here forces descend *toward* it.)
**Likeliest cause (ranked).**
1. **(a) Correct CI-NEB behavior — the climbing image is driven *uphill* to the saddle.** With `LCLIMB=.TRUE.` its along-band force is inverted so it actively ascends; energy *rising* is intended. Non-climbing images need not descend monotonically either — only the net force matters.
2. **(b,c,d) One force, several names — plus decoys that aren't it.** The reported NEB force is the **projected** force = perpendicular *real* force **+** parallel *spring* force: the real force's path-**parallel** (tangential) component is dropped — it climbs the barrier and never converges *by design* — and replaced by the spring. Reconstructable per atom from the OUTCAR rows the user already sees: **Fneb = F − (F·T)T + Fspring**, F the per-atom `TOTAL-FORCE`, T the unit `NEB: Tangent`, Fspring = k(|R_next−R| − |R−R_prev|)T along the tangent (k the spring constant) — so `FORCES: max atom` is max |Fneb| over all atoms of all *interior* images; the **climbing image drops the spring and *flips* the parallel real component** instead: F − 2(F·T)T. Only this projection is optimized and checked; it's read from the OUTCAR **`FORCES: max atom, RMS`** line (the NEB force, max-atom reduced = the scalar checked) and `nebresults.pl`/`nebef.pl` 'max force', in *both* NEB and CI-NEB. Std-NEB *additionally* labels it `CHAIN+TOTAL`; CI-NEB omits it, so `FORCES: max atom, RMS` is the portable readout. **Never the criterion:** the per-atom `POSITION … TOTAL-FORCE` block (what `grep TOTAL-FORCE`/`vef.pl`/`vef.py` read — the *full* real force *including* the parallel component, so individual atoms routinely sit several× `|EDIFFG|` at convergence: expected, not failure), the per-image `NEB: forces: …, perp REAL, dneb` *norms* (the `perp REAL` column legitimately reads above the max-atom figure), `CHAIN`/`TOTAL` alone, or the **`NEB: Tangent`** block — the energy-based improved-upwinding tangent (JCP 113, 9978 (2000); energy-weighted neighbor average at the saddle image), a path-direction *unit vector* (|t|=1, never compared to `EDIFFG`) — it gives the local reaction-coordinate *direction* (the dimer's lowest-mode axis is the analogue, dimer-diagnostics.md), distinct from the plotted "reaction coordinate" of the energy profile, which is cumulative arc length along the MEP.
**Discriminating look.** Read the *projected* force — not the energy, not the full per-atom force: OUTCAR `FORCES: max atom, RMS` (NEB *and* CI-NEB), `nebresults.pl`/`nebef.pl` (modern vtstscripts: `nebef.py`) 'max force' (whole band at once), or `imageN-fe.dat` col 1 — trending to `|EDIFFG|` ⇒ healthy. **`neb.dat`'s force column is *not* this:** `nebbarrier.pl` writes there the *real* force projected *along* the tangent (`NEB: projections … (spring, REAL)`, the REAL value; `neb.dat` col 4 of 5) = the energy *slope* dE/ds (its **sign** marks uphill/downhill along the band — negative ⇒ energy falling, positive ⇒ rising — not convergence progress) — it builds the barrier spline and stays non-zero on a converged band (≈0 only at endpoints/saddle), so it is **never** a convergence metric — but a *large* col-4 magnitude at the **highest-energy image of a plain (non-climbing) NEB** does read out that *no image sits at the saddle*, so that band's spline maximum **overestimates** the barrier: turn on `LCLIMB` and the climbing image seats onto the true saddle for the authoritative — and expectedly *lower* — barrier (a CI-NEB barrier below the plain-NEB value, often by tenths of an eV, with a longer band, is correct physics, **not** a bug). Checking `|EDIFFG|` against `nebbarrier.pl` instead of `nebef.pl` causes premature termination or needless continuation. `TOTAL-FORCE`/`vef` atoms above `|EDIFFG|` while `FORCES: max atom` is below = expected (the parallel component), not divergence; rising energy + falling force = correct climbing ascent. **Read the energy *profile* itself only from `nebresults.pl`/`neb.dat`** — per-image energy *relative to image 0*, reaction coordinate, and force in one table — never raw per-image OSZICAR *absolute* totals (they differ by only small SCF amounts, look "identical", and are not the barrier). Don't grep `CHAIN+TOTAL` in CI-NEB (not printed); never read `NEB: Tangent` as a force. *Per-step history* (all ionic steps, not just the last): `nebef.pl` gives only the final step; run `vef.pl` in one image dir for that image's full force/energy history, else parse OUTCAR `F=` blocks by hand.
**Fix family.** None — assessment only.
**Rule.** Convergence is a *force* criterion on the **NEB-projected** force — the real force with its path-**parallel** (tangential) component removed (carried by spring/CI, outside `EDIFFG`) — read from OUTCAR **`FORCES: max atom, RMS`** (NEB and CI-NEB) or `nebresults.pl`/`nebef.pl` 'max force'. Std-NEB also labels it **`CHAIN+TOTAL`**; **CI-NEB prints none.** Reaching `|EDIFFG|` on that projected force *is* a reliable saddle certificate: a poorly-resolved tangent (too few images) can't fake it — it pins the projected force *high* (plain non-convergence), never silently converges to a false saddle (the only "converged-but-wrong" mode is a *real* saddle for the *wrong reaction* from bad endpoints — workflows wrong-coordinate entry, not a tangent artifact). Never judge by the full per-atom `TOTAL-FORCE`/`vef` (parallel component → individual atoms several× `|EDIFFG|`, expected), `CHAIN`/`TOTAL` alone, image energy (climbing image rises by design), or the `NEB: Tangent` unit vector. Behavioral cross-check across **restart cycles**: a converged band *reproduces itself* — a fresh restart from CONTCAR completes in ~1 ionic iteration with no geometry change and the **climbing image stops migrating between restarts**; a barrier still drifting (esp. *upward*) or a CI jumping images each restart is *not* converged — keep cycling from CONTCAR (tighter ENCUT/k-mesh ⇒ fewer cycles; ssNEB needs the most). (Force criterion expert-stated 2019; restart-reproducibility 2016; saddle-certificate semantics 2013; `nebbarrier.pl`=slope vs `nebef.pl`=NEB force MID/2011; Fneb per-atom decomposition formula OLD/2007, physically fundamental, unchanged across VTST versions — consistent.)

## Large `TOTAL DRIFT` in NEB images — not a convergence criterion: *expected* for a frozen-layer slab, only path-geometry on long/under-sampled paths, harmless once the saddle force meets `EDIFFG`
**Signature.** Per-image `TOTAL DRIFT` (OUTCAR / `nebef.pl`) is non-negligible (~0.1–0.6 eV/Å) though the projected per-image forces already meet `EDIFFG`; read as "band unconverged / barrier wrong", often after checking it against the VASP-manual drift threshold.
**Likeliest cause (ranked).**
1. **Frozen lower layers (surface-slab NEB) carry a non-zero net force.** The VASP-manual zero-drift threshold applies only to *fully unconstrained* cells (isolated molecules); with selective-dynamics-frozen substrate layers the net force on the fixed atoms is non-zero but does not affect the dynamics — drift is *irrelevant*, not a metric. NEB itself enforces **no center-of-mass constraint**, and VASP's optimizer COM-zeroing only activates with *no* frozen atoms (ill-defined once any atom is fixed) — so never assume the chain's COM is held; cure genuine translational drift by anchoring a substrate layer, not by expecting automatic COM removal. (Classic VTST NEB has no COM flag; verify newer builds before relying on either behavior.)
2. **Residual ionic drift accumulating over many steps** — long path, sparse images near the saddle, or `LREAL=A` real-space projection error compounding. Degrades *path geometry / spline quality*, not the saddle energy.
3. (Reassure) on a *short* hop whose saddle projected force is below `EDIFFG`, drift does **not** compromise the barrier — no action.
**Discriminating look.** `grep 'TOTAL DRIFT' */OUTCAR`; check selective dynamics in the POSCARs — frozen layers ⇒ ignore drift, but confirm the frozen-atom fractional coords are *identical across all image POSCARs* (a mismatch is a real bug, atom-ordering entry above); else gauge path length / inter-image distance and whether `mep.eps`/the spline is irregular. Frozen-layer slab, or short path + smooth spline + converged saddle force ⇒ ignore.
**Fix family.** Surface slab: freeze the lower layers (selective dynamics), confirm frozen-atom coords match in every image, and ignore the drift magnitude. Otherwise, only when it matters (long path / irregular spline): add images around the saddle, tighten `EDIFF`/`EDIFFG`, or switch `LREAL=A`→`LREAL=.FALSE.` for the final single-point energies (parameters.md).
**Rule.** `TOTAL DRIFT` is not an NEB convergence criterion: for a frozen-layer surface slab it is *expected* (the manual's zero-drift threshold is for fully unconstrained cells), and on long/under-sampled paths it degrades path geometry/spline quality but not the barrier once the saddle projected force meets `EDIFFG`. Freeze lower layers (identical frozen coords across images) for slabs; densify/tighten/`LREAL=.FALSE.` only on long paths or an irregular spline. (Frozen-slab drift OLD/2008; long-path drift expert-stated 2016 — consistent, not version-specific.)

## NEB stalls at the first ionic step, one interior image's SCF diverges — linear-interpolation atom overlap on a hop/vacancy path, rebuild with IDPP
**Signature.** The band makes *zero* ionic progress after many minutes (no successful first ionic step), and an interior image's OSZICAR shows a first-step (DAV/RMM) electronic energy orders of magnitude larger than the endpoints' — e.g. 100–1000× the per-atom value — often never settling. That image's POSCAR has two atoms in unphysical contact (a pair below ~1 Å, or below the covalent-radius sum). Endpoints are fully relaxed and sane. Distinct from the us.F-crash entry below (diverging *endpoint*; wild energies on the first step of *every* image) and from the index-mismatch entry below (visible cross-overs in the movie + "too close" warnings, fixed by *reordering* atoms): here atom ordering is correct, yet the straight-line path still collides atoms in the middle of the band. *Milder same-root variant:* a path that **rotates a group about a bond** (or any large non-linear/angular displacement) — the straight-line chord cuts across the arc and strains bonds — a bond intact at *both* endpoints can read fully *dissociated* in a mid-band image — into high initial forces and poor/stalled convergence **without** any sub-1 Å contact, so the SCF-divergence/`dist.pl` tells are absent; recognize it from endpoints related by a torsion/ring-flip and a band movie cutting straight across the rotation.
**Likeliest cause (ranked).**
1. **Naive linear (straight-line) interpolation drives a migrating atom through an occupied host site.** On a vacancy-migration or atom-hopping path, the Cartesian/direct straight line between endpoints routes the moving atom across (or onto) a lattice site that is occupied in the intermediate, producing a near-zero separation in one image; that image's SCF diverges and the whole band stalls before it can take an ionic step.
2. (Rule out) a diverging *endpoint* (us.F entry) or per-atom index mismatch (cross-over entry) — both give different tells (wild energies on *every* image / movie cross-overs), not a single interior-image energy outlier with sane endpoints.
**Discriminating look.** Compare the *first* DAV/RMM energy line of every image (`grep -m1 'DAV\|RMM' */OSZICAR`, or `head` each) — one image 100–1000× the others localizes the bad image; then inspect that image's POSCAR for any interatomic pair below ~1 Å (or below the covalent sum). Outlier energy + a sub-1 Å pair in that image = linear-interpolation overlap.
**Fix family.** Regenerate the band with **IDPP** (`nebmake.py`, IDPP by default — workflows) or another bond-topology-aware interpolation (ASE **NEB-TR**, internal coordinates): both preserve bond lengths and route the mover *around* occupied sites, not through them. Still colliding ⇒ hand-place one sensible intermediate respecting all interatomic distances and split into two shorter segments. To salvage an *existing* strained band without rebuilding: **`nebavoid.pl`** pushes atoms apart to a minimum separation across the band, or crawl the short contact — cap `MAXMOVE≈0.05–0.1` Å, or a conservative warm-up (`IBRION=3 POTIM≈0.1`, or low `TIMESTEP`) until forces fall below ~1 eV/Å, then hand to a faster optimizer (interim cures; parameters.md). Re-view the rebuilt band before submitting.
**Rule.** When an NEB stalls or crashes on the first step with one interior image's SCF/forces diverging, compare OSZICAR first-step energies across *all* images first — a single wildly high energy is diagnostic of atom overlap from linear interpolation, not a code or endpoint fault. Build with IDPP (`nebmake.py`, IDPP by default) for vacancy-migration / atom-hopping bands, *any path where bonds form or break*, *and any rotation about a bond / large non-linear displacement* (high forces with **no** contact): linear interpolation doesn't preserve bond lengths or follow rotation arcs, so the straight line drags atoms through occupied sites or strains bonds even when endpoints and ordering are perfect. And NEB returns only a *local* MEP, so a poor straight-line guess can converge to a *higher/wrong* saddle, not merely fail — prefer IDPP or a hand-built intermediate for any non-linear path and verify the converged mode is the intended coordinate. Cap `MAXMOVE` only as an interim measure when you cannot rebuild.

## VESTA shows atoms moving the wrong way / edge atoms vanishing, or an interpolated image coord sits outside [0,Latt] — PBC artifact; `nebmake.pl` already min-images, no manual unwrap
**Signature.** Three faces of one root — geometry sane, nothing collides, only the *display* or the user's *expectation* is off: **(a)** in the band movie (post-`nebmake.pl`) an atom drifts/jumps *opposite* the reaction or detours across the cell; **(b)** after a relax an atom that sat at a boundary (frac ≈ 0/1) appears to **vanish** in VESTA though the CONTCAR atom *count* is unchanged; **(c)** an interpolated image coordinate sits *outside* [0,Latt] or jumps ~Latt→0 between adjacent images, prompting "must I manually unwrap the endpoints before interpolating?" — *or* a **relaxed** CONTCAR (e.g. the highest-energy image) whose fractional coords land *beyond the range spanned by the endpoint POSCARs*. Distinct from the cross-over entry below (atoms pass *through* each other ⇒ a real index bug).
**Likeliest cause (ranked).**
1. **PBC is handled correctly — the artifact is in the rendering or the expectation, not the path.** (a) *Min-image:* an atom near a face is wrapped to the other side in one endpoint; a viewer drawing bare fractional coords shows it crossing the whole cell, not a short step. (b) *Boundary-replication:* an atom at frac 0/1 is drawn at *all* equivalent corners; relaxing *off* the boundary collapses the ghost duplicates — looks lost, structure fine. (c) *Interpolation self-wraps:* `nebmake.pl`'s linear interpolation applies the minimum-image convention itself, so an out-of-range/wrapped image coordinate is correct output — a migrating atom genuinely crossing a cell face (out one side, in the opposite) is a **valid PBC path, not an error**; **no manual pre-unwrapping needed** (same for a *relaxed* image straying past the endpoint range — valid PBC). **But** when this happens in a **minimal** cell where *most* atoms translate by large fractions, the band is sampling an infinite-array **cooperative** process (effectively a phase transition), not a localized event: image positions exceeding the endpoint bounds are geometrically legitimate, and the barrier is then a per-*unit-cell* activation energy, **not** a localized-TS energy.
2. (Rule out) genuine index mismatch between endpoints — gives *cross-overs + "too close" warnings* (entry below), not a clean detour, a vanishing edge atom, or an out-of-range coord with intact geometry.
**Discriminating look.** Diff image-00 vs final-image (or pre/post-relax) coords: a frac jump ~1 across a boundary (0.95→0.03), or a "missing" atom shifted *away from* 0/1 — count unchanged either way. The same min-image jump *within* the interpolated band is (c): recognize it, don't fix it. Sane geometry + unchanged count ⇒ artifact; re-view in a minimum-image-aware viewer.
**Fix family.** None to inputs/structure when geometry is sound — and **do not manually unwrap coordinates**: `nebmake.pl` already wraps during interpolation. The *only* user responsibility is consistent atom ordering between the two endpoints (atom i in init = atom i in final — ordering entry below). Wrapping both endpoints to the *same* side of each boundary before `nebmake.pl` is **cosmetic**. The lone legitimate reason to edit coordinates: if min-image routed the atom a way you didn't intend (it should *not* cross that boundary — you want the *other* mechanism), shift that endpoint's fractional coord by ±1 so the intended route is the shorter one, then re-interpolate. To recover a *localized* TS from a minimal-cell cooperative band, **enlarge the supercell** so the moving species is dilute and the bulk is a spectator — otherwise the per-cell collective barrier is meaningful only as such.
**Rule.** "Images move the wrong way", "an edge atom vanished", or "an image coord is out of [0,Latt]" is first-suspicion a PBC effect with geometry intact — not a broken path, lost atom, or a coordinate to unwrap. `nebmake.pl` applies minimum-image during interpolation, so never pre-unwrap; the user's sole responsibility is consistent atom ordering across endpoints, and wrapping to a common image is optional movie cosmetics. The one exception that justifies touching coordinates: a deliberate ±1 endpoint shift to force a non-min-image path you actually want. And coords straying past the endpoint range in a *minimal* cell where most atoms move are not a bug but a **cooperative/per-cell** process (≈ phase transition) — read the barrier as a per-unit-cell quantity, and dilute in a larger supercell to isolate a localized TS. (PBC-robust / no-unwrap principle expert-stated as early as 2006; cooperative-cell interpretation 2007; timeless physics, not version-specific.)

## Low forces / plateau early in cNEB but a suspiciously high, stable barrier — band trapped on a ridge, not converged
**Signature.** Max force settles into a narrow band (e.g. 0.15–0.5 eV/Å) for many steps and the user reads it as "almost converged" — often reinforced by the barrier "not changing radically" and a plausible imaginary frequency. But the barrier sits at an implausibly *high*, stable value (e.g. >1 eV), and tightening EDIFF or shrinking TIMESTEP does *not* drive it down. The tell that separates this from a slow crawl (parameters.md TIMESTEP entry): pushing the band *harder* (larger TIMESTEP or a 2nd-order optimizer) makes the climbing-image force *rise sharply* — to several, even tens of eV/Å — **while the barrier steadily drops**. That is the band finally leaving the ridge for the true saddle, not a divergence.
**Likeliest cause (ranked).**
1. **Initial band sitting on a ridge / flat shoulder of the PES, not on the MEP.** The low-force region is not the saddle but a flat shoulder the band must traverse with significant geometry change to reach the real saddle. Keeping forces low (small TIMESTEP, conservative optimizer) merely *pins* the band on the ridge so it never relaxes off it, and the barrier stays stuck high.
2. (Rule out) genuine slow crawl from an over-small TIMESTEP — there the barrier is already near-correct and forces *descend* slowly; here the barrier is wrong-high and forces *rise* when you push (parameters.md).
**Discriminating look.** Track the *barrier per ionic step*, not max force: `grep 'F= ' OSZICAR` (or per-image `neb.dat`/`nebef.dat`) over 50+ steps — a barrier still trending down means *not converged* however low the forces look. Cheapest decisive test: run two short trials, conservative (`IOPT=3 TIMESTEP=0.05`) vs aggressive (`TIMESTEP=0.1`, or `IOPT=1 INVCURV~0.005`); if the aggressive trial's CI force spikes while the barrier falls, the conservative one was ridge-trapped.
**Fix family.** Do *not* tune convergence criteria or reduce TIMESTEP when forces plateau on a high barrier. Push the band off the ridge: raise TIMESTEP (0.05→0.1), or — fastest — switch to the second-order optimizer `IOPT=1` with `INVCURV≈0.005` (parameters.md). Transient climbing-image forces of tens of eV/Å under these settings are *expected* as the band crosses a steep stretch of PES; continue as long as the barrier is monotonically decreasing. Abort or revert only if the *barrier itself* rises or oscillates without trend.
**Rule.** Low forces early in a cNEB are not convergence — when the initial guess is far from the MEP, the true convergence test is the *barrier*, not the force magnitude. Monitor barrier descent and push *harder* (large TIMESTEP or `IOPT=1 INVCURV~0.005`) to escape ridge-trapped geometries; read transient force spikes as the band relaxing toward the saddle, not as failure. Abort only when the barrier stops converging.

## Nonsensical / garbled / image-count-dependent NEB barrier or mep.eps, *or* a spurious bump at image 0 — post-processing produces a wrong profile (an OUTCAR it can't read: endpoint dir/OUTCAR missing, or present-but-*unparseable* gzip/zip/empty/truncated; *or* a `nebbarrier.pl` image-0 tangent sign-inversion bug), not a bad run
**Signature.** The barrier/plot is meaningless because post-processing mis-handles an OUTCAR or computes a wrong tangent — four flavors: **(a)** the plot looks "crazy" and *changes drastically between two runs differing only in image count*, one endpoint energy wildly off the relaxed-minimum reference — the band lacks a constrained final image to interpolate/spline toward; **(b)** the endpoint OUTCAR is missing — `nebbarrier.pl`/`nebspline.pl` either **abort printing `No OUTCAR in 00`/`N+1`** (the user often "fixes" it by *deleting* those endpoint dirs, since they "normally hold only POSCAR", so the scripts run but the MEP omits the endpoints), **die `Illegal division by zero` in `nebspline.pl`** leaving `spline.dat`/`exts.dat` **zero-length** and **no `mep.eps`** (the absent endpoint reaction-coordinate makes the spline divide by a zero interval — a *missing endpoint OUTCAR* cause, distinct from the all-`0.0` *distance*-column div-by-zero of the VTST-not-linked / missing-`dist.pl` entry below), or (other versions) default silently so every image reads a large *negative absolute* energy (within ~1 eV of each other), image-00 exactly `0.000000`, like raw DFT totals not relative heights — the OUTCARs that exist are fine (post-processing only); **(c)** `nebresults.pl` runs without error but `mep.eps` is *garbled/incomplete* (nonsensical profile) even with all N+2 dirs present and the run finished — one or more image OUTCARs (any dir, incl. `00`/`N+1`) is **compressed** (gzip/zip), which the Perl scripts silently skip/misread; **(d)** a *spurious energy bump* between images **0 and 1** in the `nebspline.pl`/`mep.eps` profile (an apparent small barrier at the *first endpoint*) while the run is clean, with `neb.dat` col 4 (image-0 tangential force) pointing *opposite* the geometry trend — appearing **only when the first endpoint is not a true minimum** (e.g. an adsorption reactant far from the surface); absent when image-0 force ≈ 0.
**Likeliest cause (ranked).**
1. **A whole endpoint directory is missing.** An N-image NEB needs **N+2 dirs** (`00`…`N+1`); `00`/`N+1` hold the *fixed* relaxed endpoints the spline/interpolation anchors on. No final dir ⇒ interpolation, spline, and barrier are nonsense and disagree across image counts (anchored differently or not at all). Count dirs *before* diagnosing convergence.
2. **OUTCAR present but unreadable by the scripts — missing-at-endpoint, *or* compressed/empty/truncated in any dir.** The saddle-energy/MEP scripts (`nebbarrier.pl`/`nebef.pl`/`nebspline.pl`/`nebresults.pl`) read references from `00/OUTCAR` and `N+1/OUTCAR` (made by the *separate* endpoint relaxations — **post-processing-only: the band's SCF/force/ionic steps proceed normally without them**, so those dirs normally hold only POSCAR/CONTCAR; copy an endpoint OUTCAR in *solely* for the MEP/barrier readout — the scripts grep only its final *energy*, all NEB geometry (endpoints included) comes from POSCAR/CONTCAR, so never geometry-edit an endpoint OUTCAR to match a reordered/edited POSCAR): a missing endpoint OUTCAR defaults image-00 to `0.0` (Signature b). A **gzip/zip-compressed** OUTCAR (any image dir, endpoints included) is read as binary and silently skipped/misread → that image's energy is corrupt/missing → a garbled `mep.eps`. `vfin.pl` runs fine without endpoint OUTCARs — a working `vfin.pl` is *not* evidence the references are present.
3. **(d) `nebbarrier.pl` inverts the image-0 tangent sign.** Near line ~97 it calls `nebforces.pl` with *swapped* file arguments (`$file1 $file2` rather than `$file2 $file1`), flipping the gradient sign at the first image. Harmless at a true minimum (tangential force ≈ 0), but a non-minimum endpoint then shows a spurious bump between images 0 and 1 — the run and OUTCARs are fine, the readout isn't. *(vtstscripts bug as of early 2015; likely fixed in later revisions — verify the arg order before patching.)*
**Discriminating look.** Count dirs first: `ls -d [0-9][0-9]/` must show `00`…`N+1` (IMAGES+2); a missing final dir = cause 1. If all dirs exist: `grep -E 'TOTEN|E0=' 00/OUTCAR` absent + image-00 = `0.000000` in `neb.dat` = missing endpoint OUTCAR; if OUTCARs exist but `mep.eps` is garbled, verify each is plaintext — `file */OUTCAR` (or `head -c4` for the gzip `\x1f\x8b` magic) flags a compressed one. Distinguish from input-mismatch (below): there real totals are incommensurable but *non-zero*; here a missing/unreadable OUTCAR breaks the profile. For **(d)**: check the `neb.dat` col-4 sign at image 0 against the geometry trend, confirm the bump is localized to images 0–1 and vanishes when image 0 is a minimum, then read `nebbarrier.pl` ~line 97 for the `nebforces.pl` argument order.
**Fix family.** Provide all **N+2** dirs, `00`/`N+1` each with a valid POSCAR/CONTCAR *and a relaxed-endpoint OUTCAR* — copy one in, or run a standalone single-point in each dir so an OUTCAR with energy+forces exists; never leave the final dir absent or empty, and **never delete the endpoint dirs to silence a `No OUTCAR` complaint** — that just drops the endpoints from the MEP. **Decompress every image OUTCAR (all dirs incl. `00`/`N+1`) to plaintext before `nebresults.pl`/`nebspline.pl`** — the scripts can't parse a gzip/zip OUTCAR. Re-run the analysis and barriers become relative and image-count-stable. **(d)** After confirming the bug (newer scripts may already be correct), swap the `nebforces.pl` argument order in `nebbarrier.pl` (~line 97) to `$file2 $file1 $ocar`; for *stitched multi-resolution* NEBs, take the shared-boundary-image gradient from the **coarser** calculation rather than the fine-NEB image-0 value.
**Rule.** An N-image NEB needs N+2 dirs (`00`…`N+1`), each with a *plaintext* relaxed endpoint/image OUTCAR: the post-processing scripts silently skip or misread any OUTCAR they can't parse — dir/OUTCAR missing, empty, truncated, *or gzip/zip-compressed* — yielding a crazy, image-count-dependent, or garbled `mep.eps`. So when a barrier/plot looks wrong, check structure first: count dirs, confirm `00`/final OUTCARs are present, and `file */OUTCAR` that every one is uncompressed — a structure/reporting problem, not a failed calculation. And a spurious bump at the *first* image with a wrong-signed image-0 force — present *only* when that endpoint isn't a minimum — is an old `nebbarrier.pl` tangent-sign bug (swapped `nebforces.pl` args, ~line 97), not a bad run: verify/patch the script. A *self-coded* spline (not `nebspline.pl`) returning orders-of-magnitude-wrong energies (e.g. ~10 eV over a 0.5 eV barrier, discontinuous between segments) or a wiggly path has two classic transcription bugs: the JCP 113, 9978 (2000) original text has a **bracket typo in the cubic-spline coefficients** — use the **erratum (9978e)**; and the per-image parallel (tangential) force must **change sign across the saddle** (negative before, positive after) — all-same-sign forces wiggle whatever the cubic formula. `nebspline.pl` already does both correctly. (Compressed-OUTCAR failure expert-stated 2018; image-0 sign bug expert-stated early 2015, likely fixed in later vtstscripts — verify before patching; self-coded-spline erratum/sign caveat OLD/2010, irrelevant if using vtstscripts; missing-endpoint `nebspline.pl` div-by-zero presentation OLD/2006 — architectural, applies to all VTST versions.)

## NEB appears frozen, no VASP error — wall-time kill vs internal stall vs MPI-finalize hang
**Signature.** The run stops, or stops *advancing*, with no VASP/VTST error in OUTCAR or stderr; the user assumes a convergence hang or code bug and reaches for INCAR. Three flavors: (a) the band has no further steps after a specific ionic step (a non-round number like 84 or 869) and the process is *gone*; (b) the process is *alive* but every image's OUTCAR stopped updating mid-run, the tail ending on electronic-step (`RMM`/`DAV`) lines with no following `F=` ionic/LBFGS line — typically misread as an LBFGS-optimizer hang; (c) the job sits in RUNNING for hours/days with no new OUTCAR writes and no error, but the compute work actually *finished* and MPI is blocking on exit.
**Likeliest cause (ranked).**
1. **Process dead, clean stop → batch wall-time kill.** The scheduler terminated the job at the allocation limit. An external SIGTERM/SIGKILL leaves OUTCAR truncated mid-run with no error message, so it is *indistinguishable* from an internal hang until you read the job log. The user frequently misreads this clean walltime stop as the optimizer *"not converging"* / stuck — so before changing anything, confirm the run was actually *healthy*: max force still decreasing toward `EDIFFG` **and** the barrier still trending toward a single maximum (per-image energy deltas, not oscillating). If both hold, it merely ran out of time; "run stopped" ≠ "run stuck."
2. **Process alive, frozen *inside one image's SCF* → internal VASP electronic stall, not VTST/LBFGS.** The OUTCAR tail is electronic (`RMM`/`DAV`) with no `F=` line: VASP never reached the optimizer, so the hang is in the SCF (electronic) loop even though it surfaces during an LBFGS phase — don't blame IOPT/LBFGS. Multi-image NEB images sync at each ionic step, so *one* image stuck in SCF freezes *all* OUTCARs simultaneously; compare per-image tails to find the single image still printing electronic steps.
3. **Process alive but compute finished, MPI blocked on exit → scheduler/MPI infrastructure, not a code bug.** The calculation did its work; the MPI communicator never propagated the termination signal, so the job blocks indefinitely in `MPI_Finalize`/exit and can't release the allocation. Hallmark: RUNNING for hours with *no* new OUTCAR writes and *no* error, while the OUTCAR tail shows a completed-looking final step.
**Discriminating look.** Cause 1: read the *scheduler* log — `grep -iE 'wall|time limit|CANCELLED|TERM|killed|signal' <job-stdout>`; a wall/CANCELLED/signal-9|15 hit, with OUTCAR ionic steps *absent* after the stop (no error block), confirms an external kill. Cause 2 vs 3 (process still listed): a per-image tail ending on `RMM`/`DAV` with no `F=` = that image stuck in SCF (cause 2); *all* tails showing a *finished* step (`reached required accuracy`/NSW, final energies & timings) that won't exit = MPI blocked on finalize (cause 3).
**Fix family.** Cause 1: extend the wall-time allocation; stage the restart with `vfin.pl <subdir>` (CONTCAR→POSCAR per image + restart files), resubmit — **make no INCAR/parameter edits when force and barrier were still improving** (walltime ≠ stagnation); on a confirmed-clean trajectory you may switch first→second-order optimizer for throughput (`IOPT=3 → 1/7`, parameters.md). Cause 2: first **continue (restart) rather than kill** — a continuation often sidesteps the stall or VASP self-recovers; try this *before* declaring failure. If it *recurs* (the same image, across restarts and machines), it's a genuine SCF-convergence failure in that image — repair its electronics (`ALGO`/`NELM`/`EDIFF`, charge-density/k-point init); and when the *identical* inputs converge cleanly on a reference machine/build, suspect a platform-specific VASP **compile/MPI/library** mismatch, not the inputs or VTST. Cause 3: safe to **kill and resubmit as a continuation** (ISTART=1, ICHARG=1; CONTCAR→POSCAR per image); separately fix the scheduler's 'stop processes on exit' setting. No INCAR change is warranted in any case.
**Rule.** When a NEB (or dimer) job appears frozen with *no* VASP error, check whether the process is alive and stdout advancing, then read the OUTCAR tail — *before* touching INCAR. Dead + steps absent → wall-time kill (extend allocation, `vfin.pl`, resubmit **from CONTCAR unchanged when force + barrier were still descending — walltime ≠ stagnation**). Alive + an image's OUTCAR frozen on `RMM`/`DAV` (no `F=`) → stuck in *SCF*, not LBFGS (continue/restart often recovers; if it recurs, fix that image's electronics or suspect a platform/compile mismatch — never IOPT). Alive but the work looks *done* yet won't exit → MPI blocking on finalize: kill and continue, and fix the scheduler ('stop processes on exit'), never blame VTST.

## Atoms run through each other / "too close" warnings, image energies 100s–1000s eV high — endpoint/image atom-ordering (index) mismatch *or a malformed POSCAR mis-parsed by `nebmake.pl`*, not a real path
**Signature.** The interpolated band shows *many* atoms moving at once — atoms passing *through* each other (cross-overs) instead of one translating smoothly; the movie may instead read as *unintelligible/stationary*. Per-image OUTCARs emit "too close" warnings and intermediate image energies sit 100s–1000s eV *above* the endpoints. The *same* root cause also presents *without* a crash — a **multi-humped profile (2+ maxima) whose interior images never converge**, or **stalling at ionic step 1 despite large forces** — the tell being an atom forced *tens of Å* between adjacent images where normal diffusion is sub-Å/image, or one placed *outside the bounding box of both endpoints* (beyond any endpoint atom); chemically *identical* atoms swapping sites; *or* — when the moving **images** are ordered differently from the endpoints — the images *adjacent to the endpoints* (01, N) shoving their atoms *outward*, off the path. Distinct from linear-interpolation-overlap above (ordering *correct*; a *single* image diverges), inputs-inconsistent below (energy offset, *sane* geometry, no cross-overs), and too-few-images / real-intermediate (atoms move *sanely*).
**Likeliest cause (ranked).**
1. **Inconsistent per-atom ordering across the band's POSCARs** — index N must be the *same physical atom* in *every* file (both endpoints **and** the interpolated/moving images); any mismatch — endpoint↔endpoint *or* endpoints↔images — makes the strictly index-based interpolation and inter-image tangent pair unrelated atoms, dragging index N to a wrong site (a straight-line cross-over) so many atoms traverse the cell at once. When the *images* (not the endpoints) are the misordered ones, it shows as the near-endpoint images pushing atoms outward rather than full cross-overs. Common when an optimizer/builder re-outputs an endpoint with atoms reordered within a species, symmetry-equivalent sites are labeled differently, an **adsorbate/adatom appended at the *end* of the atom list is sequenced differently between the two endpoints** (the usual culprit in surface/adsorbate work), the images were (re)built in a different order than the endpoints, or **each geometry was converted independently from a foreign format** (`.car`/Materials Studio, `.pdb`, `.xyz` via Molden/OpenBabel etc.) — the converter reorders atoms per-file, so separately-converted endpoints/images no longer correspond.
2. (Rule out) over-aggressive interpolation of a genuinely long path — that gives near-misses, not clean pass-throughs.
**Discriminating look.** *Cheapest pre-check — is each POSCAR even well-formed?* `nebmake.pl` parses coordinate columns strictly: an element-symbol label prefixing each coordinate line (coords must be bare numbers), or a `Selective dynamics` header with the per-line `T/F` triplet omitted, shifts the columns and scrambles *every* image — atoms in nonsensical spots though both endpoints *look* valid, reproducibly across different systems. *(No column shift but the same silent-garbage outcome: a `Direct`/`Cartesian` keyword on line 7 that mislabels the actual coordinate values — `nebmake.pl` reads the label literally and interpolates in the wrong frame, so images come out geometrically wrong with no error.)* Fix the format (strip labels; a `T T T`/`F` triplet on every line iff `Selective dynamics` is declared, else drop that header; correct a mismatched Direct/Cartesian label or rescale the coords) before chasing ordering. Then the quantitative ordering check: `dist.pl 00/POSCAR 0N/POSCAR` reports per-atom IS→FS displacement — any atom moving far beyond the expected (sub-Å) migration distance flags a misordered index *before* you build the band (compare element+frac per atom; species-count line alone is **not** enough). Then `grep -i 'too close' */OUTCAR`, and on any image OUTCAR check whether the enormous forces hit *many* atoms across the cell (index mismatch) versus just the one intended mover / a single near-contact image (linear-interpolation overlap, entry above). If it *stalls* not crashes, `grep -c Iteration */OUTCAR` = 1 ionic step confirms it. View the band movie (VESTA/OVITO) *before the first run* — cross-overs, or any atom hopping tens of Å between images while the rest are sub-Å, are the giveaway.
**Fix family.** Re-relax both endpoints, then reorder atoms in one (or both) endpoint POSCAR so every index maps to the same physical site — *only the reaction-relevant atom(s) differ in position; every other sits at the same fractional coordinates in both*. With chemically *equivalent* atoms (vacancy/swapped-site paths) same-index pairing isn't enough: pair each atom to its **nearest** counterpart in the other endpoint (manually or via a nearest-neighbor / min-displacement matching script), else identical atoms swap across the cell. Re-interpolate, then re-view the movie (`nebmovie.pl`) before submitting. **Prevent it at the source (foreign formats):** convert **one** endpoint to POSCAR, validate its atom order, then derive every other geometry from that single reference (e.g. `nebmake.pl` from the two converted endpoints) — never convert each image independently.
**Rule.** Atom ordering must be identical across *all* the band's POSCARs — both endpoints **and** every interpolated image — index N = the same physical atom everywhere, differing in position only for the reacting atom(s). Any re-indexing — optimizer/builder reordering, symmetry-equivalent sites mislabeled, or images rebuilt out of order against the endpoints — routes every mis-labeled atom wrong even when species counts match (the crashes, stalls, tens-of-Å jumps, and outward-drifting near-endpoint images above). Catch it with `dist.pl`, then view the initial band movie before submitting — every atom should glide sub-Å/image. But *first* rule out a **malformed** POSCAR: `nebmake.pl`'s strict column parse turns an element-label prefix or a triplet-less `Selective dynamics` header into garbage images regardless of ordering, and a wrong `Direct`/`Cartesian` keyword (line 7) into silently mis-framed ones with no error — fix the format (labels / `Selective dynamics` triplet / coordinate-type label) before interpolating. (Index-mismatch expert-stated 2015; malformed-POSCAR format rule OLD/2009 — a perennial `nebmake.pl` input requirement, not version-specific.)

## NEB dies instantly — OSZICAR empty, OUTCAR stops at the POSCAR-echo header, stdout "No initial positions read in" — a broken image POSCAR coordinate block, not INCAR/parallelization
**Signature.** Each image's POSCAR is *found*, but one image prints `No initial positions read in`; that image's OUTCAR stops at the POSCAR-echo header and its OSZICAR is empty — VASP never began SCF because it could not parse atomic positions in ≥1 image. Distinct from the env/build "exits with *no* OUTCAR at all" (SKILL triage, launch environment) and from a malformed-but-*parseable* POSCAR that silently *scrambles* a running band (ordering entry above) — here the run never starts.
**Look → fix.** `grep -l 'No initial positions read in' */stdout` (or the scheduler log) to name the culprit image, open that `XX/POSCAR`, and confirm the `Direct`/`Cartesian` coordinate block is present and non-empty — `grep -n 'Direct\|Cartesian\|^$' XX/POSCAR` exposes a missing/empty coord section or a stray blank line before the header. Check *every* image, not just 00/NN. Regenerate the band from clean endpoint POSCARs (`nebmake.py`/`nebmake.pl`); the format pre-check in the ordering entry above applies (element labels, `Selective dynamics` triplet, coordinate-type label). *(Pre-2009 vtstscripts `Vasp.pm` `read_poscar`/`write_poscar` itself appended a spurious blank line after the atom-count line when the source POSCAR lacked `Selective dynamics`, so `nebmake.pl` emitted POSCARs VASP couldn't parse — fixed upstream by 2009-03; affects only very old script checkouts, so update the scripts.)*
**Rule.** Empty OSZICAR + "No initial positions read in" + an OUTCAR that stops at the POSCAR echo ⇒ inspect the coordinate block of every image POSCAR first, before INCAR or parallel layout.

## NEB forces frozen/static, oscillate/stall, *or finish cleanly with no climbing image* despite IOPT/LCLIMB set — VTST not in the binary (not compiled / wrong executable), the compiled-in patch out of sync with the vtstscripts, or the vtstscripts not all installed on `$PATH`
**Signature.** IOPT/SPRING/LCLIMB are all set yet act as if absent — several faces of one fault: **(1) frozen** — `IBRION=3`+`POTIM=0`+`IOPT=N`: without vtstcode IOPT is ignored and `POTIM=0` taken literally, so VASP iterates at *zero* displacement; max force pinned every step, band never moves (`nebef.pl` negligible inter-image change), energies rise monotonically IS→FS with no saddle. **(2) `neb.dat`/`nebbarrier.pl` force *or* distance/reaction-coord col all `0.0`** so the band *looks converged instantly*; the same zero data prints `nebef.pl` `0.000000` on every image and crashes `nebspline.pl` `Illegal division by zero` (a script crash masking the cause). **Which column is zero names the root.** *Force/energy* col zero ⇒ **no NEB-force hook** (binary unpatched, cause 1), *or* — **banner present and the OUTCAR carries the `NEB: forces:`/`NEB: distance` lines but lacks the `NEB: projections on to tangent` line `nebbarrier.pl` greps to fill that column** — a compiled-in patch out of sync with the scripts (cause 3); the band ran fine, only the readout is empty. *Distance/reaction-coord col zero while force/energy reads non-zero* ⇒ the binary is fine; **no ionic step completed (CONTCAR==POSCAR, cause 5), or vtstscripts incomplete/off `$PATH` (cause 4)**. **(3) native-NEB fallback** — a *displacing* IBRION (no `POTIM=0`) hands the band to VASP's **own non-VTST NEB**: it integrates the path but, lacking the VTST spring/tangent projection, each image falls toward its *own* minimum uncoupled — so it oscillates/stalls and never converges, *or* the relaxed MEP comes out **unphysically wiggly, `mep.eps`/spline showing more extrema than images** (projected NEB forces ~0 at every image despite non-zero per-atom ionic forces), *or — most insidious — finishes cleanly as a plain band with `LCLIMB` silently inactive: no climbing image, no CI saddle, exit looks normal, and the plain-NEB result is trusted as a CI-NEB TS.* **(4)** lead complaint is often an **unreasonably large barrier** (several eV) — meaningless raw interpolated geometry; route a "barrier too high" with this INCAR shape here, not to ridge-trapped/parameter entries. Don't chase any face as an `IOPT`/`SPRING`/`TIMESTEP` mis-tune.
**Likeliest cause (ranked).**
1. **vtstcode not compiled into the binary that ran the NEB.** (a) never patched (single stock build); (b) several VASP executables, vtstcode linked into only *one* (often the endpoint build) while the NEB job script points at a stock build. The tangential-force projection lives *inside* the patched binary, so stock VASP silently ignores IOPT/SPRING/LCLIMB and never writes the projected forces.
2. IOPT typo / tabs in INCAR (`IPOT`, stray whitespace) — same frozen symptom, cheaper to rule out; the fix is *only* the spelling — keep `IBRION=3 POTIM=0` (they yield ionic moves to VTST once `IOPT` parses), don't re-tune them.
3. **Compiled-in VTST patch ≠ the vtstscripts revision in use.** The binary *is* patched (banner present) but an older/divergent patch omits an output line a newer script parses (`NEB: projections on to tangent`), so the script silently fills its column with zeros — out of sync, not unlinked. Affects only the post-processing readout; the band itself ran correctly.
4. **vtstscripts incomplete / not on `$PATH`.** `nebresults.pl` shells out to its co-located siblings to compute the inter-image distances; missing them it writes a zero distance column (forces/energies still correct), so `nebspline.pl` divides by zero — a scripts-install problem, binary fully patched (banner + CHAIN block both present).
5. **Post-processed before any ionic step finished — *same* zero-distance/forces-fine surface, but nothing is broken.** `nebresults.pl`/`dist.pl` derive arc length from each image's **CONTCAR**, not OUTCAR; before the band's first ionic step the CONTCARs are absent or still equal the initial POSCARs, so distances are legitimately `0.0`. Cure: let a step complete, *not* reinstall scripts. (CONTCAR-read behavior per ~2007 vtstscripts — confirm current scripts still read CONTCAR.)
**Discriminating look.** Cheapest glance: a VTST-linked binary's `neb.dat` carries a distance (arc-length) column — `image  distance  energy` (3 cols); an unlinked one either omits it (2 cols) or writes it as `0.000000` for *every* image (energies/forces may still be non-zero), so a 2-col `neb.dat`, or all-zero distance **with the force/energy column also zero**, flags a non-VTST binary before you even grep — **but all-zero distance with a *non-zero* force/energy column is cause 4 or 5: *first* the free check — `ls */CONTCAR` and diff one against its POSCAR; absent or identical ⇒ no ionic step has run yet (cause 5), just let the band step; CONTCARs present and moved ⇒ scripts off PATH (cause 4), `which nebresults.pl` and confirm the whole vtstscripts bundle sits in one PATH dir.** On the *NEB* OUTCAR: `grep VTST OUTCAR` — no `VTST: version X.Y` banner ⇒ vtstcode not linked here; that alone is the diagnosis. **Native-NEB output is not linkage proof: base VASP's own NEB (selected by `ICHAIN=0`) prints `CHAIN-FORCE`/`TANGENT` and honors `SPRING` *without* vtstcode, so those tokens show only that *a* band ran — not that VTST-only tags (IOPT/LNEBCELL) are live.** A patched run instead **echoes the VTST CHAIN tag block (`LCLIMB`/`SPRING`/`ICHAIN`/`IOPT`) near the top** — its absence is a second confirmation, decisive when the band *finished cleanly* (no frozen/`0.0` tell). Then `grep 'vasp\.' OUTCAR` vs the *endpoint* OUTCAR banner: a different build string confirms two executables (flavor b). `grep -iE 'iopt|ibrion|potim' INCAR` for spelling/tabs and the `IBRION=3 POTIM=0` inert-optimizer shape. For cause 3 the banner is *present*, so don't stop at `grep VTST`: grep the OUTCAR for the **exact keyword the failing script parses** (`grep 'NEB: projections' OUTCAR` for `nebbarrier.pl`) — absent *with* the banner present = a patch↔scripts version mismatch, not an unlinked binary; cross-check the OUTCAR `VTST: version` against the vtstscripts rev/date.
**Fix family.** Recompile vtstcode into the NEB executable, or repoint the NEB job script at the VTST build; endpoint and NEB jobs must use the *same* VTST-linked executable. Verify a fresh OUTCAR carries the `VTST: version` banner *and* the CHAIN tag echo before trusting any band — a clean finish without them is the silent-fallback trap. Locked to a stock binary ⇒ recover the metric by hand (dot each image's OUTCAR Cartesian forces with the finite-difference inter-image tangent) — a stopgap, not a substitute. Cause 3 (patch↔scripts mismatch): rebuild VASP with the VTST patch matching your vtstscripts revision (or run the scripts shipped with the compiled patch), and keep all vtstscripts co-located in one directory (they call each other internally). Cause 4 (distances zero, binary fine): download the *complete* vtstscripts bundle into a single `$PATH` directory (`which nebresults.pl` to confirm it resolves there), then re-run `nebresults.pl` — the distance column fills and `nebspline.pl` runs.
**Rule.** If IOPT/LCLIMB are set but the NEB acts as if ignored — forces pinned/frozen, `neb.dat` all-`0.0`/`nebspline.pl` divide-by-zero, monotonic energies, an absurd barrier, "converged instantly", oscillating/stalling, an **over-wiggly MEP (more spline extrema than images, projected forces ~0)**, *or a clean finish with no climbing image* — the first check is always `grep VTST` on the *NEB* OUTCAR; absence (plus a missing CHAIN-tag echo) means a non-VTST binary ran — but a *present* banner + CHAIN echo with **only the distance column zero** (forces non-zero) is either no ionic step completed yet — `nebresults.pl`/`dist.pl` read arc length from CONTCAR, absent/==POSCAR ⇒ legitimately `0.0`, just let it step — or incomplete/off-`$PATH` vtstscripts: `ls */CONTCAR` first, then `which nebresults.pl`. Confirm endpoint and NEB OUTCARs carry the *same* banner — mismatched builds betray flavor (b). And an all-zero vtstscript output *with* the banner present = patch↔scripts mismatch: grep the script's exact keyword (`NEB: projections on to tangent` for `nebbarrier.pl`), absent ⇒ match the vtstscripts revision to the patch (VASP 5.3.5 affected, 4.6.35 not; vs TST 3.1 scripts, 2016 — older/site-modified patches may omit the line regardless). (Expert-confirmed 2013–2018 across VASP 5.2.11–5.3.x and several VTST releases; not version-specific — an unpatched binary ignores the tags regardless of release, and base VASP's `ICHAIN=0` NEB prints `CHAIN-FORCE`/`TANGENT` with no vtstcode. Scripts-incomplete/off-PATH cause 4 expert-stated 2012, VTST 2.04b — not version-specific.)

## NEB crashes at launch (internal error in us.F / segfault), forces & energies "crazy" — *or* an off/ambiguous barrier with no crash — un- or partially-relaxed endpoint, not the NEB feature
**Signature.** VASP dies on the *first* NEB step — an internal error (often `us.F`) or a segfault — with wild energies/forces orders of magnitude too large, not near-converged. The user blames an NEB-specific ingredient just added (VCA, a vacancy, a special POTCAR), assuming it needs bespoke NEB handling. **Graded severity:** a *fully diverging* endpoint crashes as above; a merely *partially-relaxed* one (forces above `EDIFFG`, not diverging) does **not** crash — its residual endpoint forces bias the band, silently corrupting the reference energies and image forces, so the symptom is instead a *suspect MEP* (**a *terminal* image 10+ eV above its neighbors; a zigzag / multi-peak profile with per-image forces oscillating well above 1 eV/Å; *images scattering off the path (large-amplitude reorientation, drifting from the saddle, no barrier)*; *or the `nebspline.pl`/`mep.eps` curve not flattening at image 00 / the final image — a non-zero endpoint tangent is a direct readout of residual force on the fixed endpoint*; *or a `nebmovie.pl` **relaxed-frame** image ordering that reverses / goes non-monotonic near an endpoint (a late image nearer the IS than its predecessor) while raw `nebmovie.pl 0` is smooth — interior images relaxed *past* the endpoint, so the band curves back to reach it*; rather than one clean barrier**) **or a CI-NEB that never converges** — the endpoint drifts off its input geometry to its own minimum *during* the run, so the band chases that endpoint relaxation instead of settling on the saddle. Route "NEB crashed at launch", "barrier looks off", "CI-NEB won't converge", and "won't converge — must need more images / a stiffer `SPRING`" (endpoints never hit reached required accuracy) here.
**Likeliest cause (ranked).**
1. **An endpoint was never *fully* relaxed** — anywhere from outright diverging (crashes) to merely partially-converged (forces left above `EDIFFG`). An unconverged endpoint makes the interpolation ill-posed: in the extreme VASP's internal routines crash; short of that there is no crash, but the barrier and image forces are corrupted band-wide. The most common root cause of apparent "NEB method" failures.
2. **A misconception that the suspect feature needs special NEB treatment** — NEB is *not* a separate VASP mode; it reads the same POSCAR/POTCAR as a regular optimization. VCA (and vacancies, special pseudos) are transparent to NEB: anything that converges in a single-image relax works unchanged in the band.
**Discriminating look.** On *each endpoint* dir: `grep 'reached required accuracy' OUTCAR` (absent ⇒ endpoint never converged); `grep -E 'energy  without|FORCES' OUTCAR` for NaN/`+++`/values orders of magnitude too large. Cheapest no-crash tell: diff `00/CONTCAR` (or `NN/CONTCAR`) against the POSCAR that dir was built from — a large displacement means it was fed an *unrelaxed* geometry (a fresh POSCAR, not its own relaxed CONTCAR) and drifted to its minimum during the run, throwing the terminal image high. Also read `neb.dat`/`mep.dat`: an endpoint (image 00 or NN) energy that is *not* below every interior image flags a partially-relaxed (or metastable) endpoint — cross-ref the MEP-topology entry. The same drift reads off the movie: `nebmovie.pl` over the *relaxed* frames non-monotonic near one endpoint (raw `nebmovie.pl 0` smooth) points at that endpoint, not the band; if it *did* reach accuracy, suspect a settings mismatch relaxing it to a different geometry than the band's (ENCUT/KPOINTS/ISPIN — inputs-inconsistent entry).
**Fix family.** Fully relax both endpoints as standalone optimizations (normal IBRION=2 run to the `EDIFFG` *force* criterion — the band's, or tighter; energy + forces stable, accuracy reached) *before* building any NEB images — then use each relaxed CONTCAR as the image-00 / NN POSCAR and reinterpolate every interior image from scratch (`nebmake.py`). Debug the suspect feature (VCA/vacancy/pseudo) in that single-image relax; once it converges, reuse the identical POSCAR/POTCAR for the band — no NEB-specific changes.
**Rule.** Validate endpoint convergence (and any special feature) in a standalone single-image relax to the `EDIFFG` *force* criterion — not merely until the SCF stops crashing — *before* blaming NEB, an ingredient, the image count, or `SPRING`: a diverging endpoint is the most common crash mistaken for method incompatibility, and a partially-converged one won't crash but silently corrupts the barrier and image forces band-wide. NEB is transparent to setup: if it relaxes in a regular optimization, it runs unchanged in the band.

## Too few images — coarse spacing under-resolves the path (slow/oscillating/stalled convergence, or multi-hump on a multi-step path); image *placement* — uniform is correct, don't weight toward an anticipated TS
**Signature.** NEB converges *very slowly*, oscillates, or stalls with no crash — or expected one barrier but see several humps / a shoulder — **or it *reports convergence* yet resolves no saddle**: the few intermediate images relax back into the endpoint basins, so their energies/geometries barely differ from IS/FS and the profile reads barrier-less though nothing crashed. Two regimes of one root: **(a)** even a *single*-step path is under-resolved (image-to-image distance >1 Å), so the optimizer has no smooth MEP and forces creep/oscillate/stall; **(b)** the path spans more than the bond change — *geometrically* (reaction + product desorbing / reactant approaching over a large distance) **or *chemically*** (proton transfer or solvent/ligand *reorganization* secretly running in two steps — transfer then H diffusion — hiding a second metastable intermediate behind one apparent barrier even when the path isn't geometrically long). IMAGES small (3-4) ⇒ only 1-2 images in the reaction region; a **single** image is the limiting case — its saddle tangent is set by the two fixed endpoints alone, so the climbing force is inaccurate: it may not converge, and if it does the barrier is at best a *lower bound* to the true highest saddle (add images — 4-8 for one barrier — when the height must be trusted, and confirm it's stable to image count).
**Likeliest cause (ranked).**
1. **Image budget too thin for the path length.** (a) sparse spacing under-resolves even a one-step MEP → slow/oscillating convergence; (b) over a multi-step path, 4 images split across a chemical step *and* a long desorption/diffusion (or *two* chemical steps) leave the reaction step a single image → multi-hump / shoulder / won't settle.
2. Optimizer/timestep compounding it — high-frequency adsorbates (CO2, CO, H2O…) need a small quickmin step (parameters.md) — but check spacing *first* (too-coarse IMAGES out-ranks an optimizer mis-tune).
3. A genuinely metastable endpoint adding a real early step (MEP-topology entry below).
**Discriminating look.** Inter-image distance (`dist.pl` — the root-sum-squared (RMS) displacement of *all* atoms between two image POSCARs/CONTCARs — **mass-unweighted, no COM projection** — a **collective** reaction coordinate, *not* a single migrating-atom Cartesian distance; the spacing metric `nebbarrier.pl` calls internally [if the two *disagree*, reconcile before trusting either: an *archival* `nebbarrier.pl` greps the OUTCAR for distances instead of calling `dist.pl` (current revisions call it directly — update the script); else you're comparing *cumulative* path length against *per-segment* spacing, or fed the two scripts *different* geometry files (POSCAR vs CONTCAR) — feed both the same] — or the OUTCAR `NEB: distance to prev, next image` line VASP prints every ionic step, or diff adjacent CONTCARs) vs `grep IMAGES INCAR` — >1 Å/image flags under-resolution, and a large *inequality* between consecutive distances in the **initial** band flags a poorly-distributed interpolation (→ uneven force convergence) to re-space before submitting (in a *converged* band uneven spacing is instead a convergence symptom `EDIFFG` equalizes — MEP-topology entry); count structural phases (bond change, desorption, diffusion, *or a likely second chemical step*); `neb.dat` for a shoulder/second maximum, `nebef.dat`/`nebgef.dat` for images clustering away from the reaction region. The `nebresults.pl` **inter-image angle** reads path curvature (= `acos` of the two *normalized* neighbor-displacement vectors — full **3N-dim** config-space vectors, a single inner product, **not** a per-atom angle computed then summed over atoms — un-mass-weighted; ≈180°=straight, **0°/<90°=sharp corner/kink**; the same bend the multi-step entry calls a >90° *tangent-turning* angle) — purely **diagnostic** (printed only to gauge how straight the band is; it **never enters the NEB force projection**), a *symptom*, not an add-images trigger: small angle **+** coarse spacing (>1 Å/image) ⇒ genuine under-resolution (add images); small angle **+** *fine* spacing ⇒ an **intermediate minimum** bending the path (relax the low mid-band image, split into segments — MEP-topology / multi-step entries), **poor convergence** faking curvature (conservative optimizer first), or **mismatched endpoint selective-dynamics** (an atom frozen `F F F` at *different* positions across the two endpoints interpolates a localized kink/high tangent angle — inputs-inconsistent entry, cause 4).
**Fix family.** Raise IMAGES until resolved — standard starting point **one image per ~0.5–1 Å of path length** (3 often enough for a short/simple path; ≥2-3 bracketing *each* phase, **≥8 when a two-step is plausible**), set *before* chasing optimizer/timestep — and let the band **optimizer** relax the geometries: **never hand-edit interior-image coordinates** to force down bond distances/forces (a single under-resolved image on a curved path tempts exactly this), which bypasses the force-consistent optimization and rarely converges; add images and re-interpolate instead. Space them **uniformly** (even linear/IDPP interpolation) — *don't* cluster images toward an anticipated early/late TS: vtstcode has no per-side image weighting, and the climbing image self-selects and converges to the saddle wherever it falls, so non-uniform spacing (or an energy-weighted spring — tried and abandoned) buys nothing.
**Rule.** When NEB converges slowly, oscillates, stalls, or multi-humps, check image *spacing* (>1 Å/image) before tuning other parameters — too few images under-resolves the path so the optimizer never finds a smooth MEP; reorganization reactions also hide a second step. Budget ~one image per 0.5–1 Å (≥2/phase, ≥8 for a suspected two-step), spaced *uniformly* (Fix); check `neb.dat` for a shoulder before trusting one barrier. **Conversely, no image count is universally correct — a band has *enough* images once its climbing image itself converges (CI projected force ≤ `|EDIFFG|`); let CI convergence, not a target number, settle the cost-vs-accuracy tradeoff — and a *large* barrier alone is no add-images cue: check that highest image's residual force (→0 ⇒ saddle well-defined, more images won't sharpen it) + play the movie (implausible barrier ⇒ *unconverged* or *unusual chemistry*, not coarse sampling)** (single-image limit excepted — Signature). A small `nebresults.pl` inter-image angle is a high-curvature *symptom*, not an add-images cue: triage intermediate-minimum → convergence-quality → endpoint-mask-consistency → image-density — add images **last**, never the reflex on first sight.

## Unphysical barrier *or* structural discontinuity at the endpoints — inputs inconsistent endpoints↔band
**Signature.** The profile is physically impossible: interior images tens-to-hundreds of eV *above* the endpoints (positive spikes, won't converge), or *below* them — up to a fully **inverted** profile (both endpoints above the apparent saddle, a negative barrier), which can appear even on a *fast, clean* convergence. A true MEP never dips below its endpoint minima, so **any converged interior image below the endpoint energy is the fingerprint — the *sign*, not the magnitude, is diagnostic**: a species-order bug depressing the *whole* band (≈10 eV below both endpoints), INCAR drift spiking images hundreds of eV the other way, a stock-vs-VTST **build mismatch** sinking interior images below image 0, or an *endpoint-relaxation* INCAR (`PREC`/`ENCUT`/`EDIFF`/functional) differing from the band's leaving a *modest* ~1–2 eV **kink localized at image 0 or the final image** (the endpoint offset from its adjacent image, not a mid-band spike). Even a *small* below-minimum dip is a settings-mismatch suspect first, a real shallow intermediate only after inputs match (MEP-topology entry below). **A *geometric* tell of the same root:** a structural shift between the band's terminal images and the relaxed endpoint (a layer/adsorbate differing in image 00/0N vs the endpoint CONTCAR) — endpoints relaxed under *different constraints/cell* than the band, even when energies look unremarkable.
**Likeliest cause (ranked).**
1. **POTCAR mismatch across dirs — variant *or* species order.** Endpoints and band images using different pseudopotential *variants* (`_pv` vs `_sv`, or a different dataset date), *or* per-element pseudos concatenated in a *different order* (or not matching the POSCAR species line, so atoms map to the wrong pseudo): either way absolute energies land on different scales — variant differences skew the barrier, an ordering mismatch shifts the whole band ~10 eV.
2. **k-point mesh / symmetry mismatch** — different KPOINTS meshes (different `NKPTS`) between endpoints and images, *or* the same nominal mesh reduced to a different irreducible k-set because the endpoint relax ran a different `ISYM`. `ISYM=0` everywhere is a stopgap that masks the symmetry discrepancy; matching KPOINTS (and `ISYM`) across dirs is the real fix.
3. **INCAR drift — band↔endpoint *or* endpoint↔endpoint.** Any differing DFT-physics tag (`ISPIN PREC ENCUT GGA LMAXMIX NBANDS ALGO ISMEAR SIGMA EDIFF MAGMOM LDAU* IVDW LREAL ISYM`) makes total energies incommensurable. The drift is frequently *between the two endpoints themselves* (separate runs → e.g. `PREC=Accurate` vs `Normal`, different FFT grid), corrupting the barrier before the band exists; equally band↔endpoint when endpoints were pre-converged at lower `PREC`/`ENCUT` then read as-is because they're 'already correct' — true for *geometry*, false for the *barrier*: pre-converge endpoint geometry cheaply if you like, but recompute the final endpoint **energies** at the band's production accuracy (a barrier cancels DFT error only when every image shares one Hamiltonian). **A dropped `ISPIN`/`MAGMOM`/`NUPDOWN` is the classic case — the band INCAR omits or *deliberately resets* the spin tags (`ISPIN=1` on an open-shell system whose endpoints relaxed `ISPIN=2`; never switch spin off mid-workflow on a magnetic system), running non-spin-polarized: every image is forced into the non-magnetic state and *all* interior images sit a clean ~2–5 eV *above both* endpoints (the band converges fine; a *uniform* above-both-endpoints lift, not a lone spike, is the tell — not the mere "bias" a small drift gives).** Other magnitudes vary: a `GGA`/`PREC`/`NBANDS`/`LMAXMIX` mismatch spikes images hundreds of eV, a milder `ISMEAR`/`SIGMA`/`ENCUT`/`LREAL` drift only *biases* the barrier — plausible yet wrong, so don't trust a sane-looking band. Root: a band INCAR written from scratch, not copied from the endpoint INCAR.
4. **Geometry-affecting mismatch — selective-dynamics or ISIF/cell shape.** A layer frozen in the endpoint relax but free in the band (or vice-versa), or a differing `ISIF`/cell, lets atoms relax in one run but not the other — the tell is the *structural* discontinuity above (a shifted layer), not an energy spike, so it slips past a `neb.dat`-only check. **Masks must be physically *consistent*, not just identical across dirs: an atom held `F F F` at a *different* position in the two endpoints (frozen yet displaced along the path) makes `nebmake.pl` interpolate through unphysical mid-band geometries — presenting *either* as a discontinuous band with forces spiking/oscillating on the frozen images, *or* (it converges fast) as a monotonic/barrier-less profile, energies sliding IS→FS with no peak where physics demands a barrier. Diff the `F F F` coordinate lines of `00/POSCAR` vs `N+1/POSCAR` — they must be bit-for-bit identical before interpolating. Freeze only atoms invariant across *every* image; free anything that moves appreciably — and a `F`-masked Cartesian component on the *migrating* atom itself (even a *consistent* mask, e.g. `x,y` frozen with only `z` free on a vacancy-hopping atom) bars the path along the frozen direction, so the diffusing atom never advances and the NEB makes no progress however the optimizer is tuned.** (Looser endpoint `EDIFF` sits here too — residual force the band then "corrects".)
5. **VASP binary/build mismatch — same inputs, different executable.** Endpoints relaxed with a stock build but the band run with a VTST-patched build (or a different VASP version) reference the absolute energy zero differently — even the *same* version with vs. without VTST shifts it 10s–100s eV. The band converges normally; nothing's off until `nebefs.pl`/`nebef.pl` relative energies come out absurd (interior images 100s eV below image 0) while single-point recalcs on one consistent binary read sane (~0–5 eV).
**Discriminating look.** `grep TITEL */OUTCAR` across endpoint and image dirs — identical pseudo variant + dataset date **and element order**, matching the POSCAR species line; `grep NKPTS */OUTCAR` + diff KPOINTS; diff DFT tags of the band INCAR vs `00/OUTCAR` **and** the two endpoint OUTCARs against *each other*. Build mismatch: `grep 'vasp\.' */OUTCAR` + `grep VTST */OUTCAR` — every dir must carry the *same* banner and VTST presence; a stock-vs-VTST/cross-version split corrupts the energy scale with INCAR/POTCAR/KPOINTS all matching. Geometric tell: overlay the relaxed endpoint CONTCAR on band image 00/0N and diff selective-dynamics `T/F` + `ISIF`. **A kink localized at *one* endpoint (image 0 or final, not a mid-band spike) while every input matches ⇒ not a mismatch but an un/partially-relaxed endpoint — `grep 'reached required accuracy'` that endpoint's OUTCAR (un/partially-relaxed-endpoint entry above).**
**Fix family.** Regenerate POTCAR/KPOINTS/INCAR from one canonical set: concatenate the element pseudos in one fixed order (matching the POSCAR species sequence) for *every* dir; **copy the endpoint INCAR verbatim, add only NEB-control tags (IMAGES, SPRING, ICHAIN, LCLIMB, IOPT, NWRITE).** Re-relax both endpoints with those identical settings *including selective-dynamics and ISIF*, re-interpolate so terminal images match the relaxed endpoints, and re-run the band — every step on one identical VASP+VTST binary. Don't "correct" energies or geometry post-hoc.
**Rule.** Every endpoint (both, relaxed separately — diff them against *each other* too) and band image must share identical POTCAR variant **and species order** (TITEL = POSCAR line), k-mesh, *every* DFT-physics INCAR tag, geometry constraints (selective-dynamics — freeze only atoms invariant across *all* images, never one that moves; ISIF), **and the same VASP+VTST binary**. Only NEB-control tags may differ — copy the endpoint INCAR and append them, never write one from scratch. Any mismatch breaks DFT error cancellation (spikes, a whole-band depression, a build-mismatch sink, or a geometry-drift discontinuity); a below-minimum interior image is the fingerprint regardless of magnitude *or how cleanly the band converged* — run the diffs (element order first) before calling a dip a real intermediate.

## MEP topology looks "wrong" — off-center CI, mid-band dip, plateau, or unequal spacing (real feature vs artifact)
**Signature.** Any of: the climbing image lands off the geometric midpoint (image 6 of 9, not 5); a mid-band image sits *lower* than both neighbors (local min) despite `LCLIMB=.TRUE.` — an **M-shaped** two-maxima/one-minimum profile is the canonical multi-saddle case (user may ask *which* peak CI climbs, and whether the intermediate-minimum energy is trustworthy); an interior image *adjacent to an endpoint* sits *lower* than it (early image below IS, or *penultimate* below FS — that endpoint may not be the true minimum); **the runtime form of this near-endpoint anomaly is vtstcode printing `NEB: image is at an extrema` for image 01/N-1 with both neighbour-energy flags `F`, contradicting the energy profile**; the *product* endpoint sits *above* the interior maximum (highest `neb.dat` energy is an endpoint, looking like no barrier); a flat *plateau* near the TS top; inter-image spacing visibly unequal, seemingly worse with more images. The user blames discretization/spacing and wants to dismiss the path — or suspects a method error / unconverged run.
**Likeliest cause (ranked).**
1. **A real PES feature** — a shallow/metastable intermediate (mid-band min, off-center CI) or a flat saddle region (plateau) — *not* a numerical artifact. (Off-center CI is also *intrinsic*: the climbing image is the highest-*energy* image, picked by energy **rank** — not the image nearest the spline/reaction-coordinate maximum, the two coinciding only for a harmonic saddle — so on an asymmetric/anharmonic profile the CI legitimately sits off the geometric midpoint.) Invisible at low image counts (1/3/5); only a dense band (≈9–11 images, more for a long path) samples it. The band then holds two (or more) transition states flanking one (or more) true intermediates — a **multi-step migration pathway** whose several metastable sites are genuine PES features, so the multi-well/asymmetric profile is physically correct (same for NEB and GSSNEB/SS-NEB). Same physics when the **product endpoint exceeds the interior maximum**: the chosen endpoints bracket an intermediate minimum (non-minimum-energy reaction coordinate), so the path's highest point is legitimately an endpoint — valid, not a failure.
2. **Unequal image spacing is itself a convergence symptom, not an artifact source.** Tightening `EDIFFG` drives spacing toward equal to arbitrary accuracy; it does *not* erase a real dip/plateau. Never cite spacing asymmetry to dismiss MEP topology.
3. (Only if forces are still large) genuine non-convergence — gives a *noisy* band, not a clean single dip/plateau; rule out before invoking physics.
**Discriminating look.** Per-image energies (`neb.dat`, `vasprun.xml`, or `grep -A2 'energy  without' OUTCAR` per image dir) for a clean local min / plateau, and compare *endpoints vs the interior max* in `neb.dat`; inter-image distances in OUTCAR; confirm forces have dropped / `EDIFFG` met so the feature isn't noise. When an image dips *below an endpoint*, first confirm endpoint↔band inputs match (inputs-inconsistent entry above) — a settings mismatch counterfeits a real dip and must be ruled out before invoking physics. Decisive test: tighten `EDIFFG` (and/or add images) — spacing equalizes while a *real* dip/plateau persists; an artifact washes out.
**Fix family.** Don't tighten convergence to chase a "symmetric" result — but DO tighten `EDIFFG`/add images to confirm the topology is stable. If a mid-band dip persists, relax the minimum-energy image in isolation; if it settles into a stable minimum, split into two NEBs (endpoint→intermediate, intermediate→endpoint) run in parallel (workflows two-step-reaction). **When the intermediate is *unsampled*** — a >1 Å gap with *no* low mid-band image to relax (tell: a suspiciously *low* CI barrier on a downhill process + high band-force at the gap's boundary image) — interpolate a probe geometry *into* the gap (≈10% from image N toward N+1) and minimize it: relaxes to the FS ⇒ the low barrier is real; falls into an intermediate minimum ⇒ run a separate NEB from that minimum to the FS for the *dominant* (higher) saddle, entirely absent from the current band. Segmentation is *optional* for the overall barrier: a single CI-NEB converges only its *one* highest-energy image to `EDIFFG` — it cannot resolve multiple saddles in one band — so the rate-limiting barrier is reliable while the intermediate-minimum and any *lower* saddle are not. To rigorously converge *every* saddle, subdivide into single-barrier NEBs (as above) or seed a min-mode run (dimer/Lanczos, `neb2dim.pl`) from each lower saddle's approximate image, pinning each intermediate with a standalone relax. **If the low image is instead *adjacent to an endpoint*** (early below IS, *or penultimate below FS*), that endpoint is metastable — relax that neighbor and, *if it settles below the original*, promote it to the corrected endpoint and restart, rather than splitting; frequent culprit is a leftover symmetry constraint pinning a defect/surface (`ISYM=0` in the endpoint relax, parameters.md). **Endpoint quality gates barrier *accuracy*, not TS *existence*: with `LCLIMB=.TRUE.` a converged climbing image gives a valid saddle energy even from a non-minimum endpoint — relaxing/promoting it sharpens the barrier (which is otherwise measured from a too-high reference) but isn't needed to *locate* the TS; don't `F`-mask only the reactant-side images to stop them sliding toward the true minimum, a partial mask spikes a discontinuous force at the T/F junction (inputs-inconsistent). (OLD/2007; version-independent.)** A *correctly* relaxed endpoint can still trigger the near-endpoint extremum / `is at an extrema` misfire if its **energy reference is missing**: VASP reads each endpoint's *geometry* but not its *energy*, so vtstcode assumes endpoints are minima and, when the true endpoint energy exceeds its neighbour, misflags image 01/N-1 — supply the endpoint energies (2011-era VTST: `EFIRST`/`ELAST` INCAR tags = the endpoints' relaxed DFT energies; current builds may instead read them from the relaxed-endpoint OUTCAR in `00/`,`N+1/` — verify which your VTST version uses before relying on either). **Product endpoint above the interior max:** no fix once forces are converged — valid physics.
**Rule.** Off-center CI, mid-band dip, plateau, and unequal spacing are convergence/physics signatures, *not* artifacts: raise image density + tighten EDIFFG — spacing equalizes but real intermediates (a multi-step pathway) or a flat saddle survive. CI-NEB converges one saddle per band (its climbing image), so the rate-limiting barrier is reliable while intermediate-minimum / lower-saddle energies need a per-segment NEB or a dimer-seeded min-mode. An intermediate can also hide in an *unsampled* >1 Å gap (no image to relax) — interpolate+minimize a probe geometry in the gap to expose a missed minimum/dominant saddle (MID/2011). A low image *adjacent to an endpoint* = a wrong/metastable endpoint (relax that neighbor, restart; check `ISYM=0` for defect/surface), not an interior intermediate — though a converged CI already yields a valid saddle from the imperfect endpoint, so relaxing it improves the barrier *magnitude*, not the ability to locate the TS. A *product endpoint above the interior max* is valid physics — endpoints bracket an intermediate minimum.

## "Will the images collapse into the lower-energy endpoint?" / "Does NEB apply when the product is *higher* than the reactant (endothermic)?" / "After inserting an extra image near the TS, must I re-space the band?" / "Can I just run each image as its own job?" — no, yes, no, no: endpoints are frozen, the spring (non-local) holds even spacing AND couples the whole band into one collective run, ΔE is irrelevant
**Signature.** A *conceptual* worry, usually before a run (or a *post-run* surprise): the optimizer will pull interior images downhill into the lower endpoint's basin, NEB can't treat an endothermic path, or — the operational version — each image was launched as a **separate independent single-point job** (one per directory, one per node, separate runs) and the reported barrier disagrees with a proper collective NEB. No VASP error — a misread of what NEB constrains.
**Clarification.** The endpoint images (`00`, `N+1`) are **held fixed** the entire optimization; only interior images move, each under (true force ⟂ path) **+** (spring force ∥ path). The spring enforces ~**equal spacing** along the band, so images cannot pile into or collapse to *either* endpoint whatever their relative energy — the reactant↔product gap is irrelevant to whether NEB applies. Because that spring force is **non-local** — each interior image's force depends on its *neighbors'* coordinates — the **whole band must be optimized in one VASP invocation** (`IMAGES`>1 set, MPI split into per-image groups, *one* run root with a single shared INCAR), **never** as independent per-image jobs in separate directories/nodes: running each image as a standalone single-point eliminates the inter-image springs entirely, so the "barrier" is just interpolated isolated single-points — meaningless and **system-size-dependent**, not a collectively optimized MEP. An endothermic path (product above reactant) needs **no special setup**: relax both endpoints, interpolate, run, exactly as for an exothermic one. The same spring **re-equalizes spacing after you insert an extra interior image near the saddle** (to better resolve the TS): no manual re-spacing needed — both sides of the climbing image relax to equal spacing at convergence. (`SPRING` only tunes stiffness; the default already prevents collapse — parameters.md. The related *post-run* case — a converged profile whose product endpoint tops the interior max — is valid physics, MEP-topology entry above.)
**Rule.** NEB freezes the endpoints and the spring keeps interior images evenly spaced, so they relax to the MEP without collapsing to an endpoint and re-equalize spacing on both sides of the climbing image at convergence even after you insert an extra image near the saddle (no manual re-spacing); exo- vs endothermic is immaterial to applicability and needs no special handling. Critically, the springs are **non-local**, so NEB is a *collective* optimizer: the entire band runs in a **single VASP invocation** (`IMAGES`>1, one shared INCAR), never as separate per-image jobs — decomposing it into independent single-points drops the springs and yields a meaningless, system-size-dependent barrier. (Endpoint-freeze/spacing expert-stated 2011; collective single-invocation requirement OLD/2015 — consistent; not version-specific.)

## Apparent saddle isn't *at* an image — sub-meV spline overshoot (soft-mode artifact) vs saddle *between* images (under-resolution)
**Signature.** The energy peak sits on no image — two opposite roots. **(a) Artifact:** a *converged* CI-NEB's `neb.dat` flags the highest-energy image as the TS, yet the cubic-spline overlay rises a *fraction of a meV* above it near the top — or, with translating/rotating fragments (dissociating ligands, mobile adsorbates), the *whole* `mep.eps` comes out jagged (irregular slope breaks) over an *inflated* `neb.dat` path length; image data fine either way. A third face of the *same* root: images **bunch in a flat energy region at one path end** (a rigid translation or ultra-soft phonon mode eats resolution there), so the spline's peak sits visibly *off* the highest image — yet `neb.dat`'s climbing-image-force column reads ~0, so the climb *succeeded* and only the saddle *position* is misdrawn (unlike (b), a genuine off-image peak from under-resolution). **(b) Under-resolution:** the spline / CI-curvature peak sits clearly off *any* image point (not sub-meV) *and* a frequency run on the highest image returns an imaginary mode whose eigenvector does **not** match the expected reaction coordinate (e.g. a ring rotation, not the bond cleavage) — that image is only a shoulder; the true saddle lies in a gap between images. (Inverse of MEP-topology above, where a surprising feature lives in the *image data* and is usually real.)
**Likeliest cause (ranked).**
1. **Soft-mode spline artifact (a).** The spline fits each image's energy *and* force vs inter-image distance, and the *plotted slope* at each image is the path-**perpendicular** force. Soft modes (rigid translation/rotation, low-frequency vibration) let adjacent images drift apart at constant energy/force and inflate the Cartesian path length `nebresults.pl` uses as the reaction coordinate, so distance and data disagree: the spline overshoots into spurious sub-meV extrema, and at larger amplitude energy-per-distance falls *below* the perpendicular force so the *whole* curve goes non-smooth however well forces converged. Image data is correct — a corrupted distance axis, not a convergence failure.
2. **Under-resolution — saddle between images (b).** Too few images, so none sits at the saddle; the highest captures a nearby shoulder/local feature. Decisive tell: wrong-coordinate imaginary mode *plus* a peak clearly off-image. *Cheaper spacing tell (no freq run):* an inter-image **gap >~1 Å** (`neb.dat` col-2 arc length) with **high residual band-force at the sparse-segment boundary image** and the spline saddle landing in that image-free gap ⇒ the cubic interpolation is unreliable *in that segment specifically* — the converged **CI energy stays the authoritative barrier**; densify the gap to <1 Å (≈7 images for a 5-image band) before trusting the interpolated MEP *shape*.
3. (Rule out FIRST) non-convergence — a lumpy/shouldered barrier with the climbing-image force still above ~0.02 eV/Å is a convergence artifact: tighten `EDIFFG` to `-0.01` and re-converge *before* blaming image density or soft modes (forces above `EDIFFG` make any bump noise). `nebresults.pl` extrema (written to `exts.dat`) are the spline's *zero-slope* points — minima **or** maxima at *interpolated* fractional positions, **not** discrete images; an extremum at a *fractional* index sitting **near** a climbing-image integer (e.g. 4.93 vs image 5), or a spurious *close pair* of near-integer extrema (a nearly-flat slope at that image, not an error — no input change), is the same incomplete-climb tell: the offset shrinks toward the integer as forces drop, and once negligible the integer climbing image *is* the TS — keep converging, don't add images or dimer-seed for a near-integer offset. An exts.dat extremum at/near an **endpoint** index (0 or N) is suspect for a *different* reason — the spline is *slope-constrained* (each image's force pins its derivative ⇒ no wild oscillation), so at the boundary only the single fixed-endpoint force anchors curvature and that force is typically near-zero with a slight negative sign, seeding a *shallow* spurious extremum. Hence when exts.dat **counts more extrema than spline.dat visibly shows**, the extras are almost always these boundary artifacts, not real barriers/intermediates: confirm by relaxing that suspect image standalone (geometry opt) — settles into a local minimum ⇒ real feature; drifts away ⇒ artifact. Trust NEB *image energies* over spline-inferred *forces* at the boundaries.
**Discriminating look.** Magnitude + frequency + *where* the extremum index falls. A *bare spline* point a fraction of a meV over an otherwise-correct highest image ⇒ (a). A fractional extremum index **near** an integer climbing image (4.93≈5), shrinking toward it as forces drop ⇒ (3) incomplete climb (converge, don't densify). A peak clearly *between* images (far from any integer) with a wrong imaginary mode (`OUTCAR` Eigenvectors / `mep.dat`) at the highest image ⇒ (b). Separate (b) from wrong-endpoints (workflows TS-confirmation): there the *path itself* is non-physical (dissociation→re-adsorption); here the MEP is smooth — only the sampling missed the saddle.
**Fix family.** (a) Sub-meV case — none: report the TS from the highest *image* with converged forces, ignore the spline extrema. *Severely* jagged whole-curve `mep.eps` (translating/rotating fragments): don't iterate the NEB to "smooth" it (corrupted distance axis, not non-convergence) — converge the climbing image to the true saddle (`EDIFFG`) and report *that*; if the MEP *shape* is genuinely needed, trace it from the converged saddle with a dimer or steepest-descent minimization in *both* directions. A reference-atom (or COM) constraint removes the net translation/rotation from the path metric — the drift inflates the Cartesian path length ≈ drift×N_atoms, bunching images at one end and starving the saddle, and `nebmake.pl` does *not* align image centroids nor reconcile atom *ordering* (check both baselines — a single reordered atom across image POSCARs fakes a huge per-atom displacement and the same overshoot); an active `LCLIMB` still pulls one image onto the saddle regardless. But an ill-chosen freeze of one atom to *suppress a rigid-translation zero mode* can itself elongate the path and oscillate the spline (per-image energies stay valid, ignore it). (b) Don't accept the highest image as the TS: cheapest is `neb2dim.pl` in the converged band to seed a VTST dimer (`ICHAIN=2`) with the *interpolated* saddle geometry **and** reaction-coordinate mode — no NEB rerun (workflows NEB→dimer handoff); else add images across the band, or run a zoomed sub-band over just the 2-3 images bracketing the suspected saddle.
**Rule.** First rule out convergence: a lumpy/shouldered barrier with CI force >~0.02 eV/Å — or a fractional `nebresults.pl` extremum *near* the climbing-image integer — ⇒ tighten `EDIFFG` to -0.01 and re-converge, *not* more images (the offset migrates onto the integer climbing image, which is then the TS). Once forces are genuinely tight, a sub-meV spline point above the highest image is a soft-mode artifact — ignore, don't chase with images or tighter `EDIFFG`. An off-*position* spline peak (saddle drawn away from the highest image because images bunched in a flat path-end region) is the same artifact: confirm the `neb.dat` CI-force column is ~0 — the climb already succeeded — then cure the resolution (freeze the rigid translation / redistribute images toward the saddle), don't re-converge. A *whole* jagged `mep.eps` over an inflated `neb.dat` path length (translating/rotating fragments) is the same artifact scaled up — converge and report the CI saddle, don't iterate to smooth it (Fix family for tracing / COM-freeze). But a spline/curvature peak clearly *between* images **plus** a wrong-coordinate imaginary mode at the highest image means an under-resolved band — that image is a shoulder, not the TS: seed a dimer with `neb2dim.pl` (inherits saddle + mode, no rerun) or densify/zoom. And an exts.dat extremum *near an endpoint* (0 or N) is a weak-boundary-constraint spline artifact more often than a real feature — relax that image standalone to confirm (real ⇒ settles to a local min; artifact ⇒ drifts off); NEB image energies beat spline forces at the boundary. (A wrong mode from a *non-physical path* = wrong-endpoints, workflows TS-confirmation; soft-mode artifact expert-stated 2015–; sparse-segment >1 Å spline-unreliability tell MID/2011; near-endpoint exts.dat artifact OLD/2007 — boundary-spline limitation is fundamental, not version-specific.)
